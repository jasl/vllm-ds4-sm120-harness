# 离线优先的任务管理：当用户信任成为系统设计的核心

对于习惯了“在线即服务”的前端开发者来说，“离线优先”往往被简单理解为：在网络断开时缓存一些数据，等网络恢复后再同步回去。这种理解停留在功能层面，却忽略了更本质的挑战——**用户信任**。当一个任务管理工具的核心价值是“承诺不丢失任何一条待办事项”时，离线能力不再是一种增强特性，而是整个架构的基石。

## 离线优先的真实代价：用户信任与数据一致性

设想一个场景：用户在地铁上创建了一个重要任务，随后关掉了手机。到达办公室后，他打开电脑，用同一个账号登录了Web应用——此时，手机端离线创建的任务尚未同步，而电脑端已经看到的是几分钟前的旧数据。如果这两个设备可以同时编辑同一个任务，那么冲突就在所难免。

离线优先不是“断网能用”，而是在断网时让用户感觉“一切正常”，在线时让数据的合并过程足够透明，让用户相信系统不会悄悄丢掉他做过的事情。这种信任一旦被打破——比如离线创建的任务在同步后莫名消失，或修改被覆盖而未通知——用户就会转向更传统的在线工具。因此，设计离线优先系统，首先得接受一个事实：网络不可靠是常态，而数据一致性是需要靠工程手段去“仲裁”的。

## 三驾马车：IndexedDB、Service Worker 与后台同步

在离线优先的架构中，有三层基础设施各自承担着不可替代的职责：

**IndexedDB** 是客户端的“持久化数据库”，负责存储任务实体、本地修改记录以及冲突元信息。与 localStorage 不同，IndexedDB 支持索引和事务，这使得我们可以安全地将本地写入操作包裹在事务中——写入失败即回滚，不会产生半残数据。更重要的是，每条任务记录可以附带一个`lastModified`字段和`localVersion`版本号，为后续冲突检测提供依据。

**Service Worker** 承担网络代理的角色。它拦截所有发往后端的请求，将请求缓存到 Cache Storage，同时在网络不可用时将写入操作存入后台同步的注册队列。关键设计在于：Service Worker 只负责“拦截”和“缓存”，它不应处理业务逻辑（比如冲突解决）。它的边界是数据传输的中继，而非数据的解释者。

**后台同步（Background Sync）与重试队列** 是离线写入转线上的“管道”。当网络恢复时，Service Worker 触发`sync`事件，将队列中的操作按顺序发往后端。这里常用的是**重试队列**模式——每个操作都被包装成一个任务单元，包含请求方法、URL、payload 和指数退避策略。如果同步失败，任务不会丢弃，而是进入待重试队列，直到达到最大重试次数后标记为“失败”，并上报给可观测性系统。

## 不容回避的冲突战场

在离线优先系统中，冲突不是异常，而是常态。至少有两类常见场景必须预设处理逻辑：

**场景一：同一任务在多端被编辑。** 用户A在手机端修改了任务标题为“买牛奶”，同时用户B在电脑端将同一任务的截止日期改为明天。两端各自离线写入本地IndexedDB。当网络恢复后，A的改标题请求和B的改日期请求几乎同时到达后端。此时，如果后端采用“后写入覆盖”的简单策略，很可能丢失其中一个修改。更合理的做法是**字段级合并**——针对不同字段的冲突，取各自最新的值；针对同一字段的冲突，保留版本号较新的或让用户手动选择。

**场景二：离线删除与在线更新交叉。** 用户在离线时删除了任务X，但在线的协作者同时更新了任务X的描述。当离线用户的删除请求同步到后端时，后端会遇到一个“删除一个已被更新的任务”的指令。如果简单执行删除，协作方的更新就丢失了。处理这类冲突，后端需要维护一个**操作日志**——删除不是直接删记录，而是标记为“已删除”，同时保留最新版本。当冲突发生时，将删除操作与更新操作对比，如果更新发生在删除之后，可以选择将删除推迟，或通知用户“有其他修改，确认删除？”。

## 简化数据流：从本地写入到最终一致

让我们串联一个简化的数据流：

1. **本地写入**：用户在离线状态下新建或编辑任务。前端首先将修改写入 IndexedDB（写本地），同时同步到组件状态让界面即时响应。这一步的关键是“乐观更新”——用户看到的是修改后的结果，即使后端还未收到。

2. **排队**：服务端请求被 Service Worker 截获。如果网络不可用，请求放入后台同步队列，附带请求重试次数初始值为0。队列中的每一条记录都包含本地时间戳和版本号。

3. **同步**：网络恢复后，后台同步事件被触发。队列中的请求按顺序逐个发送（注意顺序性对某些业务很重要，比如“先创建任务，再修改任务”不能调换顺序）。后端接收到请求后，执行冲突检测——比较本地的版本号与服务器的最新版本号。

4. **确认**：如果无冲突，后端返回成功响应，Service Worker 从队列中移除该任务，并通知前端更新本地状态——将“待同步”标志改为“已同步”。

5. **失败回滚或重试**：如果冲突发生，后端返回特定的冲突状态码（比如409 Conflict），并附带服务器端的当前版本数据。前端根据冲突类型决定：自动字段级合并（如果是无冲突字段），或展示冲突界面让用户选择。若请求因网络错误而非冲突失败，则进入重试队列，等待下一次同步窗口。

## 可观测性：追踪不可见的过程

离线优先系统最隐蔽的风险在于“沉默的错误”——用户以为同步成功了，实际写入了黑洞。因此，可观测性的维度需要扩展：

- **同步失败率**：通过埋点记录每次同步请求的成功/失败状态。区分技术失败（网络超时、服务器500）与逻辑失败（冲突）。理想情况下，冲突率不应超过总同步量的5%，超过即表示冲突检测策略需要调整。
- **平均恢复时间**：记录从用户在离线状态下发起一次写入，到该写入被后端确认成功的耗时。这个指标反映了重试队列的效率。如果平均恢复时间超过10分钟（取决于业务容忍度），可能意味着重试策略过于保守（比如退避时间过长）或后端处理能力不足。
- **冲突热点**：识别哪些字段、哪些任务、哪些用户群体最容易触发冲突。如果是日期字段频繁冲突，说明多人编辑同一任务的场景需要更细的锁定机制。

可观测性不是事后诸葛亮——在系统上线后，仪表盘上的冲突率曲线和数据同步延迟曲线，比任何单元测试都更能揭示架构的短板。

## 三条落地建议

1. **从“乐观锁”开始，不要追求完美一致性。** 对于轻量级任务管理，绝大多数冲突可以通过字段级合并解决。先落地一个简单的版本号冲突检测，观察线上冲突率后再决定是否需要引入操作变换（OT）或CRDT这样的复杂模型。

2. **让用户有知情权。** 无论如何设计冲突解决策略，都应在客户端提供一个“同步状态指示器”——让用户明确知道哪些数据已同步，哪些还在排队，哪些需要手动干预。透明的信任比完美的算法更珍贵。

3. **优先保护好“新建”操作。** 在离线场景下，用户新增的数据丢失的影响远大于修改丢失。确保离线创建的每条任务在同步前至少有两次冗余写入（IndexedDB + 内存缓存），并且首次同步后应有确认回执，才能从本地队列中移除。

离线优先不是技术炫技，它是在网络断裂的夹缝中，为用户搭建一座信任的桥梁。设计的核心不在于“让断网可用”，而在于“让用户相信，无论网络如何，他的数据终将被平等对待”。
