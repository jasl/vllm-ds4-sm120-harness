# 离线优先：一个轻量级任务管理应用的设计哲学

## 为什么离线优先是信任问题

很多团队对“离线优先”的理解停留在“断网也能用”的层面。这当然没错，但更深层的价值在于：用户一旦相信你的应用在网络不稳定时依然可靠，就会把它真正当作日常工具，而不是“勉强能用”的替代品。

想象一个工地巡检场景：工人在地下室或偏远区域创建任务、修改状态、添加备注。如果应用在网络恢复后数据丢失或产生不可预知的冲突，用户会很快放弃它。信任的建立来自一致的行为——用户操作的每一笔数据，都应该像银行转账一样可追溯、可确认、可回滚。

数据一致性是离线优先的核心难点。它不是简单的“把本地数据推到服务器”，而是要在分布式环境中模拟出“好像一直在线”的体验。

## 三块基石：IndexedDB、Service Worker、后台同步

离线优先的三个核心组件各有侧重，理解它们的边界很重要。

### IndexedDB：本地的事实来源

IndexedDB 是离线优先应用的“本地数据库”，不是缓存。这意味着：

- 用户的所有操作先写入 IndexedDB，视为“本地事实”
- 应用 UI 直接从 IndexedDB 读取，不依赖网络响应
- 每条记录携带本地版本号、时间戳、操作类型（创建/更新/删除）

它的核心职责是提供可靠的本地存储，支持事务和索引查询。任务管理的列表、详情、状态变更，都应落在这里。

### Service Worker：网络的代理层

Service Worker 不是用来存数据的，而是用来拦截网络请求，决定“什么时候走网络，什么时候走缓存”。

在离线优先模式下，Service Worker 的策略通常是：

- 写操作（POST/PUT/DELETE）：尝试网络，失败则交给后台同步机制
- 读操作：优先从本地 IndexedDB 读取，网络只作为“刷新参考”
- 静态资源：Cache First 策略确保应用本身能离线加载

### 后台同步与重试队列：数据的搬运工

这是离线优先的“发动机”。后台同步有两个关键角色：

- **同步队列**：当用户离线操作时，每个操作被序列化为一个“任务项”，按顺序排队。队列记录操作类型、目标 URL、请求体、本地记录的 ID。
- **重试机制**：网络恢复后，队列按 FIFO 顺序“回放”。失败的任务需要指数退避重试，同时保留失败原因和尝试次数。

注意：同步不是简单的“把所有本地数据推上去”。需要区分“需要同步的数据”和“仅在本地有效的数据”。例如任务的颜色标记、个人备注，可能不需要同步。

## 冲突场景：不只是“谁的后改谁赢”

冲突不是理论问题，而是日常。下面两个场景在任务管理中非常典型。

### 场景一：同一任务，多端同时编辑

用户 A 离线编辑了任务“设计登录页”，将优先级改为“高”；用户 B 在线编辑了同一个任务，将截止日期提前。

当 A 的设备恢复网络时，两个版本在服务器端产生冲突。简单的时间戳覆盖会导致一方数据丢失。

合理的策略是“基于字段的合并”：

- 优先级字段取 A 的“高”
- 截止日期取 B 的“提前”
- 如果同一个字段（如标题）两边都改了，服务器可以保留两个版本，或要求用户手动合并

这种策略要求本地记录知道“哪些字段被修改了”，而不是整个记录覆盖。

### 场景二：离线删除与在线更新交叉

A 离线删除了任务“采购清单”，B 在线给这个任务添加了一个备注“需要确认供应商”。

当 A 的设备同步时，服务器收到删除指令，但该任务还有关联数据。

处理方式可以是“软删除”：本地标记为删除，但服务器保留任务记录，删除请求作为“状态变更”处理。服务器需要判断：如果已有更远的修改，是否接受删除？通常采用“删除优先”或“提示冲突，让用户决定”的策略。无论如何，用户操作不能静默丢失。

## 简化的数据流：本地写入到最终确认

一个任务创建操作在离线优先设计中的典型路径：

1. **本地写入**：用户点击“创建任务”，数据立即写入 IndexedDB，生成本地 ID 和时间戳。UI 立即显示新任务。
2. **排队**：同时，一个同步任务被推入队列，包含操作类型、数据、本地 ID。
3. **尝试同步**：Service Worker 或后台同步脚本尝试发送请求。如果网络可用，发送成功。
4. **确认**：服务器返回正式 ID，更新本地记录的服务器 ID，标记该任务为“已同步”。
5. **失败**：如果网络不可用或请求失败，队列保留任务，标记为“待重试”。
6. **回滚**：如果请求多次失败（如服务器 500 错误），需要确认是否真的需要回滚。通常，更合理的做法是保留本地数据，让用户手动决定。

这里的关键是：本地写入是立即且可视的，同步是异步且可重试的。用户永远不需要等待网络。

## 可观测性：同步失败与冲突率不是玄学

离线优先应用需要回答四个问题：

- 有多少次同步失败？
- 冲突频率是多少？
- 平均恢复时间（从断网到完成同步）是多长？
- 数据最终一致了吗？

数据采集点：

- **同步失败日志**：每条同步任务都记录尝试次数、失败原因、时间戳。聚合后可以看到失败模式——是网络问题还是服务端问题？
- **冲突率**：每次冲突发生时记录元数据：冲突类型、触发用户、最终处理方式。冲突率越高，说明多端编辑场景越需要优化。
- **恢复时间指标**：从设备检测到网络恢复，到队列清空的时间。如果恢复时间异常长，可能是同步队列逻辑有问题，或服务端处理慢。
- **一致性检查**：定期比较本地数据与服务器数据的版本号差异，检测“未同步数据”或“残留冲突”。

这些指标可以在应用内埋点，上报到日志系统。投产后第一周，应该重点关注冲突率和恢复时间。如果冲突率超过 5%，就需要调整合并策略。

## 三点项目落地建议

1. **从“可中断”场景切入，不要一开始就追求完美一致性**
   选择那些用户对短暂不一致容忍度高的场景启动，例如任务状态变更、备注添加。不要一开始就处理复杂的多人同时编辑标题的场景。先跑通基础流程，再迭代冲突处理。

2. **本地数据模型要支持字段级差分，而不是记录级覆盖**
   IndexedDB 中的每条记录应当包含一个“修改字段列表”，而不是完整的对象。这既是冲突合并的基础，也是节省同步带宽的关键。设计 API 时，考虑支持 PATCH 而非 PUT。

3. **把同步状态暴露给用户，但保持克制**
   在 UI 上可以显示“待同步”、“同步中”、“同步失败”等状态，但不要打断用户操作。提供一个“同步状态面板”供查看详细日志，而不是在每个操作上弹框。用户需要的不是完美无缺的体验，而是可理解、可预测的体验。

离线优先不是技术炫技，而是对用户场景的深度理解。当你的应用能在火车隧道里、在地下室里、在弱网环境下正常工作，用户才会真正信任它。而这个信任，是所有好产品的起点。
