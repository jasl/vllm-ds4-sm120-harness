# 离线优先的任务管理：当“断网可用”成为信任基石

如果你做过面向C端的任务管理工具，大概率遇到过这样的用户反馈：“我明明标记了完成，怎么又变回未完成了？”或者“我出差高铁上编辑了任务，联网后数据全丢了”。这些问题的本质，不是用户操作失误，而是应用在离线状态下的数据一致性机制出了漏洞。离线优先，从来不是简单地把页面缓存下来就能解决的事，它涉及用户对数据的掌控感，以及系统在矛盾中做抉择的能力。

## 离线优先的真正命题：信任而非技术

很多团队把离线能力当作“网络不好时的兜底方案”，这是一种单向思维。真正成熟的离线优先设计，是把本地作为第一数据源，服务器作为同步对等方。用户不必担心自己离线时的操作会丢失，也不必在联网后遭遇数据回滚。这种信任感的建立，远比单纯的“断网能用”更复杂：你需要让用户确信，他的每一次点击、每一次拖拽排序，都被稳妥地记录在某处，并且能在联网后被尊重。

从技术架构上，这意味着一套以本地持久化、异步同步、冲突自治为核心的体系。下面我们从三个底层组件开始拆解。

## 三个核心角色的职责边界

### IndexedDB：本地事实的来源
IndexedDB 不是浏览器缓存，而是本地的结构化数据库。在离线优先应用中，它的职责是存储所有用户的业务数据——任务列表、状态、版本号、更新时间戳。每次用户操作，首先写入 IndexedDB，然后界面即时更新（乐观更新）。这样，即使用户关闭浏览器再打开，依然能看到最后的状态。要注意的是，IndexedDB 需要设计合理的版本号字段和变更日志表，为后续同步提供依据。

### Service Worker：请求的拦截与路由
Service Worker 更像一个代理层。它拦截所有发往后端的请求（包括API调用和静态资源），在网络可用时透传，在网络不可用或未准备好时缓存响应或返回本地数据。在离线优先场景下，Service Worker 与 IndexedDB 配合：当用户发起创建任务的请求时，Service Worker 不直接拦截这个HTTP请求（因为操作已经写入本地），而是异步触发一个同步事件，把本地变更推送到服务器。

### 后台同步与重试队列：可靠交付的保障
后台同步（Background Sync）是让离线操作在连接恢复后自动“重放”的机制。但仅仅靠系统级别的同步事件不够，我们需要一个应用级别的重试队列。这个队列记录每条待同步的变更操作（操作类型、关联ID、请求体、时间戳），并在每次同步失败时递增重试次数、记录失败原因、计算下一次重试时间。当重试次数超过阈值（比如5次）或用户主动标记为冲突时，队列将该操作标记为“冲突待处理”，不再自动重试，等待用户介入。

## 冲突场景：当离线与在线无法“自动合并”

### 场景一：多端同时编辑同一个任务
假设用户在手机端离线时，把任务A的截止日期改为明天；同时，他的同事在网页端把任务A的截止日期改为后天。当手机重连时，两条变更都被推送到服务器。服务器需要依据某种策略裁定：大多数系统使用“最后写入胜出”（LWW），但这会导致后到达的变更覆盖先到达的，而先到达的用户可能会觉得自己的操作被“吞了”。更好的做法是：记录每个字段的版本号，逐字段合并（比如截止日期以最后修改的版本为准，但备注内容可以两端合并）。如果无法自动合并（比如同一个文本字段在两侧都修改过），则生成一个冲突记录，UI上展示差异让用户手动选择。

### 场景二：离线删除与在线更新交叉
用户离线时删除了任务B，同时在线的另一个用户修改了任务B的标题。当同步发生时，服务器收到两条记录：一条是“删除任务B”，一条是“更新任务B的标题”。这两种操作在语义上互斥。一种策略是让删除优先（认为删除意味着用户完全不想再看到这个任务），更新自动失效；另一种策略是保留更新后的数据，但标记为“被删除状态”，让用户在界面上选择是否彻底删除或恢复。这取决于产品语义：是软删除还是硬删除。硬删除会丢失数据，软删除则容易堆积垃圾。需要在设计之初就决定，并在冲突处理中保持一致。

## 数据流：从本地写入到最终一致

一个典型的离线操作生命周期可以拆解为六步：

1. **本地写入**：用户在界面上修改任务状态（比如勾选完成），数据首先写入 IndexedDB 的 tasks 表，状态字段立即更新，同时向变更日志表插入一条记录（操作类型：UPDATE，任务ID，旧值，新值，本地时间戳）。
2. **乐观更新**：UI 立即反映新状态，用户感知不到任何延迟。
3. **排队**：Service Worker 检测到网络状态，若在线则直接发送请求；若离线，则将变更记录推入内部的重试队列（存在 IndexedDB 中，非内存）。
4. **同步**：当网络恢复，后台同步事件触发，重试队列逐条发送请求。每次请求携带操作记录的全部字段，服务器比对版本号。
5. **确认/失败**：服务器根据冲突策略处理，返回确认（状态码200）或冲突（状态码409）。若收到确认，本地将变更日志标记为“已同步”；若收到冲突，将操作放入“冲突待处理”区域，并尝试自动合并，合并失败则通知用户。
6. **回滚或重试**：若同步请求超时或被服务器拒绝（比如权限问题），根据重试策略：网络类错误自动重试（指数退避），业务类错误（如权限不足）记录日志并通知用户，不再自动重试。

这样，即使用户离线操作了100次，每次写入都是秒级响应，同步完全在后台不可见地完成。

## 可观测性：同步失败率、冲突率、平均恢复时间

离线优先系统最容易出现的问题是“静默失败”：用户以为数据同步成功，实际上永远停留在本地。因此可观测性必须覆盖三个关键指标：

- **同步失败率**：定义为一个时间窗口内（例如1小时）失败同步次数 / 总同步次数。失败包括HTTP错误、超时、服务器500。用这个指标判断网络稳定性或服务端压力。
- **冲突率**：定义为同时发生冲突的变更数 / 总变更数。如果冲突率超过5%，说明用户之间协作频繁，或者离线时间过长，需要考虑优化冲突合并策略（比如增加字段级别的版本控制，而不是整条记录锁定）。
- **平均恢复时间（MTTR）**：从操作写入本地开始，到服务器确认同步成功为止的平均耗时。这个指标直接反映用户数据“离安全”的时间有多长。如果某用户的MTTR超过24小时，说明他的设备长时间离线或网络极差，需要提示用户手动刷新或引导检查连接。

这些指标应该在运营面板上实时展示，并且按用户版本、网络类型（WiFi vs 蜂窝）做下钻。当同步失败率突然飙升，往往是 Service Worker 缓存了过期的同步凭证，或者服务器端的冲突检测逻辑引入了新Bug。

## 落地建议：三条实用的工程原则

1. **把同步日志当成产品功能来设计**。不要只在控制台打印日志，而是在界面上给用户一个“同步状态”入口，展示最近10次同步历史：成功、失败、冲突。让用户看到自己的数据正在被努力同步，而不是默默消失。
2. **为冲突率设置硬性上限**。如果系统冲突率超过10%，停止自动同步，转为手动确认模式。这样宁可让用户多一次点击，也不要让错误的数据自动覆盖。可以在冲突UI中提供“全部采纳本地”“全部采纳服务器”“逐条对比”三种选项。
3. **离线优先从第一天开始设计**。不要在项目中期“加上”离线功能。IndexedDB 的 schema、版本号字段、变更日志表、重试队列的数据结构，都需要在数据库设计阶段就预留。后期加的离线，往往是越想兼容现有 API 越容易出Bug，最终变成“看起来离线可用，实际不敢用”。
