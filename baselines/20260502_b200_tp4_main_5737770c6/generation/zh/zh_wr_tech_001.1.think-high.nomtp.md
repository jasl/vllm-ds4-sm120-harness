# 离线优先：为轻量级任务管理应用构建值得信赖的“双轨制”

当用户对你的任务管理应用说“我信任它，即使没网它也不会搞丢我的数据”时，才是离线优先架构真正生效的时刻。离线优先（Offline First）并非仅仅在本地缓存数据、断网时显示灰色界面——它是一场关于**数据主权与一致性共识**的架构选择。尤其对于轻量级任务管理这类频繁增删改、多端同步的场景，离线设计直接决定了用户是觉得“这个工具靠谱”还是“我宁愿用纸笔”。

## 为什么离线优先是信任杠杆

“断网也能用”解决的是可用性问题，但用户真正在意的是：**“我离线编辑的内容，联网后会不会被吃掉？”** 在线优先的应用假设网络永远可用，一旦断网，要么直接拒绝操作，要么在恢复网络后粗暴地用服务器数据覆盖本地。用户会逐渐养成“每次修改前先检查网络”的心理负担，这是信任流失的开端。

离线优先的价值在于：**让客户端成为一等公民**。本地设备承担写操作的即时响应，服务器仅作同步协调方。用户的操作先落本地，再异步与云端对齐。这种模型下，用户感知到的不是“断网时的次等体验”，而是“无论何时，我的操作立即被记录，同步只是后端需要解决的问题”。

信任还来源于**数据一致性承诺**。用户需要知道：两台设备同时修改同一个任务标题，最终结果不会丢失任何一条有效意图。这需要设计明确的冲突解决策略，而不是简单地“后到者覆盖”。

## 核心技术栈的三层职责

在浏览器环境中，实现离线优先依赖三个基础设施：

**IndexedDB** 承担“本地数据库”角色。与 localStorage 不同，IndexedDB 支持结构化数据存储、索引查询和事务。任务管理的 CRUD 操作应直接写入 IndexedDB，而不是作为网络请求的缓存。它是离线的数据基石，保证即使 Service Worker 被销毁，数据依然持久。

**Service Worker** 是网络代理层。它拦截请求，实现“优先从缓存读取、网络仅作校验”的策略。更重要的是，Service Worker 的生命周期独立于页面，它可以管理后台同步注册（Background Sync API），在设备恢复网络时自动触发同步流程，而不需要用户打开浏览器。

**后台同步/重试队列** 是同步的核心调度器。每个本地写操作首先落库，同时生成一条“同步记录”进入队列。队列记录操作类型（增/删/改）、任务 ID、本地时间戳和变更数据快照。Service Worker 利用 `SyncManager` 注册同步事件，或在离线时由队列自身进行指数退避重试。队列的设计要处理“幂等性”——同样的操作反复执行不应产生副作用。

## 冲突场景：当离线编辑变成“同期赛跑”

### 场景一：多端离线编辑同一任务

用户 A 在手机端将任务“写周报”的截止时间改为周五 18:00，用户 B 在笔记本端同步将该截止时间改为周日 12:00，两人都在离线状态下。当两台设备先后恢复网络，同步系统收到两条来自同一任务的冲突更新。

解决方案是**基于操作日志的合并策略**。如果业务允许（如截止时间可直接覆盖），可采用“最后写入胜出”并记录冲突历史。如果业务敏感（如任务标题被不同人修改），则应保留两版本并标记冲突，在 UI 中提示用户手动合并。更优雅的做法是**基于“意图合并”**：如果用户 A 改了截止时间且用户 B 改了标题，则两项变更分别合并，无需冲突提示。

### 场景二：离线删除与在线更新交叉

用户离线时删除了任务“整理报销单”，而服务器端用户在此时将该任务的状态改为“已完成”。当离线用户的同步请求到达服务器，服务器发现“删除对象”与“更新对象”指向同一 ID。

这里必须引入**墓碑机制**。删除操作不能真正抹除记录，而是在本地和服务器各保留一条标记为“已删除”的墓碑记录（包含原始 ID 和删除时间戳）。服务器收到删除同步时，检查目标记录是否存在：若存在且无更新，则执行删除并生成墓碑；若存在且已有更新，则根据业务规则决定是“删除优先”还是“驳回删除”。轻量级任务管理建议采用“删除优先”，因为用户删除的意图通常强于被动接收的更新。但这一切须记入操作日志，供后续审计。

## 简化数据流：从击键到一致性

1. **本地写入**：用户在界面修改任务名称，操作立即写入 IndexedDB 的 `tasks` 表和 `sync_logs` 表。UI 直接反映新状态，不等待网络。

2. **排队**：`sync_logs` 中新增一条记录，包含 `{operation: 'update', taskId: 'abc', data: {title: '新名称'}, localTimestamp: T, status: 'pending'}`。

3. **同步触发**：Service Worker 检测到网络恢复（或浏览器触发 `sync` 事件），读取队列中 `status='pending'` 的记录，按时间顺序向服务器发起请求。每条请求携带唯一 `operationId` 用于幂等防重。

4. **确认**：服务器处理成功后，返回包含 `serverTimestamp` 的确认响应。客户端将相应同步记录标记为 `confirmed`，更新任务的同步版本号。

5. **失败回滚或重试**：服务器可能因冲突或数据校验失败返回特定错误码。客户端根据错误码决定：临时性错误（如网络抖动）进入指数退避重试队列；冲突性错误（如版本号冲突）暂停当前同步，触发 UI 冲突提示；不可恢复错误（如数据格式违法）标记为 `failed`，记录错误详情，后续人工介入。

## 可观测性：让你的离线系统“透明化”

离线优先系统最怕“静默失败”——用户以为数据同步了，实际却丢失在队列深渊。可观测性应覆盖以下指标：

- **同步失败率**：每日/每周 `sync_logs` 中 `status` 为 `failed` 的比例。如果超过阈值（如 0.5%），说明冲突策略或幂等设计有问题。
- **冲突率**：服务器收到 `conflict` 状态操作的占比。若冲突率过高，可能需要优化合并策略或检查用户操作习惯（如频繁离线编辑同一任务）。
- **平均恢复时间（MTTR）**：从一条同步记录进入 `pending` 到变为 `confirmed` 的平均耗时。MTTR 突增可能意味着 Service Worker 同步触发异常或服务器处理过慢。
- **队列积压深度**：`status='pending'` 的记录总量。若积压持续超过百条，说明同步吞吐能力不足。

这些数据可以通过定时分析 IndexedDB 的 `sync_logs` 表生成，或由服务器端统计同步请求的响应时间与结果分布。推荐在开发者工具或管理后台暴露“同步健康面板”，让开发者和高级用户直观看到系统状态。

## 落地建议

1. **从“增量同步”起步，放弃“全量覆盖”**：不要设计“联网时拉取服务器全部数据覆盖本地”的同步逻辑。只同步变更条目，降低冲突概率与传输量。本地保有一份独立 ID 生成与版本号管理机制，避免依赖云端的自增主键。

2. **为冲突留一条“人工通道”**：自动合并策略无法覆盖所有业务场景。在 UI 中为冲突任务提供“查看历史版本”与“手动选择保留哪个版本”的交互入口。预留基于操作日志的“回滚”能力，允许用户在发现同步异常后恢复至某个历史节点。

3. **宁可慢，不可静默丢弃**：重试队列使用指数退避算法（如初始 1 秒、后续 30 秒、最长 5 分钟），避免网络恢复时瞬间冲击服务器。任何同步异常都应记录到 IndexedDB 的同步日志表，并在用户下次打开应用时以非阻塞方式提示“共 X 条变更未同步，点击查看详情”，而不是偷偷丢弃。

离线优先不是让断网体验“尚可”，而是让网络变成应用的非必要条件。当用户能理直气壮地说“没网我也能干活，数据不会丢”，你的应用才会从工具进化为可信赖的伙伴。
