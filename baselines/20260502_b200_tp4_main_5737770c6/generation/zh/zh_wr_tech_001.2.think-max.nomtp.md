# 离线优先：如何设计一款值得信任的轻量级任务管理应用

## 为什么离线优先不只是“断网也能用”

许多工程师对离线优先的理解停留在“本地缓存 + 断网提示”的层面。然而，真正有意义的离线优先，核心在于**用户信任**和**数据一致性**。

想象一个场景：用户在地铁里创建了几个任务，修改了截止日期，删除了一项待办，然后进入隧道。十分钟后信号恢复，这些操作需要可靠地同步到服务端。如果出现数据丢失或错误，用户将不会再信任这个应用。离线优先设计的根本目标，不是让应用在断网时勉强可用，而是让用户**完全不必关心网络状态**——无论在线与否，体验一致，数据最终一致。

这种信任建立在两个前提之上：第一，所有本地操作即时生效，不等待网络；第二，网络恢复后，数据能正确合并，不产生意外的覆盖或丢失。这就引出了我们需要深入理解的三项关键技术。

## 三驾马车：IndexedDB、Service Worker 与后台同步

### IndexedDB：本地持久化的基石

IndexedDB 是离线优先的“硬盘”。它提供结构化的本地存储，能够存储完整的任务对象、元数据、时间戳和操作版本号。与 localStorage 相比，IndexedDB 支持复杂的查询、事务和索引，特别适合需要离线增删改查的场景。

每个任务存储时都应该附带一个逻辑时钟（比如本地的递增序列号或 Lamport 时间戳），用于后续判断操作的先后关系。IndexedDB 还负责维护一个“未同步操作队列”——这是整个系统可靠性的起点。

### Service Worker：网络代理与离线资源管理

Service Worker 承担两个职责：一是拦截网络请求，配合 Cache Storage 实现静态资源的离线可用；二是作为后台同步的触发者。通过 Service Worker 的 `SyncManager` 接口，可以注册一个“同步事件”，浏览器会在网络可用时自动触发，即使应用页面已经关闭。

这里有一个容易被忽视的细节：Service Worker 不直接与 IndexedDB 交互（虽然技术上可以），更常见的架构是让 Service Worker 转发同步事件到主线程或 Web Worker，由后者读取 IndexedDB 中的未同步队列，执行网络请求。这样做的好处是职责分离，Service Worker 专注于生命周期和网络事件，业务逻辑留在应用层。

### 后台同步与重试队列：可靠性的最后一道防线

后台同步需要配合重试队列设计。一个典型的实现是：每次同步失败，不丢弃操作，而是将操作写入 IndexedDB 的“重试队列”，带上重试次数、上次失败原因、指数退避的等待时间。当达到最大重试次数（例如 5 次或 10 次）后，将操作标记为“同步失败”，等待用户或管理员干预。

重试队列中还要考虑“依赖关系”——如果任务 A 的创建尚未同步，任务 B 的移动操作应当暂缓发送，直到 A 确认成功。这可以通过在队列中维护一个“先决条件列表”来实现。

## 两类典型冲突：当离线与在线交错

即使有完善的同步流程，冲突也不可避免。以下是两种常见场景：

### 场景一：同一任务在多端同时编辑

用户在手机端离线时将任务标题从“购买食材”改为“购买水果”，同时在电脑端在线时将同一任务的优先级改为“高”。两个操作均基于旧版本产生。当手机端恢复同步时，服务端收到两条冲突的操作。

解决方案：采用“最后写入者胜出”策略显然不够好——它会丢失用户的意图。更好的做法是“基于字段的合并”：标题和优先级是不同的字段，可以各自合并，只要没有修改同一字段。如果同一字段冲突，可以采用操作转换（OT）或冲突标注：保留两个版本，在 UI 上提示用户选择，或者记录该字段的历史版本供用户回溯。

### 场景二：离线删除与在线更新交叉

用户在离线时删除了任务 A，但同一任务在服务器端已被另一个协作者更新了截止日期。当删除操作同步时，服务端发现该任务已不存在于用户的期望状态中。

这里的核心原则是：**删除操作应被视为一种“墓碑”**。如果服务端发现任务处于更新状态，可以采取两种策略：一是记录删除操作并标记为冲突，人工判断；二是按照“删除优先”处理，因为删除的意图是明确的。我倾向于第二种，但对于重要数据，建议采用“软删除 + 版本比对”，让用户有恢复的机会。

关键点在于：冲突解决策略需要在设计之初就确定，并在同步协议中明确传递。用户应能看到冲突记录，而不是默默丢失数据。

## 简化数据流：从本地写入到最终一致

一个典型的离线优先数据流如下：

1. **本地写入**：用户创建或修改任务，立即写入 IndexedDB，UI 更新为“已保存”状态（但显示一个小同步图标表示尚未同步）。同时，该操作被追加到“未同步队列”中，携带操作类型、目标对象 ID、修改字段、本地时间戳和当前数据版本号。

2. **排队**：Service Worker 注册后台同步事件。如果浏览器支持，立即触发“同步”请求；否则，等待下一次页面加载或网络变化事件。

3. **同步**：后台同步事件触发后，主线程或 Web Worker 从 IndexedDB 中读取未同步队列，按依赖顺序依次发送到服务端 API。每次请求携带操作的时间戳和本地版本号。

4. **确认**：服务端收到请求后，比对本地版本与服务端最新版本。如果无冲突，应用变更，返回“成功”响应及新的版本号。客户端更新 IndexedDB 中该操作的状态为“已同步”，同时更新本地版本号。

5. **失败回滚或重试**：如果服务端检测到冲突，返回冲突详情（如当前服务端版本）。客户端可以选择：自动合并（如字段级合并），或将冲突记录标记给用户。如果是网络故障（如 500、超时），将操作放入重试队列，按指数退避策略等待重试。达到最大次数后，标记为“同步失败”并报警。

这个流程的核心在于：**本地写入从不等待网络，所有同步都是事后异步完成**。用户感知到的永远是“瞬间完成”，同步状态通过 UI 非侵入式展示。

## 可观测性：让同步状态可量化

要维护一个离线优先应用，必须能回答三个问题：同步失败率有多高？冲突频率是多少？系统从故障中恢复需要多久？

### 记录同步失败

在重试队列中，每次失败应记录：失败时间、HTTP 状态码、错误信息、设备 ID、操作类型。聚合后可以分析出问题根因是网络问题、服务端 bug 还是数据异常。

### 冲突率监控

服务端每次检测到冲突，应记录冲突类型（字段冲突、删除冲突、版本冲突等）、发生时间、最终解决方式（自动合并、人工干预、覆盖等）。冲突率异常升高往往意味着设计缺陷或用户行为模式变化。

### 平均恢复时间

对于失败的同步，从第一次提交到最终成功或宣告失败的时间，就是恢复时间。这个指标反映了系统的韧性：如果大部分失败操作在 1 分钟内重试成功，说明网络短暂波动；如果部分操作始终失败，则需要人工介入。

建议将这些指标打点到集中监控系统（如 Prometheus 或 Grafana），设置报警阈值：例如同步失败率超过 1% 触发告警，冲突率异常升高提示检查同步逻辑。

## 项目落地建议

1. **优先设计冲突策略**：不要等冲突出现再修补。在项目初期，就明确你是采用“最后写入者胜出”、“基于字段合并”还是“手动解决”。选择取决于数据的重要性和用户场景。任务管理建议采用字段级合并 + 冲突标记，兼顾自动化和用户知情权。

2. **建立操作日志，而非状态快照**：离线优先系统的核心是操作序列，不是最终状态。每当用户执行操作，记录下做了什么、何时做的、基于哪个版本。这不仅是冲突解决的基础，也是审计和回滚的依据。再小型的应用，也值得维护这样一份“事件流”。

3. **从 MVP 就开始做可观测性**：不要等到上线才发现同步问题。在开发阶段，就内置失败日志和冲突计数器。压力测试时可以模拟 10% 数据包丢失、随机延迟，观察系统表现。离线优先的信任，建立在可观察、可度量的基础之上，而不是“应该没问题”的假设。

离线优先不是一种锦上添花的能力，而是一种需要精心设计的系统工程。它让用户的设备真正成为他们的工作空间，网络只是一种用来同步的管道，而非前提条件。对于任何面向个人或小团队的生产力工具，这种设计哲学带来的用户粘性，远超花哨的功能列表。
