# 离线优先任务管理：当“断网可用”成为信任的起点

在一个协同办公日益普及的时代，用户对应用的期待已经从“在线可用”转变为“任何时候都能连续工作”。对于任务管理这类高频操作场景，离线优先（Offline-First）不再只是一个技术噱头，而是一种建立用户信任的核心策略。但实现这一目标远非“本地缓存+Service Worker缓存页面”那么简单——真正的挑战在于数据一致性、冲突处理和系统可观测性。

## 离线优先的本质：信任比功能更重要

许多工程师将离线优先理解为一个网络降级方案——断网时读取缓存数据，联网后拉取最新。但用户真正在意的不是“能不能看到页面”，而是“我离线时的操作是否被尊重”。一个常见的失败案例是：用户在机场无网环境下标记了五个任务为“已完成”，恢复网络后，应用却因为拉取了服务器端旧数据而将其重置为“未完成”。这摧毁的不仅是数据，更是用户对应用可靠性的信任。

离线优先设计的核心假设是：**本地操作应被视为优先级最高的数据来源**。服务端不再是权威的“真理源”，而是分布式系统中一个对等节点。这种转变要求我们从架构层面解耦“数据写入”与“网络确认”。

## 三件核心武器：IndexedDB、Service Worker与后台同步

在轻量级任务管理应用中，我们不需要复杂的事务日志或CRDT（无冲突复制数据类型），但必须理解三个基础设施的职责分界。

**IndexedDB**是本地数据中枢。它存储所有任务实体、操作时间戳和同步状态标记。关键在于，它不仅要记录当前数据状态，还要记录“待同步的操作队列”。每个本地写入都对应一个操作记录——创建任务、更新标题、删除任务——并附上本地时间戳和操作类型。这使得即使在离线环境下，应用也能构建完整的数据变更历史。

**Service Worker**负责掌管网络状态。很多人以为Service Worker只做资源缓存，实际上它在离线优先中扮演网关角色。它可以拦截所有API请求，在检测到网络不可达时将请求转向本地IndexedDB中的操作队列，而不是盲目尝试发送导致失败。同时，它可以通过`navigator.onLine`和`fetch`事件的异常检测，通知主线程切换至离线UI模式。

**后台同步（Background Sync）** 是排队的保障。当设备恢复联网后，Service Worker触发`sync`事件，将队列中的操作按照时间顺序依次发送到服务端。更关键的机制是**重试队列**——每一次同步失败，操作不应丢弃，而是根据失败类型（如409冲突、500服务错误）设定延迟重试策略。例如网络超时重试3次，每次间隔指数递增；冲突则转入人工处理流程。

## 真实的冲突：不仅仅是“谁最后写入”

冲突场景往往被过度简化。让我们聚焦两类典型问题。

**场景一：同一任务多端编辑。** 用户在手机端将任务标题从“设计原型”改为“设计高保真原型”，同时在桌面端将任务优先级从“高”改为“紧急”。当两端的写操作同步到服务端时，服务端需要决定是“后写覆盖前写”还是“合并字段”。简单的时间戳对比会导致信息丢失——优先级变更被标题变更覆盖。合理的策略是字段级合并：服务端根据操作记录对比每个字段的最后写入时间，保留针对每个字段的最新值。对于轻量级场景，这比全量覆盖更可靠。

**场景二：离线删除与在线更新交叉。** 用户A离线删除了任务X，而用户B在线更新了任务X的截止日期。当A恢复网络后，删除操作同步到服务端。服务端如何处理？直接执行删除会丢失B的更新；拒绝删除又会导致A的本地经历不一致。一个可行方案是：将删除操作视为“软删除”，并创建一个冲突记录。在任务列表中，向用户A显示“任务X已被您删除，但有其他用户更新了它，请确认是否永久删除？”这种设计虽然增加了交互复杂度，但保护了用户的操作意图。

## 简化数据流：从写入到最终一致

一个典型的离线优先数据流如下：

1. **本地写入**：用户在表单中提交变更。数据立即写入IndexedDB，操作记录进入“待同步”队列。UI即时更新，无需等待网络。
2. **排队**：Service Worker检测到设备离线，将同步请求暂存到`sync`注册表。此时数据存在于本地，但尚未到达服务端。
3. **同步触发**：设备恢复联网（或用户手动触发），Service Worker遍历“待同步”队列，对每个操作发送带本地时间戳的请求。
4. **确认**：服务端处理请求，若成功则返回当前数据版本号；若失败则返回冲突类型。
5. **失败处理**：对于可重试的失败（超时、429限流），操作重入队列等待下一次同步。对于不可重试的失败（409冲突），客户端标记该操作状态为“冲突”，并切换到用户手动解决模式。
6. **回滚**：当冲突无法自动解决时，本地数据回滚到上一次服务端确认的基线状态，并提示用户手动处理。

## 可观测性：让沉默的冲突说话

离线优先系统一旦部署，最大的风险是“工程师不知道它在出问题”。用户可能默默地忍受数据不一致，直到投诉爆发。因此，可观测性需要覆盖三个关键指标：

- **同步失败率**：记录每次同步操作的结果。如果某类操作的失败率超过5%，应立即报警。这通常意味着服务端API变动或数据格式不兼容。
- **冲突率**：统计“需用户介入解决的冲突”占总同步操作的比例。冲突率上升可能意味着多人同时操作密集，或时间戳精度不足（如两个操作发生在同一毫秒）。
- **平均恢复时间（MTTR）**：从操作入队到用户确认冲突解决的平均时间。如果这个数值超过24小时，说明冲突解决流程过于复杂，需要优化UI引导或引入自动合并策略。

实现层面，可以在IndexedDB中增加“同步日志”表，记录每次操作的时间、结果、冲突类型和解决时间。这些数据在用户授权后，通过后台同步定期上传到分析服务，而无需实时网络。

## 落地建议：从现实出发

1. **优先保证“单用户跨设备”场景的离线一致性**。多用户实时协同的冲突解决对UI和逻辑复杂度要求极高。对于轻量级任务管理，先确保同一用户在不同设备（手机、笔记本）间的数据一致，再逐步引入团队协作功能。

2. **为“冲突解决”做显式UI设计**。不要隐藏冲突，不要自动选择胜者。在任务详情页增加“同步状态”指示器（绿色=已同步，黄色=待同步，红色=冲突），并提供一个“查看更改”面板，让用户逐条对比并选择保留哪个版本。

3. **为“不可见”的同步建立容错预算**。在服务端设置写操作幂等性（使用本地生成的UUID而非服务端自增ID），在客户端设置同步过期时间（例如超过7天的未同步操作自动转为“过期”状态，提示用户手动处理）。这能防止积压的离线操作在恢复网络后引发雪崩式冲突。

离线优先不是一劳永逸的架构选择，而是对用户时间和信任的持续投资。当你的应用能够诚实地说“即使你没网，你的操作也不会丢失”，用户才会放心地把日常工作交给你。
