# 离线优先任务管理：从“断网能用”到“数据信任”

## 一、离线优先的真正代价

大多数开发者听到“离线优先”，第一反应是“断网时还能操作”。但如果你做过两年以上前端工程化，就会明白这远远不够。离线优先的本质是在网络不稳定的环境下，维持**用户对数据完整性的信任**。用户每在离线状态下创建一条任务，就默认为“这条数据不会丢”。一旦同步后出现丢失或错乱，用户对产品的信任就会崩塌，而且很难修复。

这正是为何说“离线优先”不是功能加法，而是一种**数据治理策略**。它需要在本地和远程之间建立起一套可预测、可审计、可回滚的数据流动协议。

## 二、三道技术防线及其职责

要构建这样一个系统，需要三个核心组件协同工作，我把它们称为“离线三件套”：

### 1. IndexedDB — 本地持久化仓库
它负责存储用户的全部操作记录和当前状态。不是简单缓存，而是作为**离线时的主数据源**。任务列表、已完成状态、标签、备注等，都需要用事务保证原子性。你还需要存储一个“同步日志”表，记录每条数据最后一次成功同步的时间戳，这是后续冲突检测的基础。

### 2. Service Worker — 网络代理与离线拦截
它的角色是**请求代理**。在线时，它拦截前端发出的 API 请求，先尝试发往服务器；失败后缓存请求体，并通知主线程“需要存入队列”。离线时，它直接返回 IndexedDB 中的数据给 UI 层，保证用户看到的始终是最新本地状态。关键在于：Service Worker 不负责业务逻辑，它只做“能不能发出去”的判断。

### 3. 后台同步与重试队列 — 一致性保证者
这是最容易被轻视的部分。后台同步（Background Sync API）只是触发条件，真正的重试逻辑需要一个**指数退避的队列**。每次同步失败，记录失败次数和错误信息，下一次重试间隔增长。队列中每条记录包含：操作类型（create/update/delete）、数据版本号（由本地递增生成）、操作时间戳。当连续失败超过阈值（例如 5 次），将任务标记为“手动解决”，不再自动重试。

## 三、两种典型的冲突场景

冲突不是小概率事件，在多终端场景下几乎是必然发生的。以下两种最具代表性：

### 场景一：同一任务在多端被编辑
用户 A 离线时修改了任务标题为“买牛奶”，用户 B 在同一时刻在线修改为“买牛奶+面包”。同步时，两边的版本号都是 1，但内容不同。处理方式不应是“后同步的覆盖前者”，而应基于**语义合并**或**版本向量**。比较合理的做法是：如果字段级可合并（比如 A 改了 title，B 改了 dueDate），就合并；如果改的是同一字段，保留冲突标记，推送给用户手动选择。

### 场景二：离线删除与在线更新交叉
用户在离线状态下删除了任务 T，同时在线侧另一个用户更新了 T 的优先级。同步时，先收到删除操作？还是先收到更新操作？如果顺序处理不当，可能删除动作执行后，更新操作试图修改已删除的数据，导致数据空洞。解决方法是引入 **“墓碑标记”** ：删除操作不真正擦除记录，而是标记为 deleted 并保留同步状态。这样后续进来的更新可以侦测到“数据已被删除”，进而丢弃或者告警。

## 四、简化数据流：写入、排队、同步、确认

整个数据流可以抽象为四个阶段：

1. **本地写入**：用户操作先写入 IndexedDB，状态立即反映在 UI 上。同时生成一条“待同步”记录，包含操作类型和本地生成的时间戳。

2. **排队**：Service Worker 检测到网络状态变化（Online 事件触发），或定时轮询（如每 30 秒），从队列中取出未同步的记录。

3. **同步**：按时间顺序逐个发送给服务端。每个请求携带版本号和服务端上次确认的时间戳。服务端进行冲突检测后返回结果（成功、冲突、失败）。

4. **确认 / 回滚或重试**：收到成功响应后，将本地记录标记为“已同步”。收到冲突响应时，根据策略执行字段级合并或标记冲突。收到网络或服务端错误时，将记录放回重试队列。

## 五、可观测性：你不能管理你无法测量的东西

离线优先系统最怕“静默失败”。用户以为数据已同步，实际上永远留在了本地。因此必须内置可观测性指标：

- **同步失败率**：总同步尝试次数中，失败的占比。可以按操作类型（c/u/d）分别统计，定位哪类操作最容易失败。
- **冲突率**：同步过程中发生冲突（无法自动合并）的次数。这是一个产品健康度指标，高冲突率意味着你的冲突策略需要调整。
- **平均恢复时间（MTTR）**：从一次同步失败到解决之间的时间。如果 MTTR 超过 24 小时，考虑增加推送通知或邮件提醒。

这些指标可以在同步模块中埋点，通过后端日志或 APM 工具收集。指标不是为了监控而监控，而是用来驱动优化决策——比如发现删除操作失败率突然上升，可能是服务端数据路由有问题。

## 六、三条落地建议

1. **优先做“离线写”，再考虑“离线读”** 很多团队先做离线浏览，但用户更关注“我离线写的内容会不会丢”。先实现本地写入和同步队列，确保离线数据不丢失，再优化离线时的数据展示。

2. **版本号上锁，别依赖时间戳** 不同设备的时间可能不同，用时间戳做冲突检测不靠谱。建议用单调递增的版本号（服务端分配，本地缓存），每次同步时比对。

3. **给用户一个“同步状态指示器”** 在 UI 角落里放一个小图标：绿色表示已同步，黄色表示等待同步，红色表示有冲突待解决。用户知道系统“记住了什么、忘了什么”，比任何技术承诺都更能建立信任。
