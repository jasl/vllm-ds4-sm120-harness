# 离线优先：如何构建用户真正信任的轻量级任务管理应用

作为前端工程师，我们习惯性地把网络当作默认存在的前提。但当用户在地铁、电梯或山里打开你的任务管理应用时，“网络不可用”不应该等于“应用不可用”。离线优先不是锦上添花的技术特性，而是一种关于数据主权和用户信任的设计哲学。

## 离线优先的本质：信任比功能更重要

很多团队把离线能力等同于“断网时能显示缓存页面”，这是典型的误解。真正的离线优先意味着：用户在任何时刻对数据的操作都应该被尊重，无论网络是否存在。

这里涉及一个被忽视的心理学问题——**操作的可逆性与确定性**。当用户离线编辑了一个任务标题，他期望的是“这个修改一定会被保存”，而不是“等网络好了再看”。如果应用在联网后悄无声息地丢弃了用户的离线修改，用户将失去对系统的信任。这种信任一旦破裂，比某个功能缺失更难修复。

数据一致性是另一个核心挑战。在传统在线应用中，服务器是真理的唯一来源。离线优先则要求本地设备也持有完整的真理片段，并在网络恢复后与服务器达成共识。这不是简单的“覆盖”，而是需要设计显式的冲突解决策略。

## 三大技术支柱的分工

### IndexedDB：本地数据库，不只是缓存

IndexedDB 是离线优先应用的基石。它提供了结构化的、持久化的本地存储，能够存储完整的任务对象、元数据和变更日志。

区别于 localStorage 或 sessionStorage，IndexedDB 支持索引、事务和游标遍历，这意味着你可以按创建时间、更新时间或状态高效查询任务。更重要的是，它能存储 Blob 和 ArrayBuffer，所以未来的附件功能——比如给任务添加图片——也有扩展空间。

在任务管理应用中，IndexedDB 承担的角色是**离线时的数据主权容器**。用户的所有 CRUD 操作首先写入 IndexedDB，应用界面读取的也是本地数据，这让应用在网络断开时依然保持完整功能。

### Service Worker：网络管家与离线缓存

Service Worker 处于浏览器和网络之间，是离线体验的“看门人”。它拦截所有网络请求，并根据规则决定是返回缓存、从网络获取还是走 fallback 策略。

在任务管理场景中，Service Worker 主要做两件事：
- **API 请求的离线 fallback**：当 GET 请求失败时，返回 IndexedDB 中对应的本地数据。
- **资源文件的缓存**：CSS、JS、字体等静态资源被预缓存，确保应用本身能离线加载。

需要注意，Service Worker 不直接访问 IndexedDB（虽然有提案支持，但尚未广泛实现），它通常通过 postMessage 与页面主线程通信，由主线程操作 IndexedDB。

### 后台同步与重试队列：异步的“邮递员”

后台同步（Background Sync）是一项被低估的 API。它允许应用注册一个同步事件，当网络恢复时，即使浏览器标签页已被关闭，该事件也会被触发。

在任务管理应用中，后台同步扮演“异步提交者”的角色：用户离线创建的任务、修改的状态、删除的记录，都被序列化为一个变更队列存储在 IndexedDB 中。同步时，队列中的操作按顺序发送到服务器。

重试队列则需要实现指数退避策略：第一次失败后等待 5 秒，第二次 25 秒，第三次 125 秒……同时设定最大重试次数和手动触发重试 UI，以防用户等待太长时间。

## 冲突场景：不只是技术问题，也是产品问题

### 场景一：双端编辑同一任务

用户 A 在手机端离线状态下将任务“写周报”的截止日从周五改为周三。同时，用户 B 在线将同一个任务的负责人从 A 改为 B。网络恢复后，两份修改产生了冲突。

这里的冲突不是“谁的数据新”能解决的，因为两个修改作用于不同字段。解决方案通常采用**多版本并发控制（MVCC）**：服务器为每个任务维护版本号或最后修改时间戳。本地同步时，带上本地版本号，服务器判断是否存在冲突字段重叠。若不重叠，则合并；若重叠，则标记冲突，返回给客户端，让用户或预设策略（如“服务器优先”或“本地优先”）决定。

### 场景二：离线删除 vs 在线更新

用户 A 离线删除了一项任务。与此同时，用户 B 在线修改了该任务的内容并保存。当 A 同步删除操作时，服务器发现该任务已被 B 更新。

这里需要引入“软删除 + 墓碑”机制：本地的删除操作实际上是一条标记删除的变更记录。服务器收到后，如果该任务存在且未被其他用户修改，则执行删除；如果已被修改，则需要判断——是删除优先，还是更新优先？通常产品策略应该让用户选择，但一个更务实的做法是：将删除操作视为最高优先级，因为删除意图往往更明确。但需要记录一个冲突日志，告知双方用户发生了什么。

## 数据流全景：从本地写入到最终一致

一个典型的离线操作流程如下：

1. **本地写入**：用户在离线状态下修改任务标题。应用立即将新标题写入 IndexedDB 的“tasks”表，并在“sync_queue”表中插入一条记录：操作类型（update）、任务 ID、新字段值、本地时间戳。
2. **排队等待**：页面主线程检测到网络离线，同步队列静默等待。注册后台同步事件（若浏览器支持）。
3. **同步触发**：网络恢复，后台同步事件或页面监听的 online 事件触发同步流程。应用按创建顺序从 sync_queue 取出一批操作。
4. **发送请求**：依次向服务器 API 发送请求，每条请求携带本地版本号和变更内容。
5. **服务器确认**：服务器校验版本号，若无冲突则应用变更，返回新的版本号。应用收到 200 后，从 sync_queue 移除该记录，并更新 tasks 表中的版本号。
6. **冲突处理**：若服务器返回 409 Conflict，应用暂停后续同步，将冲突记录标记待处理，UI 弹出冲突解决面板，要求用户选择“保留本地”、“采用远程”或“手动合并”。
7. **回滚或重试**：网络中断导致的超时请求，应用将其重新放入队列，等待下一次同步。若重试超过最大次数（如 5 次），标记为“同步失败”，在任务列表中显示“同步异常”图标。

## 可观测性：量化离线系统的健康状况

离线系统的可观测性比在线系统更复杂，因为你不仅要监控后端，还要监控成千上万的客户端。

需要在客户端记录以下指标，并通过埋点上报：
- **同步成功率与失败率**：按操作类型（create、update、delete）统计，失败需附带失败原因（网络超时、冲突、服务端错误）。
- **冲突率**：总同步操作中发生冲突的比例。如果冲突率超过 5%，说明系统的版本控制策略或 UI 引导需要优化。
- **平均恢复时间（MTTR）**：从网络恢复（online 事件触发）到同步队列清空的平均时间。这个指标反映同步性能，若过长（如超过 30 秒），需要优化批量发送策略或减小单次负载。
- **队列深度与排队时长**：用户离线期间累积的未同步操作数量，以及最久未同步操作的时间。极端深度提示用户可能长期离线，需要考虑数据暂存限制策略。

以上指标应该用独立的后端接口收集，并设置告警。例如，当某一设备连续 3 次同步失败时，触发告警并主动推送修复版本的 Service Worker。

## 三条落地建议

1. **从一开始就设计离线策略，而不是后期补丁**。离线优先与数据同步逻辑应该深入应用的核心架构，而不是在“基本功能做完”之后用补丁式的方式添加。这意味着数据层（IndexedDB）、网络层（Service Worker）和 UI 层的状态管理从一开始就要考虑同步状态与冲突状态。

2. **冲突解决策略要产品化，而非纯技术化**。不要只在 console 里输出冲突日志。为冲突设计友好的 UI：显示本地版本和远程版本的对比，让用户做选择题。如果完全依赖“服务器优先”或“时间戳优先”的自动策略，用户会发现自己的修改被静默丢弃，从而失去信任。

3. **为每个操作保留幂等性和可追溯性**。同步队列中的每条记录都应包含唯一请求 ID（UUID），以便服务器识别重复提交。同时，保留完整的操作日志在本地，用户可以在“操作历史”中回查自己的离线操作何时被同步、是否遇到过冲突。

离线优先不是炫技，而是对用户场景的深刻理解。当你的应用在地铁隧道里依然响应流畅，当用户在山顶收到“同步完成”的轻快通知，你就知道，所有的工程投入都是值得的。
