# 离线优先的轻量级任务管理：从“断网能用”到“用户信任”

## 不只是技术问题：离线优先背后的契约

当用户在一个任务管理应用中添加一条待办事项，然后锁屏、走进电梯——他默认的假设是：这条数据“已经存在了”。但如果我们告诉他“等联网再同步”，他感受到的不是技术限制，而是信任裂缝。

离线优先（Offline-First）之所以重要，不只是让用户在断网时能操作，而是让**用户在任何网络状态下，都能对数据的存在和一致性建立心理安全感**。这一层信任，才是离线架构设计的真正目标。

## 三个核心部件的分工

在一个轻量级任务管理应用中，离线能力的实现依赖三个基础设施：

**IndexedDB** 负责“存储”。它是浏览器端的结构化数据库，能承载任务列表、元数据、操作日志。它的关键是：写入速度快、容量相对大、支持索引查询。你可以把它看作客户端的主数据库，所有本地操作都落在这里。

**Service Worker** 负责“拦截与缓存”。它是一个独立于页面生命周期的后台进程。当你发起 API 请求时，Service Worker 可以拦截它，优先返回缓存，同时在后台尝试联网更新。它不直接与 IndexedDB 交互，而是作为网络层的一个代理，管理“哪些请求可以绕过网络直接返回本地数据”。

**后台同步 / 重试队列** 负责“协调与恢复”。IndexedDB 存储了待同步的操作（例如“创建任务#123”），后台同步机制会在网络恢复时逐一执行这些操作。如果某次同步失败（比如服务器返回 409 冲突），重试队列会标记该操作为“失败”，并决定是回滚本地数据、还是保留等待人工解决。

三者的关系：IndexedDB 是数据底座，Service Worker 是网络缓存层，后台同步是状态协调者。它们互不替代，但协同完成“本地优先，云端最终一致”的流程。

## 两类典型冲突场景

离线优先最容易暴露问题的，不是“网络断了怎么办”，而是“网络恢复了但数据打架了怎么处理”。

**场景一：同一任务在多端被编辑**
用户 A 在手机离线时把任务“写周报”的截止日期改为“周五”，用户 B 在同一时间用电脑在线把同一任务改为“下周一”。当 A 的设备联网同步时，服务器发现该任务有两个不同的“最新版本”。这里需要明确的冲突解决策略：是按时间戳覆盖（最后写入者胜出），还是保留两份差异标记为“待确认”？对于轻量级任务管理，多数场景下“最后修改时间”是一个可接受的简单策略，但必须将这个冲突事件记录并通知用户。

**场景二：离线删除与在线更新交叉**
用户在离线时删除了任务“整理发票”，但同步前另一台设备（或同一账户的其他会话）在线更新了该任务的标题和备注。当删除操作同步到云端时，服务端面临一个选择：接受删除，丢弃之前的更新（数据丢失）；拒绝删除（删除无效）；或者将删除视为“标记为删除”并保留历史版本。对于任务管理，一个合理的处理是：离线删除作为一个“删除请求”排队，当云端发现该任务在上次同步后有更新，则拒绝删除并告知用户“该任务已被修改，请确认是否仍要删除”。用户得到反馈后，可以重新决定。

## 简化的数据流：本地优先，云端确认

一个典型的写入流程如下：

1. **本地写入**：用户新建或编辑任务，数据立即写入 IndexedDB，UI 无延迟更新。同时生成一条“操作日志”记录（例如 `{type: 'update', taskId: '123', field: 'title', value: '新标题'}`），也存进 IndexedDB 的同步队列。

2. **排队**：系统检查网络状态。如果在线，立即尝试同步；如果离线，操作留在队列中，等待下一次网络恢复或主动触发尝试。

3. **同步**：后台同步或周期性检查触发，从队列取出操作，向服务端发送请求。请求附带本地操作的时间戳或版本号。

4. **确认**：服务端处理成功，返回确认。本地将操作日志标记为“已同步”，并从同步队列移除。

5. **失败回滚或重试**：如果服务端返回冲突（409）或其他错误（比如任务已被删除），系统判断：若为临时性错误（如超时），进入指数退避重试；若为冲突，则回滚本地修改到服务器最新版本，并在 UI 中提示用户“你的修改未能同步，请查看冲突详情”。

这条数据流的核心原则是：**本地永远优先响应，云端只负责最终确认与冲突裁决**。

## 可观测性：不只是排障，更是信任仪表盘

离线系统最怕的是“数据丢了但没人知道”。因此必须建立可观测指标，至少涵盖三类：

- **同步失败率**：每分钟/每小时有多少同步操作失败。失败类型应细分：网络错误（不该告警）vs 冲突（需要关注）vs 服务端拒绝（需要人工介入）。
- **冲突率**：每个冲突事件的频率和分布。如果某用户频繁出现冲突，可能意味着他的操作模式（多端频繁切换）需要特别优化，或者冲突解决策略需要调整（比如从“最后写入者胜出”改为“即时双向合并”）。
- **平均恢复时间**：从一次同步失败到用户最终看到“同步成功”或“冲突已解决”的平均时长。这个指标直接反映离线系统的用户体验健康度。

这些数据既可以在开发者面板展示，也可以提供给用户一个“同步状态”页面，增加透明度。用户知道“系统知道我遇到冲突了，并在尝试解决”，本身就是一种信任建立。

## 三条落地建议

1. **从“乐观更新 + 静默失败”开始，而不是“全量离线”**。不要一开始就让整个应用离线可用。选择一个核心操作（比如新建任务、修改标题），只对这个操作实现乐观 UI 同步，其他操作在线执行。这样风险可控，但已经能让用户感受到“离线也能创建”的价值。

2. **冲突策略要预设，但不能默认**。不要在代码里硬编码“永远最后写入者胜出”。把冲突检测逻辑写清楚（比如基于时间戳、或基于任务版本号），但允许在不同模块中覆写。例如，优先级高的任务可以走“用户手动裁决”，普通任务自动按策略处理。

3. **给用户一个“同步状态提示”，而不是全自动静默**。不要在“断网”时只显示一个灰色图标。在任务旁边显示一个小点（● 表示已同步，◌ 表示等待同步，⚠ 表示冲突），让用户感知状态。自动同步当然好，但用户一旦发现“我说改了但没改”，就会立刻失去信任。让状态可见，比让一切自动更重要。

离线优先不是一场技术的炫技，而是一场关于信任的设计。当用户在任何角落打开任务列表，看到数据完好、状态清晰，他就不会再想“这东西到底存没存”。这才是离线体验的终极目标。
