# 离线优先：为任务管理应用构建值得信赖的同步架构

## 一、离线优先的真正挑战：不只是断网可用

当团队决定构建一款“离线优先”的任务管理应用时，最容易犯的错误是将它简单等同于“断网时能用”。实际上，离线优先的核心命题是：**在不可靠的网络环境中，用户仍然可以像在线一样工作，且数据不会丢失或错乱。**

从用户视角看，离线优先意味着一种信任——我在地铁里编辑的任务，到办公室后不应该消失；我在手机端标记完成的待办，在电脑端不应该再次出现。这种信任一旦被打破（例如同步后出现重复任务或丢失修改），用户会迅速放弃应用。

从技术视角看，离线优先要求我们重新思考数据流：不再是「用户操作→服务端→返回结果」，而是「用户操作→本地存储→尝试同步→冲突处理→最终一致」。这要求架构设计必须优先保障本地体验，再将同步逻辑作为后台任务处理。

## 二、三件核心工具的职责分配

### IndexedDB：离线场景的“本地数据库”

IndexedDB 是离线任务的存储基石。它的关键职责不是简单的键值缓存，而是：

- **存储任务的完整状态**：包括任务标题、描述、截止时间、附件引用等结构化的字段。
- **维护同步元数据**：每条本地记录都需要附带 `syncStatus` 字段（如 pending、synced、conflict）、`lastModified` 时间戳、以及 `clientId` 来标识修改来源。
- **支持事务与索引**：能按状态查询待同步的队列、按时间查询变更记录。

与 localStorage 或内存缓存相比，IndexedDB 能存储大量结构化数据，且异步 API 设计天然适合与 Service Worker 配合。

### Service Worker：网络代理与同步调度器

Service Worker 在离线优先中的角色远不止“拦截请求并返回缓存”。它承担了：

- **透明代理**：拦截所有 API 请求，尝试与远端同步。若网络可达，将本地变更推送给服务端；若不可达，记录失败并进入重试队列。
- **后台同步触发器**：注册 `sync` 事件，在浏览器断定网络恢复时自动触发同步任务。这可比轮询优雅得多。
- **工作线程隔离**：同步逻辑运行在独立线程，不阻塞主线程的 UI 渲染。用户操作照常写入 IndexedDB，后续同步由 SW 负责。

### 后台同步/重试队列：保证最终一致性

这是“离线优先”的发动机。一个简单的实现思路是：

- 所有写操作（创建、更新、删除）不直接发送网络请求，而是先写入 IndexedDB 的“操作日志表”，包含操作类型、时间戳、受影响的任务 ID、新旧数据快照。
- 一个队列处理模块（可在 Service Worker 中）持续从日志表中取出 pending 状态的记录，依次执行网络请求。
- 成功时，更新日志状态为 synced，同时更新任务记录的同步版本号。
- 失败时，根据错误类型处理：网络错误则保留并等待下次重试，冲突错误则标记为 conflict 交给用户处理。

## 三、两类典型的冲突场景

### 场景一：多端同时编辑同一任务

用户 A 在手机端修改任务标题为“准备演示 PPT”，用户 B 在网页端同时修改为“准备演示视频”。双方离线，各自写入本地 IndexedDB。当两台设备先后上线时，服务端收到两条针对同一任务（相同 taskId）的更新。

传统方案可能用“最后写入者获胜”（LWW），但这样会丢失其中一个用户的修改。更好的做法是：

- 服务端记录每次更新的增量（diff）和版本号。
- 当检测到版本冲突时，将两个版本的快照返回给客户端。
- 客户端展示冲突视图，让用户选择保留哪个版本，或手动合并。

### 场景二：离线删除与在线更新交叉

用户 A 离线时将任务“召开周会”标记为已删除。与此同时，用户 B 在线为该任务添加了备注“需提前准备议程”。当 A 的设备上线时，服务端已经存在基于 B 修改的新版本。

如果简单按时间戳处理，A 的删除操作可能会覆盖 B 的更新，导致内容丢失。合理的处理方式是：

- 删除操作不应直接删除记录，而是标记为 `deleted: true` 并保留最后版本。
- 冲突时，服务端返回“你正要删除的任务已被更新，是否确认删除？”的提示。
- 用户确认后，服务端才真正删除，且删除操作本身也作为一次变更记录到操作日志。

## 四、简化的数据流

```
用户操作（创建/编辑/删除）
        │
        ▼
写入 IndexedDB（本地立即生效，状态：pending）
        │
        ▼
Service Worker 监听到网络恢复 / 定时触发同步
        │
        ├─── 成功 → 更新 IndexedDB 状态为 synced
        │          发送确认通知（可选）
        │
        └─── 失败：
                ├── 网络错误 → 保留 pending，等待重试（指数退避）
                └── 冲突错误 → 状态标记为 conflict
                              通知用户 → 手动解决 → 重新提交
```

关键设计原则：**本地写入永远优先于网络响应**。用户点击“保存”后，任务立即出现在本地列表，不需要转圈等待。同步是后台的、可恢复的、用户可见（但非阻塞）的流程。

## 五、可观测性：如何衡量同步健康度

没有可观测性的离线应用，就像没有仪表盘的驾驶。你需要跟踪以下指标，并在应用内提供可视化面板或日志导出：

1. **同步失败率**：`失败次数 / 总同步尝试次数`。如果超过 5%，说明网络条件差或服务端接口有稳定性问题。
2. **冲突率**：`冲突次数 / 同步成功次数`。高冲突率提示多端并发编辑频繁，可能需要优化冲突检测策略（如实时协作的 CRDT 方案）。
3. **平均恢复时间**：从任务状态变为 pending 到最终 synced 的平均延迟。这是评估用户“信任感”的关键指标——超过 30 秒的延迟会影响体验。
4. **同步队列深度**：当前 pending 状态的任务数。队列越深，说明积压越多，可能需要调整重试策略或提升服务端吞吐。
5. **失败类型分布**：按 4xx 客户端错误、5xx 服务器错误、网络超时等分类。帮助定位是前端逻辑问题还是后端问题。

这些指标可以通过在每次同步操作后向日志系统（如 Sentry、自建监控）发送事件来实现，无需侵入业务代码太多。

## 六、三条项目落地建议

1. **从“乐观写入”开始，逐步过渡到离线优先**。先确保所有写操作都能本地即时生效，同步失败不阻塞用户。即使初期同步逻辑简陋（只是每隔 15 秒轮询），也比等到完美方案再发布好。用户习惯一旦形成基于在线的心态，后续改造成本极高。

2. **冲突处理设计要“默认保护用户劳动”**。当检测到冲突时，永远不要静默丢弃任何一方的修改。默认展示两个版本让用户选择，或者提供一个自动合并后的预览。宁可让用户手动解决一次冲突，也比偷偷覆盖数据后让用户愤怒强百倍。

3. **为测试投入至少 30% 的离线相关预算**。离线优先应用的传统“刷新页面看结果”的测试方式完全失效。你需要模拟断网、弱网（3G/2G 延迟）、网络抖动、服务端宕机等场景。使用 Chrome DevTools 的 Network Throttling、Mock Service Worker 拦截同步请求，或者直接写 E2E 测试脚本模拟完整的离线→同步→冲突流程。没有充分测试的离线逻辑，就是定时炸弹。
