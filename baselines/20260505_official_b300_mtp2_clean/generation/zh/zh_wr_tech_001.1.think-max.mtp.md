# 离线优先任务管理：不只为断网，更为信任

## 为什么离线优先不只是“断网也能用”

很多团队在第一次尝试离线优先时，会带着一个朴素的理解：让用户在没网的时候也能打开页面、看到数据、做点操作。这个理解没错，但只对了一半。

离线优先真正的挑战，是在网络恢复后，如何让用户觉得“一切如常”。用户离线时做了一个标记、改了一个截止时间、删除了一条任务——这些变动在本地设备上是正确的。但一旦多端同步，冲突就来了。用户最怕的不是断网，而是断网后自己做的事被“静默覆盖”或“神秘消失”。

从信任角度看，离线优先本质上是在回答一个问题：**当用户把工作交给你，你是否能在不确定的网络环境下，仍然保证他的劳动不丢失、意图不被扭曲。** 这不是技术上的“可用”，而是产品上的“可靠”。

## 三大基础设施：分工明确，各司其职

建立一个可靠的离线优先系统，需要三个基础组件协同工作。它们没有谁更重要，缺谁都会出问题。

### IndexedDB：本地的事实来源

IndexedDB 是离线数据的心脏。服务端的数据在本地有一份镜像，用户的每一次操作都首先写入 IndexedDB。这里的写入逻辑必须是**乐观的**——用户点了“标记完成”，界面立刻更新，数据写到本地存储，不需要等网络回来。

这里有一个容易被忽视的设计：本地数据的结构应尽量贴近服务端的 Schema，但可以多一些元数据字段，比如 `syncedAt`、`localUpdatedAt`、`syncStatus`。它们是后续冲突判断的基础材料。

### Service Worker：网络请求的交通警察

Service Worker 不是数据持久化的主角，它的核心任务是拦截网络请求，在离线时返回缓存，在线时决定是否走缓存优先还是网络优先。对于任务管理这类场景，我们通常采用“Cache First, Network Update”的策略：先给用户看本地缓存，同时在后台尝试拉取最新数据，不阻塞界面渲染。

Service Worker 还有一个容易被低估的作用：**代理推送事件和后台同步事件**。它是连接浏览器生命周期与你同步逻辑的桥梁。

### 后台同步与重试队列：不可靠网络下的可靠交付

用户的离线操作不能永远留在本地。后台同步机制负责在网络恢复后，把积压的操作按序发送到服务端。这听起来简单，但难点在于**顺序与幂等**。

假设用户离线时做了三个操作：改标题、加标签、改截止时间。这三个操作的发送顺序必须与执行顺序一致，否则服务端可能得到不一致的最终状态。

重试队列则负责处理失败场景。如果第一个操作失败了（比如权限不足），后面的操作不能盲目继续；如果失败原因是网络超时，可以指数退避重试。这里的关键是：**重试不应该是永无止境的，达到上限后要进入人工处理流程。**

## 冲突场景：离线优先真正的噩梦

无论同步逻辑设计得多精巧，冲突总会出现。以下是两类最常见也最棘手的场景。

### 场景一：同一任务在多端被编辑

用户 A 在手机上把任务标题从“完成前端联调”改为“完成前后端联调”，而用户 B 在电脑上同时把它改为“完成联调并部署”。两者都在离线状态，各自持有原始数据的快照。

当两端都在线后，服务端收到两条基于同一版本的修改请求。这里有一道经典选择题：**最后写入者获胜（Last Write Wins）还是合并？**

对于任务标题这种非结构化字段，合并几乎不可行。一个更务实的做法是：让服务端拒绝第二个写入，返回冲突信息，客户端弹出提示让用户手动选择。这不优雅，但诚实。用户宁可多操作一步，也不希望自己的修改被静默替代。

### 场景二：离线删除与在线更新交叉

用户 A 离线时删除了一个任务，同时用户 B 在线编辑了这个任务的描述。当 A 重新上线，删除操作被同步到服务端，B 的编辑被覆盖了。

这种情况下，一个可行的策略是软删除：标记为删除而非真实删除，保留一个“墓碑”记录。当冲突发生时，服务端可以判断：如果一条记录在被删除之后又有了编辑，说明删除可能不合理，应该通知相关用户。

## 数据流：从写入到确认，每一步都有状态

一个完整的离线数据流，可以简化为五个阶段：

1. **本地写入**：用户操作触发本地 IndexedDB 写入，界面即时更新，操作进入同步队列。
2. **排队**：同步队列中的每一条记录都包含操作类型、目标 ID、时间戳、操作前的数据快照（用于回滚）。
3. **同步**：网络可用时，后台同步脚本按队列顺序向 API 发送请求。每个请求携带 `clientVersion` 和 `lastServerVersion`。
4. **确认**：服务端返回 200 表示成功，本地更新该记录的 `syncedAt` 和 `syncStatus`。
5. **失败回滚或重试**：非幂等操作失败时，本地数据回滚至操作前状态，用户会看到操作被撤销的提示，同时错误信息被记录下来。

## 可观测性：离线优先系统的仪表盘

离线优先的系统比纯在线系统多了许多隐蔽的故障点。不暴露给用户不等于不存在。作为工程团队，至少要关注三个指标：

- **同步失败率**：失败的同步操作占总同步操作的比例。正常场景应在 1% 以下。如果某个 API 接口频繁失败，说明服务端可能没有正确处理幂等性或版本号。
- **冲突率**：发生了多少个需要人工解决的冲突。这个数字过高意味着业务逻辑或冲突策略需要调整。
- **平均恢复时间（MTTR）**：从操作写入本地到收到服务端确认的平均时间。如果这个值波动很大，说明重试队列或指数退避策略需要优化。

记录这些指标不需要复杂的设施。在 IndexedDB 的操作记录中加入一个字段记录尝试次数和每次尝试的状态，定时上报到监控系统即可。

## 三条落地建议

1. **先做一个“可离线但同步时有冲突提示”的版本，再追求“无缝同步”。** 无缝同步是终极目标，但在团队经验不足时，主动暴露冲突反而能帮你更快理解业务场景。

2. **为每一条本地记录准备一个“操作快照”字段。** 同步失败需要回滚时，快照是唯一的救命稻草。不要心存侥幸，认为失败不会发生在你身上。

3. **把同步队列的可视化调试工具放进开发环境。** 一个简单的面板，显示队列中每一条记录的当前状态、重试次数、最近一次错误。它能在你排查问题时节省至少 50% 的时间。
