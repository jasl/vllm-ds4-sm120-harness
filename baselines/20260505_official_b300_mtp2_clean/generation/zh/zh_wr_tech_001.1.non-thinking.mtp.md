# 离线优先的轻量级任务管理应用：从可用到可信的设计之道

## 一、离线优先的真正代价：信任而非功能

当我们谈论“离线优先”时，很容易陷入一个误区：让应用在断网时也能打开和使用就够了。但真正需要思考的是——用户凭什么相信，他在离线状态下做出的修改，最终能和云端的数据“握手言和”？

对一个任务管理应用而言，用户的核心诉求不是技术上的“能离线”，而是**心理上的“无后顾之忧”**。他需要确信：离线编辑的待办事项不会消失、不会被奇怪地覆盖、不会在同步后出现莫名其妙的重复。这种信任一旦被破坏，即便应用在本地缓存了所有数据，用户也会因不确定感而拒绝使用。

这意味着，离线优先设计本质上是一种**数据一致性契约**。用户离线时所见的“当前状态”，是一种基于本地缓存的时间线快照；而在线同步则是两条时间线的合并过程。如果我们不能优雅地处理这个合并过程，离线功能就只是一次性体验——用一次、出问题、弃用。

## 二、三驾马车：IndexedDB、Service Worker、后台同步

在浏览器环境中，这三个组件构成了离线优先应用的骨骼和肌肉，但我们往往混淆了它们的分工边界。

**IndexedDB** 承担的是**持久化本地状态**的职责。它是数据库，不是缓存。我们在设计 schema 时，需要把任务项、时间戳、操作版本号、未同步标记存储为结构化数据。需要注意，IndexedDB 在移动端 Safari 中的表现并不理想，存储上限、写入性能都要提前做好压测。

**Service Worker** 是**请求代理和服务缓存**。它的核心价值在于：控制网络请求的流向。对于读取操作，可以从 Service Worker 缓存的响应中直接返回；对于写入操作，Service Worker 可以拦截请求并放入后台同步队列，而不是直接发给服务器。但一个常见误区是试图让 Service Worker 直接操作 IndexedDB 来完成数据同步逻辑——这违反了职责分离原则。Service Worker 应该只关心“请求要不要发出去”，而不关心“发出去之后数据要怎么合并”。

**后台同步（Background Sync API）与重试队列**是**可靠性的最后保障**。当用户关闭页面或切换网络环境时，尚未同步的写入操作会被注册为一个 Sync Event。浏览器会在网络恢复后唤醒 Service Worker 执行重试。重试队列本身需要设计指数退避策略：第一次失败等待 5 秒，第二次 30 秒，第三次 5 分钟，超过一定次数后标记为“待人工处理”。

## 三、两大冲突场景：离线优先设计的试金石

### 场景一：同一任务的多端编辑

用户 A 在手机端（离线状态）将任务“准备PPT”的优先级从低改为高，同时用户 B 在网页端将该任务的标题改为“准备演讲稿”。当两端都上线时，服务端收到两个基于不同基线的修改。此时如果采用“后到者胜”策略，必然有一方的修改被覆盖。我们需要引入**字段级冲突检测**：两个修改的字段不同，可以合并；如果修改了同一字段，则依赖时间戳或用户权重决定。

### 场景二：离线删除与在线更新的交叉

用户离线时删除了任务 X，同时在线端有人编辑了任务 X 的截止日期。这种情况下，删除操作在同步后可能被赋予最高优先级（“已经删了就不该再出现”），但这会导致在线编辑丢失。更合理的做法是：暂时保留被删除任务的“墓碑记录”，让在线编辑在中止后弹出冲突提示，由用户决定是否恢复该任务并合并编辑。

## 四、数据流设计：从本地写入到最终一致

一个稳健的离线优先数据流可以简化为以下阶段：

1. **本地写入**：用户在界面操作，应用立即更新 IndexedDB 中的本地副本，并在每条记录中附带 `syncStatus: 'pending'`、`localTimestamp` 和 `revision` 字段。UI 立即反馈“已保存”。

2. **排队**：同时将本次操作推入后台同步队列（SyncManager 或自定义队列）。队列中记录操作类型、数据负载和重试次数。

3. **同步**：网络可用时，Service Worker 按队列顺序逐条发送到服务端。每条同步请求中附带本地版本号，服务端根据版本号检测冲突。

4. **确认**：服务端返回 200 后，将 IndexedDB 中的对应记录的 `syncStatus` 改为 `synced`，并更新 `serverRevision`。队列中移除该操作。

5. **失败回滚或重试**：如果服务端返回 409 冲突，则拉取服务端最新状态，对比后通过 UI 提示用户“请手动合并”。如果返回 5xx 或网络超时，保留队列状态，等待重试。

注意一个关键细节：同步应该是**幂等的**。服务端需要根据 `clientOperationId` 判断是否已经处理过该操作，避免重复执行。

## 五、可观测性：不只是“在线/离线”的开关

很多团队在开发阶段过度关注功能实现，上线后才发现根本无法判断是同步失败率太高还是用户压根没触发离线功能。我们需要设计以下几个观测指标：

- **同步失败率**：`(失败次数 / 总同步操作数)` 按操作类型（创建、更新、删除）和错误码分类。如果某个 API 端点的失败率超过 1%，说明接口设计或冲突检测逻辑有问题。

- **冲突率**：`(发生冲突的操作数 / 同步操作数)`。这个指标反映了用户行为模式。如果冲突率突然上升，可能是版本号逻辑写错了，或者用户习惯在多个设备上同时修改同一批任务。

- **平均恢复时间（MTTR）**：从用户上一次成功同步到本次成功同步之间的时间差。如果 MTTR 超过 1 分钟，说明后台同步的触发时机或重试策略太保守。可以用 `navigator.onLine` 事件结合 IndexedDB 中的时间戳来近似计算。

- **队列积压深度**：未同步的任务数量。这个指标可以在 Service Worker 中定时上报，如果某个用户的积压深度超过 50 条，可能需要主动推送通知提醒他注意同步状态。

## 三条落地建议

1. **先验证冲突逻辑，再实现离线功能**。在开发初期，用模拟数据跑 100 次并发冲突测试，覆盖“同一字段”、“不同字段”、“删除与编辑”三种基本场景。宁可在这里多花一周，也不要在用户面前暴露数据丢失。

2. **保持本地状态与 UI 状态分离**。IndexedDB 是持久层，Redux/Vuex/React Context 是视图层。UI 展示的数据可以来自内存，但写入时直接持久化到 IndexedDB，避免“写入了视图但没写入数据库”的状态不一致。

3. **做好降级预案**。当后台同步重试超过 10 次仍失败时，不要无限重试。改为在 UI 中标记“待上传”，并提供一个“手动同步”按钮，让用户主动触发。同时记录失败详情到日志系统，避免用户困惑。
