# 离线优先：任务管理应用中的信任重建与数据韧性设计

当用户在地铁隧道里创建一条待办事项，在电梯间勾选完成，又在另一个设备上修改备注——这条任务的生命周期，已经跨越了多个网络盲区。离线优先（Offline-First）不是简单地让应用在断网时还能显示界面，而是一种数据架构哲学：**用户不应当为网络波动而买单**。对于任务管理这类高频、低数据量、对时效性敏感的应用，离线优先不只是“断网也能用”，它直接决定了用户是否愿意把关键信息托付给这个产品。

---

## 离线优先为什么关乎信任？

用户对“保存成功”的信任，本质上是对数据一致性的信任。一个典型的反例：用户在离线状态下编辑了任务详情，切回在线后，应用直接丢弃了离线修改并用服务端数据覆盖——这类“最后写入者胜出”的默认策略，会导致用户丢失工作成果。一旦用户意识到应用可能在“静默地”丢失自己的数据，信任便瓦解了。

离线优先的核心挑战在于：**本地写入是即时的，远程同步是异步的，冲突是必然的**。面对这一现实，我们必须从三个基础设施层来构建韧性。

---

## 三大支柱的职责分工

- **IndexedDB** 是离线写入的最终归宿。它承担**本地数据库**的角色，存储任务的全部字段、版本号、本地修改时间戳以及同步状态（待同步/同步中/已确认/冲突）。IndexedDB 的异步 API 和较大的存储配额，使其比 localStorage 更适合存放结构化任务数据。

- **Service Worker** 是离线请求的守门员。它拦截应用的网络请求（如同步 API 调用），在网络可用时透传，不可用时将请求缓存到后台同步（Background Sync）队列。此外，SW 还可以预缓存应用外壳（App Shell），保证首屏在离线时也能渲染。

- **后台同步 / 重试队列** 是离线写出的“邮递员”。当浏览器检测到网络恢复，SW 会触发 `sync` 事件，遍历队列中的待同步任务，逐一发送到服务端。如果某次同步失败（如服务端返回 409 冲突），队列不应立即丢弃该请求，而应将其转入**重试队列**，并附带递增的退避策略（如 1s、2s、4s…），直到成功或达到最大重试次数。

---

## 冲突场景：当离线与在线交错

### 场景一：同一任务的多端编辑

用户 A 在手机离线时重命名任务“买牛奶”→“买燕麦奶”，同时用户 B 在电脑在线时将同一任务截止时间改为 18:00。两端的修改基于同一个旧版本（version=1）。当 A 的设备上线，发送 `{id: 42, name: "买燕麦奶", version: 1}`，服务端已有 B 的修改 `{version: 2, due: 18:00}`。此时发生版本冲突。

常见的解决策略是**基于最新版本的差异合并**（CRDT 风格或字段级最后写入者胜出，但需要用户确认）。更好的做法是：服务端保存每个字段的原子修改时间轴，比较后返回冲突信息，客户端展示差异让用户选择。

### 场景二：离线删除与在线更新的交叉

用户离线删除了任务，客户端在 IndexedDB 中标记为 `deleted=true, syncStatus=pending`。同时，该任务在服务端被另一个用户标记为“已完成”，更新了 `completed=true`。当离线删除的同步请求到达，服务端发现该任务已被修改，删除请求与更新请求冲突。

此时，单纯接受删除会丢失已完成标记。更合理的做法：服务端先标记该任务为“待归档”，不直接物理删除，而是保留一条记录并返回冲突提示。客户端收到冲突后，可展示“此任务已被其他用户更新，确定要删除吗？”，由用户抉择。

## 简化数据流：从本地写入到确认回滚

1. **本地写入**：用户在离线状态下编辑任务，客户端立即写入 IndexedDB，标记 `syncStatus: 'pending'`，并乐观地更新 UI 为成功状态。
2. **排队**：后台同步 API（或自定义重试队列）将此次写操作包装为一条请求记录（含操作类型、任务 ID、完整 payload、本地版本号）。
3. **同步**：网络恢复后，Service Worker 触发同步；请求以幂等方式发往服务端（使用 If-Match/版本号头）。
4. **确认**：服务端处理成功，返回新的版本号；客户端更新 IndexedDB 中的 `syncStatus: 'confirmed'` 和版本号。
5. **失败回滚或重试**：服务端返回冲突（409），客户端弹窗让用户处理；若返回服务器错误（5xx），则重试队列按退避策略重试；若返回网络超时，则留在队列中等待下一次同步事件。

---

## 可观测性：让同步失败不再隐形

离线优先系统最怕“静默失败”。需要记录以下指标，并通过埋点上报到日志或监控平台：

- **同步失败率** = 未达到 “confirmed” 状态的请求数 / 总待同步请求数。应区分“冲突失败”和“服务端异常失败”。
- **冲突率** = 收到 409 响应的次数 / 同步请求总数。冲突率高说明多端同时编辑频繁，可能需要改进冲突解决策略或增加实时协作能力。
- **平均恢复时间（MTTR）** = 从任务进入“pending”状态到首次成功“confirmed”的平均时间。若 MTTR 过高，可能是重试退避策略不合理或服务端处理慢。

具体做法：在 IndexedDB 中为每个任务记录 `syncAttempts`、`lastSyncError`、`syncTimestamps` 数组。在每次同步事件结束时，将汇总的统计信息通过 `navigator.sendBeacon` 或后台同步上传。

## 三条项目落地建议

1. **从未冲突到冲突，渐进式设计**：初期采用“最后写入者胜出 + 版本号校验”即可，但必须预留冲突提示 UI 的扩展点。待用户量增长后再引入字段级差异对比和用户选择面板。

2. **将重试队列做成可视化面板**：在应用“设置”或隐藏菜单中提供一个“同步状态”页面，展示当前待同步任务数量、最近冲突记录、重试次数。这既能帮助用户理解离线行为，也是调试神器。

3. **对写操作做幂等设计**：服务端接口应支持幂等键（由客户端生成的 UUID），避免因重复发送请求导致数据重复。同时，前端应保证 IndexedDB 写入的事务性，防止部分写入导致脏数据。

离线优先不是炫技，而是对用户时间与注意力的尊重。当一条任务能在任何网络条件下可靠地保存、同步、解决冲突，用户才会真正信任这个应用——而信任，是任务管理工具唯一的护城河。
