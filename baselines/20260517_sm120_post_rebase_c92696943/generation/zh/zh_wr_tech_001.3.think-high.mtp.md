# 离线优先任务管理：从“断网可用”到用户信任工程

## 为什么离线优先不只是“断网也能用”

在传统 Web 应用中，“断网”意味着灾难：用户输入丢失、页面白屏、操作无响应。离线优先的初衷是让应用在无网络时依然可用，但真正挑战并不在于缓存几页 HTML，而在于**用户信任**——用户必须相信，哪怕离线操作了 30 分钟，重启应用后数据不会丢失，冲突不会悄无声息地被覆盖，离线时做的删除也不会被线上错误的旧数据“复活”。

更关键的是**数据一致性**。任务管理应用看似简单，实则涉及多端（手机、电脑、平板）同时编辑、移动、删除、更改状态等操作。如果只做“乐观同步”而缺乏冲突处理机制，用户会看到任务列表出现鬼影、重复、状态回滚，最终失去对产品的信任。离线优先并非技术炫技，而是一场**用户信任工程**。

---

## 三大基础组件的职责分工

一个轻量级的离线优先架构，通常依赖三个核心组件。它们各司其职，但需要协同设计：

### IndexedDB：本地可信存储

IndexedDB 是所有离线操作的“地面数据库”。它承担两个职责：**持久化**和**版本化**。所有任务在用户点击“保存”或“删除”后，首先写入 IndexedDB，并附带一个本地时间戳和操作状态标记（pending / synced / failed）。此外，每条记录应拥有全局唯一 ID（而非仅靠数据库自增），以便后续同步时多端能正确关联。

需要注意的是，IndexedDB 的读写是异步的，且事务粒度较粗。设计时应避免频繁全量读取，而是利用索引按状态、时间筛选“待同步”的记录。

### Service Worker：网络拦截与离线缓存

Service Worker 主要做两件事：**缓存应用静态资源**，以及**拦截网络请求以实现“离线写缓存，上线再同步”**。但很多错误设计将 Service Worker 当作万能代理，导致复杂的缓存策略。对于任务管理这类动态数据，SW 更适合只缓存 HTML、CSS、JS，而数据层的离线写入完全交给 IndexedDB。SW 还可以监听 `sync` 事件（后台同步），当网络恢复时自动触发同步流程，避免用户手动“点击重试”。

### 后台同步与重试队列：有序的同步引擎

这是离线优先的“大脑”。一个简单的重试队列可以维护在 IndexedDB 中（或使用 BroadcastChannel 与 SW 通信）。每次本地写入完成后，将操作（如 `{type: 'update', taskId, patch}`）追加到队列，并设定重试次数上限。同步引擎按 FIFO 顺序发送请求（但需注意幂等性），成功则从队列移除，失败则根据错误类型决定回滚还是延迟重试（指数退避）。

---

## 两类典型的冲突场景

### 场景一：同一任务在多端被编辑

用户 A 在手机端将任务标题改为“买牛奶”，用户 B 在电脑端将同一任务改为“买酸奶”，两段操作在离线时分别写入本地 IndexedDB。当网络恢复，两个同步请求几乎同时到达后端。

此时需要**基于向量时钟或版本时间戳的冲突检测**。简单做法：后端维护每个任务的 `lastModifiedAt`，若请求中的 `lastModifiedAt` 小于当前记录的时间戳，则判定为冲突。冲突解决策略可以交给用户（展示用户 A 和用户 B 的版本，让用户选择保留哪一个），或采用“最后写入胜出”（LWW）但必须配合提示。对于任务管理，推荐使用“合并+提示”：后台将两个版本合并（如标题取时间较新的，描述取另一个），并标记“有冲突待确认”，在 UI 上显示冲突提示。

### 场景二：离线删除与在线更新交叉

用户离线时删除了任务 X，同时在线的同事又更新了任务 X 的截止日期。当同步开始：删除操作的请求到达后端，将任务 X 标记为 deleted；随后同事的更新请求到达，却发现任务 X 已被删除。更棘手的情况是，删除请求先于更新请求到达，但更新请求是基于旧数据的“乐观更新”。

解决方案是**软删除+双向版本控制**。任务不真正物理删除，而是设置 `isDeleted: true`，并保留 `updatedAt`。同步时，后端需要检查每个操作的 `baseVersion` 是否与当前版本匹配，不匹配则返回冲突。前端收到冲突后，对于“删除 vs 更新”的冲突，可以设计“优先删除者胜出”，但需通知更新方“你编辑的任务已被删除”。对于轻量级应用，强一致性要求不高时，也可采用“删除操作优先级最高”的规则。

---

## 简化的数据流：本地写入 → 同步 → 确认 → 回滚/重试

以用户离线创建一个新任务为例：

1. **本地写入**：用户点击“添加任务”，前端立刻将任务写入 IndexedDB，状态为 `pending`，并赋予本地唯一 ID（UUID）。界面立即刷新，用户感觉“成功了”。
2. **排队**：同时，同步引擎将该写入操作加入重试队列（同样存储在 IndexedDB 中）。队列元素包含操作类型、任务 ID、完整数据载荷、创建时间、重试次数。
3. **同步**：Service Worker 监听网络恢复（或 `sync` 事件），唤醒同步引擎。引擎按顺序逐条发送 HTTP 请求到后端。请求中携带本地 ID 和版本戳。
4. **确认**：后端接收请求，持久化，并返回服务器生成的全局 ID（或确认码）。前端收到响应后，将 IndexedDB 中对应记录的状态更新为 `synced`，并替换 ID（如果前后端 ID 不一致）。
5. **失败回滚或重试**：如果请求因网络波动失败，同步引擎将重试次数 +1，等待指数退避（如 1s、2s、4s…）。如果超过最大重试次数（如 5 次），则标记为 `failed`，并弹窗提示用户“部分操作同步失败，请手动检查”。对于因冲突导致失败（如版本不匹配），引擎“暂停”该任务，将冲突信息写入另一个冲突表，并通知 UI 展示冲突解决界面。用户解决后，重新生成操作入队。

---

## 可观测性：让离线状态透明化

离线优先应用最怕“黑盒”——用户不知道哪些操作未同步，运维不知道同步失败了多少次。因此，必须埋点并暴露以下指标：

- **同步失败率**：统计每个操作（create / update / delete）从入队到最终失败（超过重试次数或冲突）的比例。按操作类型、设备、网络环境分类，可帮助定位是超时问题还是冲突算法缺陷。
- **冲突率**：统计所有同步请求中因版本冲突被退回的比例。冲突率过高说明 LWW 策略或冲突检测逻辑有问题，需要调整合并算法。
- **平均恢复时间（MTTR）**：从网络恢复（或应用启动）到队列清空（所有待同步操作变为 synced 或 failed）的平均时间。这衡量同步引擎的性能和网络稳定性。
- **队列深度与滞留时间**：监控 IndexedDB 中 pending 状态记录的数量和停留时长，超过阈值则告警，可能表示后端处理瓶颈或网络中断。

在 UI 的右上角，可以提供一个“同步状态”图标：绿色表示全部已同步，黄色表示有 pending 操作，红色表示有失败操作。点击可查看详情，包括冲突记录和手动重试按钮。

---

## 三条项目落地建议

1. **从“乐观 UI”开始，逐步加入冲突处理**：第一版只做本地 IndexedDB 写入 + 简单的 LWW 策略，同步失败则丢弃。上线后观察真实冲突率，再决定是否需要接入向量时钟或用户协商冲突界面。不要过早引入复杂算法。

2. **统一数据层的版本号生成规则**：前后端使用相同的“单调递增时间戳 + 设备 UUID”来生成版本号，避免因时钟偏差产生的误判。如果使用自定义 ID，务必保证每个任务的版本号全局唯一且递增。

3. **为每个操作设计幂等性**：由于网络重试可能发送多次，后端必须确保重复执行同一操作不会产生副作用。比如创建任务时使用客户端生成的 UUID 做幂等键，更新操作使用 `if-match` 头或版本字段校验。

离线优先不仅是一种架构，更是一种对用户承诺的兑现。当用户在山洞中也能顺畅管理任务，并相信一切数据终将安全同步，你的应用就真正赢了。
