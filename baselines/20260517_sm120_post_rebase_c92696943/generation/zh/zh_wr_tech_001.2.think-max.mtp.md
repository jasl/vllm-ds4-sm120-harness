# 离线优先的轻量级任务管理：设计可信任的同步边界

如果开发一个任务管理应用，我们自然希望用户在任何网络条件下都能快速创建任务、打勾划掉待办。但“离线优先”真正考验的并非离线时的流畅度，而是联网后能否让用户相信——数据没有丢失、没有乱序、没有“消失的修改”。离线优先本质上是在构建一种**信任契约**：用户愿意在没有网络时操作，换来的是最终一致且可回查的数据。这背后涉及本地存储、网络代理、同步队列与冲突处理的设计，而不仅仅是利用 Service Worker 缓存几个页面。

## 离线优先的核心：不只是断网可用

当用户离线修改了一个任务，他们期望的是：一旦联网，这个修改会准确到达服务器，且不会被其他端的操作静默覆盖。如果同步后数据出现紊乱，用户的信任就会崩塌。因此离线优先必须解决两个矛盾：**本地响应的即时性**与**云端数据的最终一致性**。这要求我们在客户端维护一个可靠的本地数据层，并设计一套健壮的同步机制，让操作要么最终成功，要么明确失败回滚并告知用户。

## 关键技术的职责划分

- **IndexedDB**：承担本地数据库角色。任务列表、待办状态、甚至修改记录都需要持久化存储在 IndexedDB 中。它允许我们在离线时执行读写，查询也就可以完全脱离网络。它是同步机制的数据基础，存储着尚未同步的变更记录（例如操作日志或版本标记）。

- **Service Worker**：在离线优先中主要接管网络请求和资源缓存。但对于任务数据本身，Service Worker 通常不直接管理数据的同步逻辑，而是用于缓存应用的静态资源（HTML、JS、CSS）以及处理推送通知。数据同步的重任通常放在 JavaScript 主线程或 Worker 中，配合 Background Sync API 或自定义重试队列来实现。

- **后台同步与重试队列**：这是数据一致性的核心。每当用户离线执行写操作（增、改、删），我们首先更新 IndexedDB 并乐观地更新 UI，同时将这次操作包装成一条指令存入同步队列。此后，监听网络状态或定时尝试从队列中取出指令发送给服务器。如果成功，则从队列移除；如果失败（网络不可达、超时或服务器回执不一致），则保留并增加重试间隔，直到达到最大重试次数后标记为永久失败并提示用户。Background Sync 可以省去轮询，但自定义队列给予更细粒度的控制，比如支持批量同步和断点续传。

## 两类典型的冲突场景

冲突一旦发生，用户的信任便会受到挑战。以多端操作为例，常见的冲突有两类：

1. **同一任务的两端离线编辑**：用户在手机端离线将任务标题“买牛奶”改为“买酸奶”，同时另一用户在电脑端离线将同一任务标题改为“买豆奶”。恢复网络后，两个本地版本都有本地写入时间戳，但据服务器而言，它们都是基于同一个旧版本的修改。简单的时间戳对比未必能解决问题，因为两个离线修改的发生顺序是无法确定的。此时要么采用“最后写入获胜”（LW W）但会丢失一方修改，要么引入版本向量（Version Vector）或 CRDT 来合并内容，甚至交由用户手动决策。

2. **离线删除与在线更新交叉**：用户在手机离线删除了任务 A，但与此同时，另一设备在线将任务 A 的截止日期更新了。当手机同步删除指令时，服务器上的 A 已被更新。冲突来了：是该先执行删除再“复活”该版本，还是认为删除比更新更“强势”？常见的策略是标记为软删除并保留更新版的记录作为历史，或者给每个操作赋予一个唯一 ID，按语义优先级处理——但无论哪种，都要保证双方操作不丢失。

## 简化的数据流转过程

为了让概念更清晰，我们描述用户的一次离线创建任务到云端同步完成的典型流程：

- **本地写入**：用户创建新任务，立即写入 IndexedDB 并分配一个本地唯一的 ID，UI 立即显示该任务。同时将“创建任务”记录封装为一个同步指令（含操作类型、对象 ID、内容及本地时间戳），存入同步队列。
- **排队与状态监控**：队列记录指令的尝试次数和最后尝试时间。应用监听 Online 事件或定期检查网络可用性。
- **同步**：一旦在线，开始逐条取出队列中的指令发送到服务器 API。服务器依据操作 ID 进行幂等校验。
- **确认**：服务器返回成功，标明新任务在服务端的最终 ID（替换本地 ID，并更新 IndexedDB 中对应的关联），该指令从队列移除。
- **失败回滚或重试**：若返回 409 冲突（例如基于版本号冲突），本地根据预设策略自动解决（如自动合并或标记为需要用户确认），并更新队列状态。若因网络或服务器错误失败，则保留指令，指数退避等待下次重试，超过阈值时标记失败并向用户展示“同步异常”状态。

## 可观测性：让同步变得透明

维护用户信任，光有机制还不够，还需要**可视化的健康指标**。我们可以记录并上报以下关键数据：

- **同步失败率**：队列中最终失败的指令数 ÷ 总指令数。持续上升表明冲突策略或网络质量存在问题。
- **冲突率**：服务器返回冲突的次数 ÷ 同步成功次数。高冲突率提示用户多端操作频繁，需要优化合并算法或引导用户使用同一设备。
- **平均同步恢复时间（MRT）**：从指令入队到被确认的平均耗时。该值反映了用户离线后等待数据到达云端的时间，是直观感受“滞后感”的指标。
- **队列积压情况**：当用户在离线状态下产生大量操作未同步时，在前端提示“待同步数量”，并在后台日志记录。

这些指标可以通过应用内日志定期上报到分析服务，用于迭代优化冲突策略和同步算法。此外，对用户展示简单的同步状态（如“已同步”“1 项更改待同步”）也有助于建立透明的信任感。

## 三条项目落地建议

1. **冲突策略先求简单再求完善**：对于轻量级任务管理，大多数用户不会同时多端离线操作。初期采用“最后写入获胜”配合版本戳，然后在后台记录冲突并允许用户查看历史版本，远比一开始就实现 CRDT 更容易落地且成本可控。

2. **保证操作的幂等性**：无论是创建（用客户端 UUID 作为 ID）还是更新（用版本号或操作序列号），都要让服务器能识别重复执行的操作，避免一次同步成功后因为重试队列未清除而再次执行导致数据错误。

3. **为用户提供同步可见性**：在界面中保留一个小的同步指示器，列出“待同步项”数量。当发生无法自动解决的冲突时，让用户能查看并选择保留哪个版本。这种透明度比后台静默处理更能赢得用户信任。

离线优先不是一种炫技，而是对用户使用场景的深度理解。当离线体验与在线同步同样流畅、可靠时，用户才会真正信赖你的应用。
