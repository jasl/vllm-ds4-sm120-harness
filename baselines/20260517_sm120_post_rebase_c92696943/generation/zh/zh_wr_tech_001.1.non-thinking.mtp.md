# 离线优先的任务管理应用：不只是“断网也能用”

## 一、重新理解“离线优先”

不少团队对“离线优先”的理解停留在“网络断开时应用不崩溃”的层面，认为给关键数据加个本地缓存就够了。但真正服务过用户后你会发现，离线能力本质上是**用户信任**的工程化表达。

当用户在网络不稳定环境下创建了一个任务，系统静默失败——数据看似写入成功，实则并未持久化。这种“假成功”对用户信任的摧毁比直接报错更严重。用户会开始怀疑：我的操作到底生效了吗？数据会不会丢失？

离线优先设计的核心矛盾在于：**本地响应的即时性与全局数据的一致性之间的张力**。我们需要让用户感觉一切顺畅，同时确保多端数据最终能对齐。这不是缓存策略问题，而是一个分布式系统中的数据拓扑问题。

## 二、三大基础组件的职责边界

### IndexedDB：本地状态的真实来源

IndexedDB 不是“离线时的缓存”，而应当被视为**客户端的主数据库**。用户在离线状态下创建、编辑、删除的所有任务，都应以 IndexedDB 中的记录为准。这意味着我们需要在 IndexedDB 中维护与服务器端一致的数据模型，包括版本号、更新时间戳、删除标记（软删除而非物理删除）。

关键设计点：每条本地记录都应携带一个 `localId` 和 `syncStatus` 字段，用于标记“新建待同步”“已同步”“待更新”“冲突待解决”等状态。这是后续冲突处理的基础。

### Service Worker：网络代理与请求仲裁

Service Worker 是离线能力的“守门员”。它的职责不是缓存页面，而是**代理所有写请求**。当用户提交一个任务变更时，浏览器并不会直接发送 HTTP 请求，而是触发一个 Service Worker 中的 `fetch` 事件。

设计上，Service Worker 维护一个**请求队列**。网络正常时，写请求被转发到服务端；网络断开时，请求被序列化存入 IndexedDB 的 `pending_requests` 表中。这些请求包含完整的操作类型（create/update/delete）、操作数据、时间戳和请求唯一 ID（idempotency key），以防重复提交。

### 后台同步与重试队列：恢复连接的桥梁

后台同步（Background Sync）是 Service Worker 的一个能力，但它解决的是“时机”问题——告诉浏览器有挂起的请求需要处理，请在网络恢复时唤醒 Worker。

更重要的基础设施是**重试队列**。队列需要支持：
- 指数退避重试：首次失败后 30 秒重试，然后 2 分钟、5 分钟……直到上限。
- 最大重试次数与死信处理：超过 N 次失败后，标记为“同步失败”，等待人工介入或降级策略。
- 顺序保证：同一任务的多个变更不能乱序提交。

## 三、冲突场景：理论很清晰，现实很混乱

### 场景一：双端同时编辑

用户 A 在线下修改了任务“报告撰写”的标题为“Q3 报告”，用户 B 在线下同一时刻将其标题改为“述职报告”。两边都认为自己是正确的。

服务器收到两笔修改时，无法简单判断谁更新。解决方案：**最后写入者胜出（LWW）** 是最简单的策略，但丢失了一方编辑。更好的做法是**基于操作化转换（OT）或 CRDT**：把每次编辑建模为“替换第 X 到 Y 位的字符为 Z”，合并后生成双方都接受的结果。对任务管理应用而言，我们可以在 UI 层展示冲突，让用户手动解决：显示“标题有两个版本，请选择一个”。

### 场景二：离线删除 vs 在线更新

用户 A 离线时删除了一个名为“待办清单”的任务。用户 B 在线时更新了这个任务的截止日期。当 A 重新联网，删除操作与更新操作同时到达。

如果简单按时间顺序覆盖，可能造成：A 的删除被 B 的更新吞没（撤销了删除），或者 B 的更新被 A 的删除清掉（丢失了有效数据）。**合理的策略是：以删除为最高优先级操作**。当服务器收到删除操作时，将该任务标记为 deleted，后续任何对该任务的更新都产生“操作目标不存在”的错误，返回给 B 一条提示：“该任务已被删除”。

## 四、数据流：从本地到云端

一个完整的离线写入流程如下：

1. **本地写入**：用户在界面创建任务，数据写入 IndexedDB，状态设为 `pending_create`。
2. **排队**：同时，一个写请求（包含操作类型、数据、时间戳）被送入 Service Worker 的请求队列。
3. **同步尝试**：网络可用时，Service Worker 按队列顺序逐个发送请求。每个请求携带 idempotency key。
4. **确认与状态更新**：服务器返回成功，Service Worker 更新 IndexedDB 中对应记录的 syncStatus 为 `synced`，并清除队列中的请求。
5. **失败与回滚**：服务器返回冲突或业务错误，Service Worker 将记录状态标记为 `conflict` 或 `failed`，同时触发 UI 提示。如果是临时网络错误，进入重试队列，指数退避等待。
6. **同步完成**：后台同步会定期检查是否有未同步的任务；如果有，唤醒 Service Worker 重新执行第 3 步。

## 五、可观测性：看得见的同步健康度

离线优先系统的最大隐患在于“静默失败”——用户以为数据已同步，但服务器从未收到。我们需要在运维层面建立可观测性指标：

- **同步失败率** = 标记为 `failed` 的请求数 / 总请求数。持续上升说明网络质量或服务端接口有问题。
- **冲突率** = 发生冲突的操作数 / 已同步操作数。高冲突率提示前端版本管理策略需要调整，或者多端协作模式需要改进。
- **平均恢复时间（MTTR）** = 从标记为 failed/pending 到成功同步的平均时间。这个指标直接反映重试队列的健康度和网络恢复效率。
- **死信队列大小**：超过最大重试次数的请求数量。需要人工介入时，这是优先级最高的指标。

这些指标可以通过 Service Worker 在 IndexedDB 中记录日志，定期上报到服务器端的监控系统（如 Grafana、Prometheus）。也可以在 UI 上做好“同步状态指示灯”，让用户自己感知数据是否安全。

## 六、落地建议

1. **从“同步完整性”开始，而不是“同步速度”**：宁可让用户等 10 秒确认数据同步完成，也不要让用户看到绿勾却发现任务消失了。选择数据正确性优于即时性。

2. **为每个操作设计幂等性**：无论是 create、update 还是 delete，都要保证相同的请求发送两次不会产生副作用。这是重试机制的前提。

3. **把冲突交给用户决定**：自动合并算法永远存在边界情况。对任务管理这类低频率更改的应用，让用户在界面上选择“保留哪个版本”是最直接的方案，也最容易理解。

离线优先不是技术炫技，而是一种对用户时间的尊重——让用户在任何网络条件下都能完成工作，无需关心底层的数据流。这才是真正的好体验。
