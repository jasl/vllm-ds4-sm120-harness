# 离线优先：构建值得信赖的轻量级任务管理应用

「离线优先」早已不是一个功能特性，而是一种设计哲学。对于任务管理这类高频、低延迟的轻量应用，用户通常希望在任何网络条件下都能快速创建、编辑或完成任务——哪怕是在地铁隧道、飞机上或信号不佳的郊区。如果操作后数据丢失，用户会迅速失去信任。离线优先的核心，不仅在于“断网也能用”，更在于让用户相信：无论网络如何，数据最终会安全到达服务器，并在多个设备间保持一致。

## 离线优先的信任基石

用户对一款工具的依赖，建立在“我说出去的话不会被吞掉”的安全感上。离线优先要求我们在本地持久化写入后立即响应，同时背后默默处理同步。一旦同步失败或产生冲突，用户需要得到清晰的反馈，而不是静默丢失。这意味着数据一致性不再是后端独自承担的责任——前端必须参与到冲突检测、版本管理甚至回滚逻辑中。对工程师而言，离线优先迫使我们将客户端从“展示层”升级为“有状态的参与者”。

## 三驾马车：IndexedDB、Service Worker 与后台同步队列

- **IndexedDB** 是本地数据的主仓库。所有用户操作都先写入这个浏览器内嵌的 NoSQL 数据库，保证刷新或重启后数据不丢。它承担了“离线持久化”和“同步状态标记”的角色——每条记录除了业务字段，还需要携带版本号、时间戳和同步状态（如待同步、同步中、已确认）。

- **Service Worker** 是网络代理和离线资源的管家。除了缓存应用外壳与静态资源，它还能拦截请求，并在网络恢复后触发后台同步事件。更重要的是，Service Worker 提供了“离线环境下请求排队”的能力——我们将需要同步的操作记录为请求，由它按序发送。

- **后台同步/重试队列** 常由 Service Worker 的 `Sync Manager` 或独立的前端队列库实现。它负责管理那些尚未送达服务器的操作：按发生顺序排队，网络可用时逐一发送；遇到服务端错误或网络抖动时，按指数退避策略重试；连续失败阈值后标记为“待人工处理”。队列的设计直接决定了数据吞吐和最终一致性。

## 两类典型冲突场景

- **多端编辑同一任务**：用户在手机端离线将任务标题改为“买菜”，平板端离线又改成了“去市场”。两端均认为自己持有最新版本。当网络恢复并同步时，服务器无法判断谁的变更应被采纳。若采用“最后写入者胜利”，可能导致一方覆盖另一方意图，引起用户困惑。

- **离线删除与在线更新交叉**：用户 A 在离线状态下删除了任务 T；与此同时，用户 B（或同一用户其他设备）在线更新了 T 的截止日期。A 上线后，删除操作与服务器上更新的记录发生冲突：服务端应当保留一个已经被删除的记录，还是接受删除而丢弃更新？如果直接应用删除，B 的更新就无声丢失了。

解决这类冲突通常需要版本向量（如 Lamport 时间戳）或字段级别的合并策略。轻量级应用中，可以优先使用“基于时间戳的 LWW”，但在 UI 上明确展示“此任务可能已被其他操作覆盖”，并提供撤销入口。

## 简化数据流：本地写入 → 排队 → 同步 → 确认 → 重试/回滚

1. **本地写入**：用户操作直接写入 IndexedDB，操作记录（含唯一操作 ID、类型、数据、本地时间戳）同时入队。前端立即响应，用户感知流畅。
2. **排队**：队列存储在 IndexedDB 或 Service Worker 缓存中，标记每个操作为“待同步”。队列保证顺序，避免乱序覆盖。
3. **同步**：Service Worker 监听到网络恢复，遍历队列，依次向 API 发送请求。每个请求携带操作 ID 和冲突检测所需的版本信息。
4. **确认**：服务器处理成功后返回确认（含最新版本 ID）。前端更新本地数据状态为“已同步”。
5. **失败与重试**：若网络错误或服务端返回冲突，队列将该操作标记为“失败”并安排重试。超过重试阈值后，操作进入“冲突”状态，由前端提示用户选择保留本地、保留服务端或手动合并。

这一流程强调了“本地写优先”，后端是最终权威，但前端保留了所有操作的原始记录，便于回滚与人工介入。

## 可观测性：让同步健康度可见

离线优先的调试与运维比纯在线应用复杂得多。建议在 IndexedDB 中维护一块“同步日志”表，记录每次同步操作的时间、操作 ID、状态、失败原因、冲突类型。通过简单的聚合查询即可得到关键指标：

- **同步失败率**：失败次数 / 总同步次数，反映网络或服务端稳定性。
- **冲突率**：冲突次数 / 同步次数，如果突然升高，可能需要审视冲突解决策略是否合理。
- **平均恢复时间（MTTR）**：从操作入队到被成功确认的平均时间。过长的 MTTR 意味着队列积压或重试策略过于保守。

这些指标可以通过埋点上报到监控系统（如 Sentry、自定义日志），帮助团队及时发现版本管理漏洞或后端接口瓶颈。

## 落地三建议

1. **先用最后写入者胜利作为默认策略，但预留手动合并入口。** 轻量级任务管理初期冲突概率低，简单策略可降低工程复杂度；同时通过 UI 提示“此内容可能已过期”让用户自行纠偏，为后续升级为字段级合并留下扩展点。

2. **在 CI 中引入离线模拟测试。** 使用工具（如 Playwright 的 `emulateNetworkConditions`）模拟离线、断线重连、并发冲突等场景，编写自动化用例验证同步队列的可靠性。离线优先的 bug 往往只在真实网络劣化时暴露，提前模拟能大幅降低线上风险。

3. **为同步队列设计可观测的仪表板。** 在开发或管理界面中，展示当前队列长度、待重试次数、最近错误消息。这不仅有助于开发者调试，也能在产品层面给用户一个“数据安全”的可见信号——例如一个微妙的同步图标，从旋转变为绿色对勾，比任何文档都更能建立信任。

离线优先的轻量级任务管理，不是将后端的重量搬到前端，而是以“本地可信、同步可靠”的理念重新分配责任。当用户在地铁里快速勾掉一项待办时，他真正在意的不是“能否上网”，而是“我的进度不会丢失”。作为开发者，我们的目标就是守护这份安全感。
