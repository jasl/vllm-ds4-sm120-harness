# 离线优先的轻量级任务管理：架构决策与数据一致性之道

当“断网还能用”不再只是锦上添花，而成为用户对产品的基本信任时，离线优先（Offline-First）便从一种技术选型上升为一种产品哲学。尤其对于任务管理这类高频、低延迟敏感的应用，用户期望在任何时刻都能添加、修改或完成一个任务，并确信这不会导致数据冲突或丢失。本文面向已有一定经验的前端工程师，探讨如何设计一款真正可用的离线优先任务管理 Web 应用，重点聚焦架构模块、冲突处理、数据流与可观测性。

## 离线优先的本质：信任与一致性

离线优先的核心不是“断网时能用”，而是“断网时用户的操作是安全的，联网后数据是正确的”。如果用户在离线状态下编辑了任务标题，联网后被服务端的旧版本静默覆盖，用户会立刻丧失信任。类似地，如果用户删除了一个任务，而另一台设备同时修改了它，用户期望得到明确的冲突提示或自动合并，而非数据无端消失。

因此，离线优先设计的根本矛盾在于：**本地即时响应 vs 远程最终一致**。我们需要一套机制，既让用户感觉一切都在掌控之中，又保证多端数据最终趋向合理的一致。这要求我们在存储、网络、同步与冲突处理四个维度做出清晰的分层。

## 三大技术支柱的职责划分

- **IndexedDB**：本地数据的主存储器。它负责保存任务的全部字段（包括客户端生成的 UUID、版本号、时间戳），以及本地未同步的操作队列。它的关键职责是提供事务性写入和索引查询，保证离线时 CRUD 操作的原子性与可遍历性。
- **Service Worker**：充当网络代理层。它拦截所有 /api/tasks 的请求，在在线时透传并缓存响应；在离线时从缓存返回静态页面，同时允许 IndexedDB 正常读写。更重要的是，它为主线程的同步逻辑提供了一个“后台”环境——即使页面关闭，Service Worker 仍可监听 sync 事件触发重试。
- **后台同步/重试队列**：这通常由 Service Worker 的 SyncManager 或自实现的队列实现。职责是：从 IndexedDB 中读取待同步操作（包括本地修改的时间戳、操作类型、完整 payload），按顺序发送到服务端，根据响应更新操作状态（成功/失败/冲突）。失败时判断重试次数与指数退避策略，超过阈值则标记为“冲突待决”并通知用户。

三者的协作关系可简化为：用户在 UI 上的每一次修改，先写入 IndexedDB，再将操作加入同步队列（由 Service Worker 管理），后者在网络恢复时自动触发同步。

## 两类典型的冲突场景

1. **同一任务在多端被编辑**：用户 A 在离线状态下将任务标题改为“买牛奶”，用户 B 同时在线将其改为“买面包”。当 A 联网同步时，服务端收到一个基于旧版本（version=1）的更新，而当前版本已是 version=2。此时需要抛出冲突。解决方案可以采用“最后写入者胜”（LWW）配合版本号检测，或采用“客户端合并提示”——对于任务管理，最实用的做法是保留两个版本，让用户选择合并或覆盖。
2. **离线删除与在线更新交叉**：用户 A 离线删除任务 T，用户 B 在线修改了 T 的描述。A 联网同步时，服务端收到删除请求，但 B 的更新还未被 A 知晓。若直接接受删除，B 的修改将丢失。合理的做法是：服务端在收到删除请求时检查该任务最近是否有更新，若有则返回冲突标识，A 的客户端应显示“该任务已被其他设备修改，是否强制删除？”并使用户做出决策。

## 简化的数据流：本地写入→排队→同步→确认→回滚/重试

1. **本地写入**：用户在页面上点击“完成”按钮。前端立即将任务状态更新到 IndexedDB（使用乐观 UI 让界面秒变），同时生成一条操作日志，记录 `{ op: 'update', taskId: '...', field: 'status', newValue: 'done', timestamp: T, version: V }`。
2. **排队**：操作日志被推入一个有序的同步队列（同样存在 IndexedDB 中），并注册一个 sync 事件（若支持）或以轮询方式等待网络。
3. **同步**：Service Worker 或主线程检测到网络可用，从队列中取出最旧的操作，发送 POST/PUT 到服务端，附带 `lastKnownVersion` 和 `clientTimestamp`。
4. **确认**：服务端处理成功，返回新的版本号和服务器时间。客户端更新本地任务的 version 字段，并从队列中移除该操作。
5. **失败回滚或重试**：若服务端返回版本冲突（409），则暂停当前队列，进入冲突处理流程（如弹出对话框让用户选择）。若是网络错误（502），则根据错误类型判断是否重试：临时错误按指数退避（1s、2s、4s…）重试；永久错误（如 404）则直接标记为失败，并通知用户手动处理。

## 可观测性：同步失败、冲突率与平均恢复时间

离线优先的系统最怕“黑盒”——你无法确认用户是否在离线状态下丢失了数据。因此必须建立可观测性机制：

- **同步失败记录**：在 IndexedDB 中为每条操作日志增加 `state` 字段（pending/synced/failed/conflicted）。定期将失败记录上报到监控平台（如通过 beacon API 在页面 visibilitychange 时发送），并关联用户 ID、设备 ID 和操作时间。
- **冲突率**：统计所有同步操作中返回 409 的比例。如果冲突率超过 5%，应反思同步策略（例如版本号的粒度太粗，或 LWW 策略不适用）。可以按字段类型进一步细分：任务标题的冲突率通常远高于“完成状态”。
- **平均恢复时间（MTTR）**：从一条操作被标记为 pending 到其最终变为 synced 或 conflicted 的时间。平均值过高说明同步队列堆积严重，可能需要优化 Service Worker 的 sync 事件触发策略，或引入 WebSocket 加速实时同步。

注意，可观测性不能仅依赖前端——服务端同样需要记录每个请求的 `clientTimestamp` 和 `version`，以便在冲突分析时还原时间轴。

## 三条项目落地建议

1. **从“乐观锁”起步，而非 CRDT**：对于轻量级任务管理，引入向量时钟或 CRDT（无冲突可复制数据类型）的复杂性往往超过收益。使用基于版本号的乐观锁（每次更新携带 `lastKnownVersion`），结合客户端冲突提示，足以覆盖 90% 的冲突场景，且实现成本低、用户理解容易。
2. **优先确保离线写入的持久性，再优化同步速度**：不要因为离线而丢失用户输入。IndexedDB 写入是异步的，务必在 UI 反馈前等待事务成功回调（或使用 `transaction.oncomplete`）。同时为操作队列设置上限（例如 1000 条），超过时提示用户清理或强制同步。
3. **为用户提供“同步状态”的可视化反馈**：在 UI 角落显示一个小图标（已同步/待同步/同步失败），并允许用户手动触发同步。这不仅是技术需要，更是建立信任的关键——用户看到“待同步”便不会怀疑数据丢失。配合冲突发生时的操作日志查看功能，可极大降低客服压力。

离线优先不是单纯的缓存改造，而是一次从“用户永远在线”到“网络只是偶尔可用”的范式转换。当我们把数据一致性的决策权明确交给用户，并在后端支持版本检测与冲突告知时，用户才会真正信赖你的应用——即使它此刻没有网络。
