# 离线优先任务管理应用：设计信任与一致性的技术基线

## 1. 离线优先：从“能用”到“可信”

离线优先（Offline-First）常被误解为“断网也能展示数据”。但真正让用户产生信任感的，不是离线时的数据可见性，而是离线操作与在线操作之间最终一致、无丢失、无冲突的体验。一个任务管理应用，如果用户离线标记了任务完成，上线后发现该任务又恢复为“未完成”，信任就会瞬间崩塌。

因此，离线优先设计的核心目标不是“离线可用”，而是“离线可写” + “在线可恢复一致”。这种信任需要建立在三层基础设施之上：本地存储、网络代理、冲突与重试管理。

## 2. 三引擎的分工：IndexedDB、Service Worker、后台同步

### IndexedDB：本地数据的权威源

IndexedDB 承担离线时的“数据库”角色。它不是缓存，而是应用在离线状态下唯一可信的数据源。设计时要注意：

- 每次本地写操作（创建、编辑、删除）都应直接写入 IndexedDB，并立即更新 UI。
- 所有操作应携带本地时间戳与唯一操作 ID，用于后续冲突检测。
- 不应将 IndexedDB 当作临时缓存，而是视为“离线状态下的主存储”。

### Service Worker：网络状态的门控与请求拦截

Service Worker 负责拦截所有网络请求。在离线优先架构中，它的核心职责不是缓存页面，而是：

- 代理所有与后端 API 的通信，判断在线/离线状态。
- 在线时将本地操作队列中的数据发送至后端，离线时拦截请求并让前端直接读取本地数据。
- 管理“后台同步”事件：当浏览器检测到网络恢复时，触发同步过程。

### 后台同步与重试队列：有序的数据管道

后台同步（Background Sync）是浏览器提供的能力，但开发者仍需实现一个重试队列来管理同步的顺序与失败处理。这个队列在 IndexedDB 中以表形式存储，每条记录包含：

- 操作类型（create / update / delete）
- 本地操作 ID
- 目标任务 ID（如存在）
- 请求体数据
- 创建时间
- 重试次数
- 状态（pending / syncing / failed / conflict）

## 3. 冲突场景：当离线操作与在线操作交错

### 场景一：同一任务在线与离线同时编辑

用户 A 在离线时修改任务标题为“写报告”，同时在线的用户 B 将其标题改为“准备演示”。用户 A 上线后，Service Worker 会尝试同步其修改。后端收到请求时，需要检测本地版本的时间戳或版本号是否落后于当前在线版本。

处理方式：后端返回 409 Conflict，并附带当前在线版本数据。前端收到冲突响应后，应将用户 A 的本地数据标记为“冲突”，在 UI 中展示两个版本，让用户手动选择或合并。

### 场景二：离线删除与在线更新交叉

用户离线删除了任务 X，但与此同时，在线端对任务 X 进行了字段更新。用户上线后，删除请求到达后端，但任务 X 已被在线更新。此时如果后端直接执行删除，则覆盖了在线更新。

设计原则：删除操作在冲突中应具有更高优先级？不一定。需根据业务判断。更安全的做法是：后端收到删除请求时，若发现任务的最后修改时间晚于删除操作的本地时间，则标记为冲突，由用户确认是否仍然删除。

## 4. 数据流：从本地写入到最终一致

一个典型的数据流如下：

1. **本地写入**：用户操作触发函数，首先写入 IndexedDB（本地数据立即更新，UI 响应），同时将操作元数据写入重试队列（状态为 pending）。

2. **排队**：Service Worker 检测到网络状态变化（从离线到在线），触发后台同步事件。前端从重试队列中取出 pending 状态的操作，按时间顺序逐个发送（某些场景需保持顺序性）。

3. **同步**：请求发送至后端 API，后端验证版本号或时间戳。若无冲突，确认成功，前端将队列中对应记录状态改为 “synced”，并更新本地时间戳。

4. **确认**：后端返回最新数据，前端用后端数据覆盖本地 IndexedDB 中的对应记录，保持与在线端一致。

5. **失败回滚或重试**：
   - 若网络超时或服务器 5xx：重试次数 +1，若未超过上限，状态仍为 pending，等待下次后台同步事件。
   - 若返回 409 Conflict：状态改为 “conflict”，记录冲突数据，停止自动重试，等待用户介入。
   - 若返回 404（数据已被其他端删除）：本地将任务状态改为 “deleted_remotely”，UI 隐藏或提示。

## 5. 可观测性：让离线架构“看得见”

离线优先架构引入了大量异步、重试、冲突等复杂流程。如果没有可观测性，开发者只能等待用户报 Bug。

需要记录的关键指标：

- **同步失败率**：统计重试队列中最终状态为 failed 的记录数 / 总操作数。若超过阈值（例如 1%），说明网络或服务端存在问题。
- **冲突率**：统计返回 409 的请求数量 / 总同步请求数。高冲突率提示设计层面需优化版本控制策略。
- **平均恢复时间（MTTR）**：从操作进入队列到最终状态变更（synced / conflict）的平均时间。若过长，需排查后台同步事件的触发频率或重试间隔。
- **重试分布**：记录每个操作的重试次数分布，若大量操作需重试 3 次以上，可能网络波动或服务端超时设置不合理。

这些数据可以通过在 Service Worker 中埋点，或从 IndexedDB 的重试队列表中定时采集，汇总后发送至分析服务。

## 6. 项目落地建议

- **先从单用户场景开始，再扩展到多端协作**：单用户离线优先的复杂度远低于多端冲突处理。先验证本地写入 + 重试队列的可靠性，再引入版本戳和冲突 UI。
- **为冲突准备清晰的用户交互层**：不要期望开发者手动合并数据。冲突 UI 应展示“本地版本 vs 在线版本”的对比，并提供“保留本地”、“采用在线”、“手动编辑”三个选项。
- **在开发阶段就引入可观测性工具**：离线架构的 Bug 往往在真实网络环境下才暴露。上线前先用模拟器测试网络断开/恢复循环，并在开发阶段就搭建同步失败和冲突率的看板。

离线优先不是一项“附加功能”，而是一种重新思考数据流与用户信任的架构选择。每一条离线状态下创建的任务，都应当像在有线世界中一样，被认真对待。
