# 离线优先的轻量级任务管理：从“能用”到“可信”的设计之道

当我们谈论“离线优先”时，许多工程师的第一反应是“断网也能编辑”。但这只是冰山一角。真正的离线优先设计，解决的是用户对数据的信任问题——当你在高铁隧道里修改了一个任务，五分钟后网络恢复，你的改动还在吗？会不会和服务器的数据“打架”？会不会无声无息地消失？

一个轻量级任务管理应用，恰恰是离线优先理念的最佳实践场：任务数据不复杂、用户对实时性要求适中、但“别丢我数据”的心理预期极高。本文从架构视角，拆解如何让应用在离线与在线之间平滑切换，并让用户愿意信任你的系统。

## 三大基础设施：各司其职的分层设计

离线优先的底层依赖三个核心组件，它们各守一道防线。

**IndexedDB** 是本地数据的“持久层”。它存储任务的全文、状态、时间戳。为什么不用 localStorage？因为任务数据可能包含文件附件、多个标签、复杂索引查询。IndexedDB 能提供结构化存储和事务支持，保证写入时不会出现半更新状态。这里有一个容易被忽略的细节：本地数据模型应与服务端保持松耦合，但必须有统一的主键和版本号字段，这是后续冲突检测的命脉。

**Service Worker** 是网络的“代理层”。它拦截所有 API 请求，在在线时透传，离线时返回缓存的“空壳页面”或提示信息。但它的真正价值不在于缓存静态资源，而在于作为“同步管道的守护者”——当 Service Worker 检测到网络恢复，它能主动唤醒应用的后台同步流程。注意，Service Worker 的生命周期与页面解耦，这意味着即使标签页已关闭，它仍可以触发同步，这是“可靠离线”的基石。

**后台同步/重试队列** 是数据一致性的“调度器”。它不是一份简单的请求队列，而是一个包含请求体、重试次数、最后尝试时间戳、冲突处理策略的“任务单元”。队列需要支持持久化（存入 IndexedDB），防止应用被杀后丢失。每次同步失败，队列不立即重试，而是采用指数退避：第一次 1 秒后重试，第二次 10 秒，第三次 1 分钟……避免在网络不稳定时对服务器造成无意义压力。

## 冲突的两面：怎么打，怎么解

冲突是离线优先中最棘手的部分。两类典型场景值得深究。

**场景一：同一任务的双向编辑**
用户 A 在飞机上修改任务“二期需求评审”的标题为“二期需求定稿”，同时用户 B 在公司给同一条目增加了截止日期。网络恢复后，两条更新在服务端“撞上”了。此时“先到先得”或“最后写入覆盖”都会造成数据丢失。合理的策略是字段级冲突检测：服务端识别到同一任务的两个版本来自不同用户，且没有交集字段，则自动合并；如果修改了同一字段（比如两人都改了标题），则保留最新版并通知冲突方，用 UI 标记“需人工确认”。

**场景二：离线删除与在线更新的交叉**
用户 A 在离线状态下删除了任务“设计评审报告”，而同一时间用户 B 在线给该任务标记了“已完成”。当 A 的设备重新上线，删除请求被发送，但服务端需要决策：是接受删除（丢失 B 的更新记录）？还是拒绝删除（返回“任务已被更新”）？建议采用“软删除 + 版本链”机制：服务端不直接删除，而是记录一个“删除标记”，同时保留最新版本。服务端比对版本号后，若发现 A 的删除版本低于当前版本，则返回“冲突：该任务已被更新，删除被驳回”，并要求 A 手动确认——彻底将决策权交还人类。

## 数据流：从本地到服务器的完整循环

以最常见的“创建任务”操作为例，走一遍完整数据流：

1. **本地写入**：用户点击“新增”后，数据立即写入 IndexedDB，UI 同步更新，用户感觉“秒存”。此时记录一个 `syncStatus: "pending"` 的状态字段。
2. **排队**：后台同步队列收到一个“CREATE”请求，包含任务 payload、本地时间戳、生成唯一 UUID。
3. **同步**：Service Worker 监听到网络可用时，按队列顺序逐个发出请求。这里的“顺序”至关重要：如果用户离线时先创建了任务 A，然后又删除了任务 A，队列中应当合并为一条“无操作”或直接丢弃。
4. **确认**：服务端成功处理后返回任务 ID，队列将本地记录中的 `syncStatus` 改为 `"synced"`，并更新服务端 ID。
5. **失败回滚或重试**：若同步失败（服务端返回 500 或网络超时），队列递增重试计数，将记录存入 IndexedDB 持久化。当重试超过上限（例如 7 次），标记为 `"failed"` 并弹出 UI 通知：“任务‘设计评审’同步失败，请稍后手动检查”。此时不自动丢弃数据，给用户最后一道保险。

## 可观测性：让“不可见”变得可审计

离线系统最大的挑战是“黑盒”。用户不知道自己的操作有没有同步成功，开发者不知道冲突率有多高。以下三个指标应当被记录并可视化：

**同步失败率**：按失败原因分类统计（网络错误 vs 冲突 vs 服务端拒绝）。如果某个接口的冲突率突然上升，可能意味着服务端合并策略出了问题。

**冲突率**：统计每分钟内服务端检测到的字段级冲突次数。结合事件时间线，可以评估到底有多少操作在离线期间“撞车”。高冲突率往往意味着 UI 上缺少“冲突预览”或“手动合并引导”。

**平均恢复时间**：从用户设备离线到所有操作成功同步至服务端的时间。这个指标能衡量重试策略是否有效——如果大部分同步在 30 秒内完成，说明指数退避参数恰当；如果平均需要 3 分钟，可能需要更激进的重试。

这些数据可以暴露给内部运营面板，也可以在用户侧提供一个“同步状态”小图标，点击后展开队列详情，让用户对自己处于什么状态有掌控感。信任，往往就建立在这些透明的小细节上。

---

三个落地建议：

1. **把冲突决策权交给用户，而非系统自动处理。** 自动合并在多数场景下高效，但遇到“删除与更新交叉”等敏感操作时，宁可以“请确认”的弹窗打断用户，也不要沉默地丢掉信息。

2. **重试队列必须持久化，且支持优先级排序。** 写入 IndexedDB 是底线，同时队列中“删除”请求的同步优先级应高于“编辑”，避免用户先删除后编辑，结果删除却先同步导致编辑请求失效的尴尬。

3. **在应用启动时做一次“离线健康检查”。** 检查本地 IndexedDB 是否有大量未同步记录、队列是否超过合理长度、是否有超过 24 小时未同步的任务——用这些信号触发预提示，而非让用户在操作失败时才意识到问题。
