# 离线优先的轻量级任务管理应用：从“断网能用”到“用户信任”

当一位工程师说“我的应用支持离线”时，通常只是指它在断网时不会崩溃。但真正的离线优先，是一种更深层的承诺——用户需要相信，即使在网络不稳定的地铁里、在信号微弱的山区，他们对数据的每一次操作都不会丢失，也不会在重新联网后莫名其妙地被覆盖或冲突。

这种信任，才是离线优先设计的核心价值。

## 离线优先的认知升级：不只是一个技术开关

传统在线应用的用户心理模型是“写即达”——我点了保存，数据就到了服务器。离线优先打破了这一模型，我们必须让用户理解并接受“本地先写，异步同步”的新模型。这意味着应用需要提供明确的视觉反馈：这是草稿、正在同步、同步失败、需要手动处理冲突。

数据一致性也不再是服务端的事务保证，而变成了一个分布式系统问题。本地数据库与远程数据库之间，需要一套清晰的协议来确保最终一致性。这不是“断网也能用”五个字能包含的。

## 三大核心技术的职责划分

离线优先的架构，依赖三个关键技术组件，各自承担不同的角色，任何组件的设计失误都会破坏整个信任链条。

**IndexedDB 负责本地持久化。** 它是应用的本地数据源，所有用户操作首先写入这里。关键在于设计合理的存储结构：任务列表、本地修改时间戳、同步状态（pending/synced/conflicted）。本地写入必须保证原子性——要么完整写入，要么不写，避免半保存状态。

**Service Worker 负责网络拦截与缓存。** 它位于浏览器与网络之间，决定哪些请求走网络、哪些走缓存。对于我们的任务管理应用，Service Worker 需要缓存应用的静态资源（HTML、JS、CSS），使得首次加载后即使断网也能打开应用。更重要的是，它需要拦截 API 请求，当检测到离线时，将请求存入一个“待发队列”，而非直接返回失败。

**后台同步/重试队列负责保证最终送达。** 这通常通过 Service Worker 的 SyncManager 或手动实现的重试机制来完成。当网络恢复时，队列中的操作按顺序重放。重试需要有指数退避策略，避免在弱网环境下反复冲击服务器。同时，队列需要持久化（通常存储在 IndexedDB 中），确保浏览器关闭后也不会丢失待发请求。

## 冲突的两种典型场景

冲突不是理论问题，而是日常操作中一定会发生的现实。以下是两个必须处理的具体场景：

**场景一：同一任务在多端被同时编辑**
用户在手机端离线编辑任务标题“买咖啡”，同时在电脑端（在线）将同一个任务标题改为“买牛奶”。当手机重新联网同步时，服务器已经有一个更新的版本。这是典型的“多端并行修改”冲突。解决方式通常有三种：最后写入者获胜（简单但可能丢失数据）、三向合并（复杂但更准确）、强制用户手动选择。对于轻量级任务管理，最后写入者获胜配上版本号比较通常足够，但需要给用户一个“查看历史版本”的入口。

**场景二：离线删除与在线更新交叉**
A 用户在离线状态下删除了任务 X，B 用户（在线）此时将任务 X 的截止日期更新为明天。当 A 重新联网，删除请求与更新请求同时到达。如果删除先到达，更新会失败（任务不存在）；如果更新先到达，删除会生效，但 B 的修改丢失。这需要服务器端设计“操作顺序”和“幂等性”——比如删除操作携带版本号，如果服务器上的版本更新，则拒绝删除并返回冲突。

## 简化的数据流设计

一个干净的离线优先数据流应该像这样：

1. **本地写入**：用户创建/修改任务，数据先写入 IndexedDB，标记为“待同步”（pending）。UI 立即更新，用户看到操作生效。
2. **排队**：若网络在线，立即发起同步请求。若离线，请求进入持久化队列（存储在 IndexedDB 中），并显示“等待同步”状态。
3. **同步**：后台同步触发时，队列中的请求按创建顺序发送到服务端。每次请求包含本地版本号和时间戳。
4. **确认**：服务器处理请求后，返回确认响应（含新的版本号）。客户端将本地记录标记为“已同步”。
5. **失败回滚或重试**：如果服务器返回冲突错误，客户端将对应记录标记为“冲突”，并将服务器的当前数据拉取到本地，展示给用户让用户决定如何处理。如果是网络错误，请求保留在队列中，等待下一次重试。

## 可观测性：让系统健康可衡量

离线优先系统的最大问题是：当出现同步异常时，用户不知道，开发者也不知道。因此，必须建立完整的可观测性体系。

我们需要追踪三类核心指标：

- **同步失败率**：每 100 次同步请求中，有多少次最终失败（非网络错误，而是冲突或服务器错误）。这个指标超过 5% 说明冲突处理策略需要调整。
- **冲突率**：不同设备之间发生数据冲突的比例。高冲突率通常意味着两个问题：一是用户跨设备使用的频率极高，二是我们的冲突解决策略（如最后写入者获胜）过于激进。需要根据冲突率决定是否引入更细粒度的合并逻辑。
- **平均恢复时间**：从用户设备恢复网络到所有待同步任务完成确认的平均时间。这个指标帮助判断重试策略是否合理——如果恢复时间过长，说明重试队列的调度需要优化（如增加并行度，或使用更激进的指数退避）。

此外，还需要记录每次冲突的具体类型、用户主动解决冲突的比例、用户放弃操作（关闭页面并重新打开）的次数。这些数据可以帮助发现用户体验的痛点和系统设计的漏洞。

## 三条落地建议

1. **先从“乐观写入+最后写入者获胜”开始，而不是一开始就实现复杂的三向合并。** 大多数轻量级场景下，用户冲突并不频繁。简单的策略配合版本号回退机制，在实现成本和用户体验之间找到了合适的平衡点。等产品真实数据积累到一定程度，再根据冲突率曲线决定是否升级策略。

2. **把同步状态可视化放在 UI 的第一优先级。** 每个任务卡片上都需要清晰显示“已同步”、“等待同步”、“同步失败”或“存在冲突”。用户需要知道自己操作的结果在哪里、是否确认。模糊状态是离线系统最大的信任杀手。

3. **第一版本不要追求“无感同步”，而是要“可控同步”。** 让用户知道何时离线，看到待同步队列，甚至可以手动触发同步或选择冲突版本。等用户习惯这个模型后，再逐步优化为更智能的后台处理。信任建立在透明度之上。

离线优先不是新技术堆砌，而是对用户场景的深度尊重。好的离线应用，让用户即使在最差的网络条件下，也能保持生产力，并且完全相信系统不会背叛他们的每一次点击。
