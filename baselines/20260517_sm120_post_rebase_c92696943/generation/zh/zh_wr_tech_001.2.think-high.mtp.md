```markdown
## 离线优先的任务管理应用设计：从存储到同步的完整路径

“离线优先”在今天的前端圈子里已经不新鲜了。很多团队会把“断网也能用”当成一个功能开关，加上本地缓存就完事。但真正落地过的人都知道，离线优先的核心挑战不是离线本身，而是“离线-在线”切换时，用户对数据的信任感。一个任务管理应用，用户在上面记录的是工作、日程甚至生活承诺。如果同步过程中数据丢失、冲突无告、或者恢复时间不可预期，用户就会对系统失去信心。因此，我们必须把离线优先设计成一个**可靠性工程**问题，而不是一个缓存策略问题。

### 三大技术支柱各自的分工

要构建一个轻量级的离线优先任务管理器，有三项技术是避不开的：IndexedDB、Service Worker 和后台同步/重试队列。它们各司其职，拼成完整的数据链路。

- **IndexedDB** 是本地持久化的“硬盘”。传统 localStorage 有容量限制和同步阻塞问题，不适合存储频繁变动的任务数据。IndexedDB 提供键值存储、索引查询和事务支持，允许我们在本地维护一个完整的任务集合，甚至可以建立“未同步”标记、时间戳、版本号等元数据字段。它的职责是：**保证用户的操作在离线状态下被可靠保存，并且在应用重启后不丢失**。

- **Service Worker** 是网络拦截的“守门员”。它不在渲染线程中运行，因此可以拦截所有 fetch 请求。对于任务管理应用来说，Service Worker 的主要职责不是缓存静态资源（那是基础能力），而是**接管 API 请求的转发逻辑**。当设备在线时，Service Worker 直接透传请求；当设备离线时，它返回本地 IndexedDB 中的任务数据，或者给出一个“离线中”的占位响应。同时，Service Worker 也负责监听网络状态变化，触发后台同步事件。

- **后台同步/重试队列** 是数据一致性的“调度室”。Service Worker 提供了一个 `sync` 事件，当网络恢复时浏览器会自动触发它。但仅仅依赖原生注册还不够，我们需要自己维护一个**操作队列**：每一次用户创建、更新、删除任务，都先写入 IndexedDB，然后向一个本地队列（同样存储在 IndexedDB 中）推入一条“待同步操作”记录，包括操作类型、任务 ID、操作数据、时间戳等。这个队列是幂等设计的：同一操作被重复执行不会产生副作用。后台同步事件逐一消费队列，遇到网络失败则推迟重试，并记录失败次数与原因。

### 冲突场景：用户信任的试金石

离线优先最容易引发信任危机的地方是**冲突**。任务管理应用的用户很可能在多端（手机、电脑）同时操作，或者一个人离线编辑、另一个人在线处理同一任务。下面两类冲突是必须提前应对的。

**场景一：同一任务在多端被编辑**
假设用户 A 在手机端离线将任务“写周报”的优先级改为“高”，同时用户 B 在电脑端在线将同一任务的截止日期改为“今天”。两边的本地副本都产生了新状态。同步时，如果没有冲突策略，要么后者覆盖前者（导致优先级丢失），要么前者覆盖后者（导致截止日期丢失）。更糟糕的是，如果两个操作都基于同一个旧版本，它们的修改是**语义上不冲突但数据字段冲突**的。这需要我们在本地存储中维护每个字段的原子修改记录，或者在同步时采用“最后写入者获胜（LWW）”加上字段级别的合并。对于轻量级应用，LWW 配合时间戳通常够用，但必须给用户提供“冲突已自动解决”的提示，让用户有机会检查。

**场景二：离线删除与在线更新交叉**
用户在离线状态下删除了任务“需求评审”，同时另一位同事在线编辑了该任务的描述。同步时，系统会拿到两条操作：删除请求和更新请求。如果简单按时间顺序执行，先删后更新会导致“更新一个不存在的任务”的错误；先更新后删则会丢弃删除操作（因为删除晚于更新）。这类冲突本质上是一个 **“删除标记”** 问题。可靠的做法是：本地删除时不真正删除数据，而是标记为“已删除（软删除）”并推送到同步队列；同步服务器收到删除标记后，对该任务加锁，拒绝后续对该任务的更新请求，并返回冲突告警。如果服务器已经接受了更新，则删除操作应当以“最终删除”为准，但需要把被覆盖的更新内容作为附件通知用户。

### 简化的数据流：从本地写入到最终一致

一个可靠的离线优先流程可以抽象为五个阶段：

1. **本地写入**：用户对任务进行任何操作（增、改、删），应用首先将最新状态写入 IndexedDB，同时在 IndexedDB 中追加一条操作记录到“同步队列”表。写入完成后立即更新 UI，让用户感觉“秒级响应”。

2. **排队**：同步队列中的每条记录包含唯一 ID、操作类型、操作对象、操作数据、创建时间戳、重试次数和状态（待发送/发送中/已确认/失败）。队列按创建时间排序，确保 FIFO 顺序。

3. **同步**：Service Worker 监听 `sync` 事件或通过轮询网络状态，从队列中取出“待发送”记录，逐一向服务端 API 发送请求。每次发送前，先从本地读取最新的任务数据（因为用户可能在排队过程中又修改了同一任务），将最新快照作为请求体。

4. **确认**：服务端返回成功（HTTP 200/201）后，客户端将该记录标记为“已确认”，并从 IndexedDB 中删除该操作记录。服务端返回的响应如果包含冲突标识（如版本不匹配），则进入冲突处理分支。

5. **失败回滚或重试**：如果服务端返回网络错误（如 5xx、超时），客户端将该记录的重试次数 +1，并设置下次重试的延迟时间（指数退避）。如果重试次数超过阈值（如 5 次），将该记录标记为“永久失败”，并在 UI 上提示用户“同步失败，请手动检查”。如果服务端明确返回冲突（如 409），则根据预设策略（如 LWW 或手动合并）决定是否覆盖本地数据，或者将冲突信息推给用户界面，让用户手动解决。

### 可观测性：让“看不见的问题”变得透明

离线优先系统天然存在“黑盒”风险——用户离线时一切正常，但那只是因为错误被隐藏了。需要建立三个维度的可观测性：

- **同步失败日志**：每条操作记录从“待发送”到“永久失败”的全生命周期都应写入日志。日志中至少包含操作类型、任务 ID、失败原因（网络超时/服务端异常/冲突类型）、重试次数、最终状态。日志可以存储在 IndexedDB 中的“同步日志”表，或者通过 Service Worker 在后台发送到远程日志服务（如 Sentry 或自定义端点）。关键指标是**同步失败率** = 永久失败的操作数 / 总操作数，这个数值应控制在千分之一以下才值得信任。

- **冲突率**：冲突不是 bug，但高频冲突意味着设计缺陷。我们需要统计每次同步时发生冲突的次数，并按冲突类型（字段级覆盖/删除-更新交叉）分类。如果冲突率超过某个阈值（比如 5%），就应该考虑调整冲突解决策略或引入更细粒度的版本控制。

- **平均恢复时间（MTTR）**：从用户失去网络到所有待同步操作被成功确认的平均时间。这个指标能反映重试策略是否健康。如果 MTTR 超过用户可接受的范围（比如 5 分钟），则应该优化指数退避参数，或允许用户手动触发同步。也可以在 UI 上显示一个“同步状态指示器”，让用户看到还有多少操作在排队、最近一次同步时间、以及失败的记录数量。透明度本身就是建立信任的手段。

### 三条项目落地建议

1. **从关键场景切入，逐步扩展**：不要一开始就给所有页面加上离线能力。选择一个核心场景（比如“创建任务”和“修改标题”，而非“拖拽排序”或“附件上传”），先打磨好本地写入和重试队列，再逐步覆盖删除、批量操作、字段冲突。每增加一个场景，都要设计对应的冲突处理测试用例。

2. **为冲突解决策略设计显式反馈机制**：不要默默执行 LWW 或者服务端优先。无论选择哪种策略，都要在同步完成后，在 UI 上用一个横幅或通知告诉用户：“任务‘需求评审’的优先级由你本地的修改覆盖了在线版本，因为你的操作时间更晚。” 用户至少知道发生了什么，而不是发现数据不对后再去猜测。

3. **把可观测性纳入产品需求，而非事后打补丁**：在开发初期就规划好同步日志、冲突统计、MTTR 的采集。这些指标应该出现在产品的管理后台或监控仪表盘上，而不是只留在工程师的控制台里。当用户反馈“数据丢了”时，你能迅速定位到是哪次同步失败、是哪个操作导致了冲突。这不仅能修复个案，还能推动全局设计改进。
```
