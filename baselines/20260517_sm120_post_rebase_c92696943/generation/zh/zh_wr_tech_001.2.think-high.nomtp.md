# 离线优先：如何让任务管理应用在断网时依然可靠

当用户在地铁里打开任务管理 App 编辑待办事项，突然进入隧道断网——如果他发现编辑的内容消失了，或者下次联网时出现不可预期的数据碰撞，那么对这个应用的信任就会一落千丈。离线优先（Offline-First）不仅仅是“断网也能凑合着用”，它意味着应用在脱离网络时依然能提供完整、一致的操作体验，并且在与服务器重新连接后，能优雅地处理用户预期与现实之间的落差。这种信任的建立，依赖一套轻量但严谨的架构设计。

## 为什么离线优先关乎用户信任与数据一致性

用户对“本地应用”的直觉是：我做的操作应该立即生效，且不会丢失。而传统在线应用将数据主权交给服务器，一旦网络中断，用户的“心理模型”就会崩塌。离线优先的核心理念是：**本地是权威，网络是同步通道**。它承诺用户：即使没有服务器确认，你的操作也会被认真对待，后续的冲突会以可预测、可理解的方式解决。这种承诺直接转化为用户信任。

但信任一旦被破坏（比如同步后产生重复任务或意外覆盖），修复成本极高。因此，数据一致性成为离线优先的关键挑战——不是追求强一致性（这在离线场景下不现实），而是追求**最终一致性**，且冲突处理逻辑必须对用户透明或可控。

## 三大技术组件的职责分工

实现离线优先通常依赖三个互补的浏览器能力：

- **IndexedDB**：作为本地持久化数据库。它负责存储任务数据、版本号、操作日志等。与 localStorage 不同，IndexedDB 支持结构化数据、索引和事务，可以高效管理大量任务。它是离线操作的基础，保证用户关闭页面后数据不丢失。

- **Service Worker**：充当网络代理。它可以在用户离线时拦截请求，从缓存中返回内容，或者将请求暂存起来。更重要的是，它能在后台运行，即使页面关闭，也能在恢复网络后触发同步。Service Worker 的生存周期与页面独立，这为后台同步提供了条件。

- **后台同步/重试队列**：这个组件通常由 Service Worker 中的 `SyncManager` 或手动实现的任务队列组成。它负责管理待发送的操作——当用户触发增删改时，先写入本地（乐观更新），再将操作入队。后台同步会在网络可用时自动执行，失败后按指数退避策略重试，直到成功或达到最大重试次数。

## 至少两类冲突场景

离线优先最棘手的问题是冲突。以下两个典型场景值得细究：

**场景一：同一任务在多端被编辑**
假设用户手机离线，将任务“买牛奶”改成“买豆浆”；同一时刻，电脑在线，将同一任务标题改为“买酸奶”。当两者同步时，服务器收到两个基于不同版本的修改，产生冲突。处理策略可以是“最后写入者胜出”（简单但可能丢失重要修改），或基于时间戳的“自动合并”，或干脆将冲突记录展示给用户，让用户手动选择。

**场景二：离线删除与在线更新交叉**
用户离线时删除了某个任务，但服务器端同一任务被另一位同事（或用户在线设备）更新了内容。同步时，删除操作与更新操作发生冲突：该保留更新还是执行删除？常见做法是引入“墓碑”（tombstone）——删除操作实际上在本地记录一条“删除标记”，同步时服务器检查如果该任务已被标记删除，则忽略所有后续对该任务的更新。但若更新操作先于删除到达服务器，则可能产生一个幽灵任务，需要额外的冲突检测。

## 简化的数据流：一次操作的完整旅程

以用户离线时创建一个新任务为例，数据流如下：

1. **本地写入**：用户点击“添加任务”，任务立即出现在 UI 中（乐观渲染），同时生成一个唯一客户端 ID（UUID）和本地时间戳。IndexedDB 存储任务完整对象，状态标记为“待同步”。
2. **排队**：操作细节（操作类型、任务 ID、变更内容）被推入一个持久化的重试队列（同样存储在 IndexedDB 中）。
3. **同步触发**：当网络恢复，后台同步事件（通过 Service Worker 或 Page Visibility API）将队列中的操作依次发送到服务器 API。
4. **服务器确认**：服务器接收操作，返回确认（例如带有服务端版本号）。客户端收到后，更新本地状态为“已同步”，并从队列中移除该操作。
5. **失败回滚或重试**：若服务器返回错误（如冲突）、或网络中断导致请求超时，操作保留在队列中，等待下一次同步。如果重试超过阈值（如 5 次），可将任务标记为“同步失败”，并展示给用户，让用户手动处理。

## 可观测性：同步失败、冲突率与平均恢复时间

没有监控的离线优先系统就像蒙眼开车。关键指标包括：

- **同步失败次数**：统计队列中操作被标记为“永久失败”的数量。结合用户 ID 可以定位是否某个用户的同步长期异常。
- **冲突率**：服务器返回 409 Conflict 的比例。冲突率过高意味着需要优化冲突解决策略或降低离线操作的并发程度。
- **平均恢复时间（Average Time to Recovery）**：从用户离线到所有待同步操作成功确认的平均耗时。这个指标能反映同步机制的效率及网络质量。实现方式：在操作入队时打上时间戳，同步成功时计算差值。

在工程实现上，可以将这些指标通过定期上报（使用 Beacon API 或 Service Worker 的 fetch 事件）发送到后端日志系统，或者集成到现有的监控平台（如 Sentry、Prometheus）。不需要客户端代码，只要在关键节点埋点即可。

## 三条项目落地建议

1. **从“乐观更新 + 重试”起步，再逐渐引入冲突解决**。切勿一开始就设计复杂的冲突合并算法。先让本地操作立即显示，同步失败时自动重试，多数场景可满足。当用户投诉数据不一致时，再逐步增加冲突检测和手动解决界面。

2. **为每个操作分配全局唯一 ID 和版本号**。每一个客户端创建一个 UUID 作为操作标识，同时本地记录任务版本号。服务器端通过版本号判断是否冲突，操作 ID 用于幂等去重。这是避免重复提交和乱序处理的基石。

3. **在开发阶段就搭建可观测性面板**。不要等上线后才想起监控。从第一个离线功能上线时，就要能统计同步延迟、冲突频次、错误模式。这会极大降低线上排查成本和用户投诉。

离线优先不是银弹，但它让应用在脆弱的网络环境下依然可靠。当用户发现“即使断网，我的待办也能安全无损地到达服务器”时，他们获得的不仅仅是功能，更是对产品设计的一种信任。这种信任，比任何炫酷特性都更值得投入。
