# 离线优先设计：当任务管理应用不再依赖“在线”这个假设

## 为什么离线优先不只是“断网也能用”

对于大多数前端工程师来说，“离线优先”这个词的第一反应可能是：本地存数据，没网也能操作。但这个理解只停留在功能层面。真正的离线优先设计，核心解决的是**用户对数据安全的信任问题**——用户需要确信，在离线状态下所做的任何操作，不会因为网络恢复而丢失或产生不可预知的冲突。

当一个任务管理应用承诺“离线优先”，本质上是向用户传递一个信号：你的数据是你的，无论网络状况如何，应用的响应速度和数据完整性都不会打折扣。这要求我们重新审视整个数据流的设计——不再以“服务器为真相源”，而是以“本地为真相源”，服务器只是同步的协作方。

## 三驾马车：IndexedDB、Service Worker、后台同步

实现离线优先，需要前端三剑客各司其职。

**IndexedDB** 承担的是“本地持久化存储”。它可以存储比 localStorage 大得多的结构化数据，支持索引和事务。在任务管理场景中，它负责保存所有任务数据、操作日志、同步状态。一个关键设计是：任务在 IndexedDB 里需要有版本号或时间戳字段，这是后续冲突检测的基础。

**Service Worker** 负责“网络代理”。它拦截所有 API 请求，在在线时透传，在离线时返回缓存数据。但它不直接参与业务逻辑，更像一个智能路由器。需要特别注意的是：Service Worker 不应该缓存 POST/PUT 请求的响应，而应该让这些写请求进入“后台同步”队列。

**后台同步（Background Sync）** 是离线优先的引擎。当用户离线时做的每个写操作（创建、修改、删除任务），都被序列化后存入队列。网络恢复时，按顺序逐个尝试与服务器同步。这里需要引入“重试队列”机制：同步失败的操作（如网络波动、服务端校验失败）应放入重试队列，按指数退避策略重试，而不是死循环或直接丢弃。

## 两类典型冲突场景

### 场景一：同一任务在多端被编辑

用户A在手机离线时修改了任务标题为“买牛奶”，用户B在电脑上同时（网络正常）修改相同任务标题为“买酸奶”。当两端都上线后，服务器会收到两个“更新”请求。

解决策略之一是“最后写入者获胜”结合时间戳。但问题是：离线端的请求有时间戳，但它是本地时间，可能与服务器时间不一致。更靠谱的做法是让服务器记录每次更新的版本号（或冲突向量），离线端同步时带上本地版本号，服务器比对后决定是否接受。

### 场景二：离线删除与在线更新交叉

用户离线时删除了任务A，但相同任务A在服务器上已经被用户B在线修改。当离线端同步删除请求时，服务器发现该任务在离线期间有更新——如果简单执行删除，用户B的修改就丢失了。

此时应该设计为“软删除+冲突提示”：离线端的删除请求转化成“标记删除”，服务器检测到冲突后，将删除操作和更新操作都保留，返回给离线端一个冲突记录，让用户手动裁决。或者更激进的做法：删除操作永远需要在线确认，离线时只允许“标记为待删除”。

## 简化数据流：从本地写入到最终一致

1. **本地写入**：用户操作（创建/编辑/删除）首先写入 IndexedDB，状态标记为“待同步”。
2. **排队**：同时将操作指令（包含操作类型、数据变更、本地时间戳、任务ID）加入后台同步队列。
3. **同步**：Service Worker 监听网络状态。当网络恢复，Background Sync 事件触发，队列中的操作按顺序发送到服务器 API。
4. **确认**：服务器处理成功，返回新的版本号或确认标记。本地更新该任务的同步状态为“已同步”。
5. **失败回滚**：如果服务器处理失败（如冲突、校验不通过），本地标记该操作为“同步失败”，同时保留用户本地数据不覆盖。可以设计一个“冲突面板”让用户查看失败原因并手动处理。
6. **重试**：指定时间内未被处理的失败操作，进入重试队列，逐步增加重试间隔。

## 可观测性：让离线状态不再是个黑盒

离线优先系统最怕的是“不知道数据在哪一步出了问题”。因此必须建立可观测性机制：

- **同步失败记录**：每条失败操作应记录失败原因、时间、本地数据快照、服务器返回的错误信息。存储在 IndexedDB 的一个专门集合中，供开发者和用户查阅。
- **冲突率**：统计“同步时遇到冲突的操作数 / 总同步操作数”。冲突率上升可能是多端编辑频率过高，或者时间戳/版本号设计不合理。
- **平均恢复时间（MTTR）**：从操作进入队列到最终同步成功（包括重试）的平均时间。这个指标能直观反映后台同步策略是否健康。如果 MTTR 持续偏高，可能需要调整重试间隔或优化冲突处理流程。

实现方式：可以在后台同步的每个步骤中埋点，将数据写入控制台日志（在线时发送到监控系统）或保存在 IndexedDB 中的事件流表。

## 落地建议

1. **先离线后同步，而非先在线后容灾**：设计阶段就把“离线可用”作为默认行为，而不是让在线作为常态、离线作为异常。这意味着所有的写操作都通过本地 IndexedDB 写入，再走后台同步，而非直接调用 API 再回写本地。

2. **冲突策略宁可笨，不能丢**：初期可以采用最简单的“最后写入者获胜”配合版本号，避免引入复杂的向量时钟或 CRDT，这些机制的维护成本对于轻量级应用过高。用户数量少、任务维度简单时，冲突很少发生，简单策略加上人工介入就足够了。

3. **为调试留后门**：提供一个“同步状态面板”（仅在开发环境或调试模式下），展示所有任务当前的同步状态、队列长度、失败记录。你不知道什么时候一个“离线写入未同步”的 bug 会消耗你一整天的排查时间。
