# 离线不是降级：设计一个值得用户信任的轻量级任务管理器

当团队选择“离线优先”架构时，最常见的误解是“断网时能打开页面就行”。实际上，离线优先的核心挑战在于 **数据一致性** 与 **用户信任**。以任务管理这类高频读写、多端协作的应用为例，用户不会关心你用的是 IndexedDB 还是 localStorage，他们只在意：**在隧道里修改的任务，出隧道后会不会丢失？手机和电脑同时编辑同一个任务，谁会赢？**

下面我们从架构设计的角度，拆解一个轻量级离线优先任务管理应用的核心要素，并给出可落地的实践建议。

---

## 一、为什么离线优先是信任问题

传统的“在线优先”应用中，用户的所有操作都直接依赖服务器响应。一旦网络抖动，页面可能陷入“loading”状态，用户无法确定操作是否生效。而离线优先应用承诺：**你的每一次点击都会立刻反馈，无论网络如何。**

这种承诺建立在两个前提上：一是本地写入必须可靠（不丢数据），二是同步时必须解决冲突（不覆写他人成果）。如果用户离线修改了任务标题，上线后却发现被其他人的删除操作覆盖了，他会立即丧失信任。因此，离线优先本质上是**状态管理与冲突治理**的问题，而非网络缓存问题。

---

## 二、三驾马车：IndexedDB、Service Worker、后台同步

### 2.1 IndexedDB：本地事实来源
IndexedDB 是离线持久化的基石。它不像 localStorage 那样有 5MB 限制，且支持结构化数据、索引和事务。在任务管理应用中，IndexedDB 存储三个核心集合：任务实体、本地变更日志（用于重试）、冲突记录。变更日志是关键——每次用户操作都会先写入本地乐观副本，同时记录一条“待同步”的操作日志，确保即使用户关闭页面再打开，未同步的操作也不会丢失。

### 2.2 Service Worker：网络代理与离线路由
Service Worker 拦截所有 API 请求。当网络可用时，它像透明代理一样转发请求；当网络不可用时，它返回 IndexedDB 中的缓存数据。更重要的是，Service Worker 负责**触发后台同步**——通过 `sync` 事件通知浏览器在恢复网络后执行重试。注意，Service Worker 不适合直接操作 IndexedDB（会引入竞态），它只负责排队和触发同步上下文。

### 2.3 后台同步与重试队列
这是离线优先的“心脏”。一个典型的重试队列设计包含：**操作类型（create/update/delete）、本地 ID、时间戳、冲突策略（如“以最后写入为准”或“合并内容”）**。队列由业务层维护，每次网络恢复后，按顺序将操作发往服务器。服务器返回成功则清除队列项；返回冲突则进入冲突处理流程；返回失败则根据错误类型决定重试次数或回滚。

---

## 三、两类典型的冲突场景

### 3.1 多端离线编辑同一任务
用户 A 在手机端离线修改任务标题为“调研方案”，用户 B 在桌面端离线同时将标题改为“方案评审”。两人都写入了本地 IndexedDB，并产生了变更日志。当网络恢复后，两条同步请求几乎同时到达服务器。服务器需要选择一个裁决规则：例如基于时间戳的策略（后到的覆盖先到的），或者基于操作类型的策略（编辑 vs 删除，删除优先）。无论哪种，都必须**通知落败方**——例如返回冲突记录，客户端收到后弹出提示：“你的修改被其他人的更新覆盖了，你可以选择手动合并或撤销。”

### 3.2 离线删除与在线更新交叉
用户 A 离线删除了任务 T，同时用户 B 在线更新了任务 T 的截止日期。当 A 的设备上线，同步删除操作时，服务器发现任务 T 已经被 B 更新过。如果直接删除，B 的更新就丢失了。合理的做法是：服务器记录“软删除”，并返回冲突信息给 A，告知“该任务已被其他用户修改，删除操作被拒绝”。A 的客户端收到后，回滚本地删除，重新显示任务并标记为“待确认”。

---

## 四、简化数据流：写入→排队→同步→确认→失败处理

1. **本地写入**：用户操作 → 更新 IndexedDB 中的乐观副本 → 同时写入变更日志（附带本地时间戳）。
2. **排队**：后台同步模块定期检查网络状态。如果在线，立即执行同步；如果离线，排队等待 Service Worker 的 `sync` 事件。
3. **同步**：按操作顺序逐个向服务器发送请求。每次请求携带本地 ID 和冲突策略标识。
4. **确认**：服务器返回成功 → 从变更日志中移除该项，同时更新本地副本的 `lastSyncedAt` 时间戳。
5. **失败回滚或重试**：
   - 网络错误（如 503）：标记为“可重试”，指数退避重试（最多 3 次）。
   - 冲突（如 409）：进入冲突处理模块，根据策略决定是否覆盖或提示用户。
   - 业务错误（如 404）：回滚本地变更，提示用户操作无效。

---

## 五、可观测性：同步失败、冲突率、平均恢复时间

没有可观测性的离线系统是黑盒。你需要监控三个关键指标：

- **同步失败率**：监控变更日志中标记为“失败”的项数量占比。如果超过阈值，可能意味着重试队列设计有问题或服务器端校验过于严格。
- **冲突率**：记录每次服务器返回 409 冲突的次数，除以总同步请求数。高冲突率提示用户操作习惯（例如多人同时编辑高频字段）或冲突策略不够包容。
- **平均恢复时间（MTTR）**：从用户产生离线操作到该操作被成功同步到服务器的时间差。该指标综合反映了网络恢复速度、重试间隔、冲突处理效率。若 MTTR 过长，需要优化重试算法或引入推送通知。

这些数据可以通过在 IndexedDB 中埋点记录，并通过后台同步附带元数据上报。不建议依赖前端日志库，因为离线期间日志可能丢失。

---

## 最后：三条项目落地建议

1. **从最小可行离线场景开始**：不要一开始就做全量离线。先支持“只读缓存”，再支持“前端突变 + 手动同步”，最后实现“自动后台同步”。每一步都验证用户信任度后再推进。
2. **为冲突准备“撤回”机制**：无论采用什么冲突策略，用户都需要一个“撤销”或“查看修改历史”的入口。在 IndexedDB 中保存最近 N 条变更快照，让用户可以恢复到上一次同步前的状态。
3. **将可观测性作为特性而非“监控”**：在管理面板中给用户展示“最近同步时间”、“未同步操作数”甚至“冲突记录”。这不仅能提升透明度，还能帮助运营发现设计缺陷，形成闭环。

离线优先不是技术的炫技，而是对用户“数据安全”的承诺。只有当每一次点击都有回应、每一次冲突都能被理解和解决，用户才会真正信任你的应用。
