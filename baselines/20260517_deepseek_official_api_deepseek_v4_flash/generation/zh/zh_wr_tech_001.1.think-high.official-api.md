# 离线优先：从“断网可看”到“信任无感”的任务管理设计

当用户打开一个任务管理应用，最糟糕的体验不是功能缺失，而是——明明改完了任务描述，点击保存后所有修改消失，页面只留一片空白。这种“数据幽灵”往往出现在网络抖动或短暂离线后，而用户会本能地归咎于应用“不可靠”。离线优先（Offline-First）并非仅仅让应用在断网时能展示缓存内容，它本质上是**用户信任的基建**：无论网络如何波动，用户的每一次操作都必须被尊重，数据最终应收敛到唯一正确状态。本文面向已有前端基础的全栈工程师，探讨如何为轻量级任务管理应用设计一套可靠的离线优先架构。

## 1. 离线优先的真正命题：信任与一致性

许多人误以为离线优先就是“断网也能打开页面”，这实际只是“缓存优先”的浅层认知。真正的离线优先意味着：

- **用户操作始终成功**：即便本地无法发出请求，应用也要即时响应用户的增删改，并在后台排队等待同步。
- **数据最终一致**：一个任务在A设备离线时被标记为“已完成”，在B设备在线时被修改了截止日期——这两个状态在恢复同步时如何正确合并，而不丢失信息？
- **用户无感恢复**：当网络恢复，冲突自动解决或给出清晰提示，让用户觉得“一切尽在掌控”，而不是“又出错了”。

信任来源于可预测的行为：用户需要确信，无论当下是否连网，他们付出的每一次点击都不会被悄悄吞噬。

## 2. 三大技术支柱：各司其职

要实现上述目标，需要三个核心组件协同工作：

**IndexedDB** 充当本地持久化数据库。它负责存储完整的任务实体、操作日志（Mutation Log）以及版本号。不同于 localStorage 的简单键值对，IndexedDB 支持事务、索引和游标，足以对离线数据进行查询和分页。任务管理应用中的每一个本地变更都应先写入 IndexedDB，确保页面刷新不丢失。

**Service Worker** 是网络请求的“代理层”。它拦截 fetch 请求，在在线时透传，在离线时返回缓存。但对于写请求（POST/PUT/DELETE），Service Worker 更重要的职责是**将请求的 payload 克隆并暂存**，以便后续重放。它不负责业务逻辑，只负责解决“请求发不出去时怎么办”的问题。

**后台同步与重试队列** 是“负责最终发送”的角色。后台同步（Background Sync API）允许在用户恢复网络后自动触发同步任务，无需用户主动刷新。而重试队列是一个存储在 IndexedDB 中的有序列表，记录每一条待同步的操作（操作类型、目标 URL、请求体、时间戳、重试次数）。同步模块会按队列顺序逐个发送，成功则移除，失败则根据错误类型决定是直接丢弃（如 404）还是按指数退避重试。

这三个组件的关系可以概括为：**IndexedDB 负责“存得住”，Service Worker 负责“接得住”，重试队列负责“送得到”**。

## 3. 典型冲突场景：离线不只是“一个人的事”

即使只有一个人使用，多设备登录也会产生冲突。以下两类场景最具代表性：

**场景一：同一任务在多端被编辑**
用户在手机端离线时修改了任务标题为“买苹果”，同时在电脑端在线修改了同一任务的描述为“记得买红色”。当两部设备先后同步，系统收到两条针对同一 taskId 的更新。若简单以最后到达的请求为准，一条修改将被覆盖。
- 解法：采用**最后写入胜利 + 字段级别冲突检测**。每个字段附带时间戳，同步时只合并未冲突的字段；若同一字段冲突，保留时间戳较新的值，并将旧值作为“已丢弃的版本”记入冲突日志，供用户事后审核。

**场景二：离线删除与在线更新交叉**
用户在离线时删除了一条任务，而在线的另一方同时更新了该任务的截止日期。当离线设备上线同步删除操作时，服务端发现目标任务已被更新——如果按顺序处理，更新后的数据会被删除覆盖，导致更新丢失。
- 解法：使用“软删除 + 墓碑机制”。离线删除只标记为 deleted = true 并附带删除时间戳。同步时，服务端比较删除时间戳与最后一次修改时间戳：若删除操作晚于修改，则真正删除数据；若删除操作早于修改，则说明删除是在“旧版本”上发出的，更新应胜出，删除操作被拒绝并返回冲突记录。客户端收到冲突后，可将被删除的任务恢复为“已更新”状态并提示用户。

## 4. 简化的数据流：写入、排队、同步、确认、回滚

整个流程可以抽象为五个阶段，每一步都围绕“用户无感知”设计：

1. **本地写入**：用户创建或修改任务，前端立即更新 IndexedDB 中的本地副本，并返回 UI 即时反馈（乐观 UI）。同时将操作记录（含操作 ID、时间戳、实体快照）插入同一条事务中的操作日志表。
2. **排队**：操作日志被推入重试队列。队列中每条记录都会标记为“待同步”。即使页面关闭，队列也会因 IndexedDB 的持久化而存在。
3. **同步触发**：当 Service Worker 监测到网络恢复，或页面主动通过 navigator.onLine 检查到在线，同步模块开始依次处理队列。若使用 Background Sync，系统会自动唤醒 Service Worker 执行同步。
4. **确认**：服务端收到请求后，返回带有最新版本号（或时间戳）的成功响应。客户端据此更新本地记录的本地版本号，并从队列中移除该操作。
5. **失败回滚或重试**：若同步失败，区分错误类型。对于 4xx 类冲突（如版本冲突），进入冲突处理流程，可能回滚本地修改并拉取服务端最新数据。对于网络超时、5xx 等临时错误，按指数退避（1s、2s、4s……）重新入队，重试次数上限可设为 5 次，超出则标记为“同步失败”，在用户界面显示一条非阻塞的红点提示。

## 5. 可观测性：你无法改进无法度量的东西

离线优先最大的陷阱是“沉默的失败” —— 用户以为保存了，实际同步从未成功。因此必须建立可观测性指标：

- **同步失败率**：统计重试队列中最终标记为失败的操作数量，除以总操作数。这个指标应低于 0.1%，否则说明冲突逻辑或服务端 API 设计有问题。
- **冲突率**：记录每次冲突解决的 tally，例如“字段级冲突次数”、“删除 vs 更新冲突次数”。冲突率过高可能意味着应用中存在多人协作的高频修改场景，此时应提示用户增加自动合并策略，或者引入“锁定”机制。
- **平均恢复时间（MTTR）**：从一次离线操作写入本地到该操作被服务端确认所经过的平均时间。如果 MTTR 长时间偏高（如超过 10 秒），可能是重试队列处理速度不足或服务端吞吐瓶颈，需要优化批量同步接口或增加 WebSocket 推送。

这些数据可通过收集 Service Worker 和同步模块的日志，汇总到分析工具（如自建的简单日志 API）中。前端工程师应能随时在管理后台查看当前队列长度、最近 24 小时冲突分布。

## 结语：三条落地建议

1. **从“乐观 UI + 本地优先”开始**，不要一开始就设计复杂的 CRDT 算法。先让所有写操作都写入 IndexedDB 并立即更新视图，然后用简单的“最后写入胜利 + 时间戳”处理冲突。绝大多数个人任务管理应用根本不会遇到真正的高并发冲突。
2. **为每一次操作赋予全局唯一 ID**（如 UUID）。这不仅是幂等性的基础，还能让你在调试时通过日志精确定位是哪个操作导致了冲突。没有 unique id 的离线系统就是一场噩梦。
3. **建立一个“同步健康”的检查点**：在应用设置或开发者工具中提供“查看同步队列”“强制同步”“放弃未同步变更”等按钮。让用户（尤其是内部测试人员）能手动干预，比全自动但没有逃生出口的设计更值得信赖。

离线优先不是锦上添花，而是将应用从“网络可达”的旧世界迁移到“以用户为中心”的新范式的桥梁。设计得当，用户甚至会忘记网络的存在——那才是真正的信任。
