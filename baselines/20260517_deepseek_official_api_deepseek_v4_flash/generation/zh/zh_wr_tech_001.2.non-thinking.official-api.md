# 离线优先任务管理：从数据信任到冲突解决的工程实践

## 为什么离线优先不只是“断网也能用”

在移动办公、弱网环境下，用户对“即使断网也能顺畅操作”的期待早已不是锦上添花。但很多团队将离线优先简单理解为“缓存一份数据到本地，网络恢复后再同步”。这种理解忽略了两个核心问题：**用户信任**与**数据一致性**。

用户信任来源于“所见即所得”——离线时创建的任务，必须在联网后原样存在，不能丢失、不能错误合并。同时，如果多端同时编辑，用户期望看到清晰的结果，而不是“你的修改被覆盖了”这样的惊吓。换句话说，离线优先的设计本质是在分布式环境下提供**本地优先的确定性**，让用户感觉数据始终是自己的、可预期的。这要求我们不仅要解决“存得下”，还要解决“拼得对”。

## 三驾马车：IndexedDB、Service Worker、后台同步队列

一个轻量级离线优先应用，不需要引入复杂的CRDT或OT算法，但需要明确三个基础设施各自扮演的角色：

- **IndexedDB** 是离线数据的“冷硬盘”与“本地真相”。所有用户操作（增、改、删）首先写入IndexedDB，并附带一个递增的本地操作时间戳和操作ID。它不负责网络通信，只保证即使浏览器强制关闭，数据也能恢复。任务列表、元数据、冲突日志都存储于此。

- **Service Worker** 是网络的“智能代理”。它拦截前端发起的同步请求，当网络不可用时，将请求暂存到缓存队列；当网络恢复时，自动发送。更关键的是，Service Worker可以拦截来自其他设备（如同账号另一台电脑）的推送通知或后台同步事件，从而触发本地数据合并。它的存在让应用在离线时依然能“假装有网络”——用户点保存按钮，界面立即响应，而真实请求由Service Worker排队处理。

- **后台同步/重试队列** 是“调度器”。Service Worker只能暂存请求，但无法处理业务逻辑（比如冲突检测）。因此我们需要一个独立的同步模块（可以是Web Worker或主线程的微任务队列），它负责从IndexedDB中读取尚未同步的操作，按照顺序逐一发送到服务器，并处理响应。如果服务器返回冲突或失败，该模块将操作放回重试队列，并附带指数退避策略；如果成功，则更新本地记录的同步状态。这个模块同时负责监听服务器的“在线状态变更”事件，触发新一轮同步。

## 两类典型的冲突场景

### 场景一：同一任务在多端被编辑

用户A在手机端（离线）修改了任务<撰写周报>的标题为“周报V2”，用户B在电脑端（在线）同时修改了同一任务标题为“周报终稿”。当A联网同步时，服务器发现该任务已有最新版本（基于最后修改时间戳判断）。此时最简单的策略是**最后写入者胜出（LWW）**，但会造成一方数据丢失。更优的方案是：服务器将两个版本都保留，返回“冲突”标记，前端根据预设规则（如保留本地版本、保留服务器版本、或提示用户手动合并）进行决策，并将选择结果作为新版本再次同步。对于轻量级任务管理，通常采用“本地优先”策略——用户应当信任自己本地看见的内容，因此**本地版本覆盖服务器版本**，同时将服务器版本作为历史版本保存在IndexedDB中，供用户事后查阅。

### 场景二：离线删除与在线更新交叉

用户A在离线时删除了任务<买咖啡>，用户B在在线时修改了该任务的截止日期。当A联网后，同步Delete操作到服务器，服务器需要判断：该任务是否已被其他用户修改过？如果直接删除，B的修改将丢失。合理做法是：服务器不立即删除，而是标记“软删除”，并返回冲突响应。前端收到冲突后，将本地删除操作暂缓，提示用户“该任务已被他人更新，是否仍要删除？”用户确认后，再发送强制删除请求。反之，如果用户不确认，本地保留的删除操作则被回滚，从IndexedDB中恢复该任务。

## 简化的数据流：写入、排队、同步、确认、回滚/重试

1. **本地写入**：用户在界面操作后，数据立即更新到本地IndexedDB，同时生成一个包含操作类型、操作ID、时间戳、数据快照的记录，标记为“待同步”。
2. **排队**：同步模块从IndexedDB中按操作顺序（按本地时间戳升序）取出所有待同步记录，放入内存队列。
3. **同步**：逐条发送到服务器API，携带操作ID和冲突检测条件（如version字段）。
4. **确认**：服务器返回200 OK，同步模块将该记录标记为“已同步”，并从等待队列中移除。同时更新本地数据的时间戳与版本号。
5. **失败或回滚**：若服务器返回冲突（409）、网络错误或超时，同步模块将该记录放入重试队列，应用指数退避（1秒、2秒、4秒……最多5次）。若重试仍然失败，则标记为“同步失败”并通知用户。对于服务器明确拒绝的操作（如权限问题），直接标记为“不可恢复”，并将本地数据回滚到前一个合法版本（例如从IndexedDB的历史快照中恢复）。

## 可观测性：让离线状态可见、可度量

离线优先应用最大的隐患是“静默失败”——用户以为自己保存了，实际上数据从未离开过本地。因此必须建立可观测性体系：

- **同步失败率**：计算一段时间内（如过去1小时）同步失败的操作数占总操作数的比例。当失败率超过阈值（如5%）时，触发告警，提示服务器或网络可能存在问题。
- **冲突率**：统计服务端返回冲突响应的次数与总同步次数的比值。高冲突率意味着多端协作频繁，可能需要优化冲突策略或引导用户减少并发操作。
- **平均恢复时间（MTTR）**：从操作进入重试队列到最终成功（或被用户手动处理）的平均时间。如果MTTR过长，说明重试策略或网络条件需要改进。
- **本地队列长度**：IndexedDB中待同步记录的数量。如果队列持续增长且远超正常值（比如超过100条），可能意味着同步流程出现瓶颈或服务器不可达。

以上指标可以借助浏览器的performance observer、自定义日志（输出到控制台或远程日志服务）以及用户主动触发的“同步状态面板”来呈现。不一定要做成炫酷的监控大屏，但在开发者工具中清晰暴露关键数据，对排查问题和优化体验至关重要。

## 三条落地建议

1. **从“乐观写”开始，再逐步引入冲突处理**
   初期不要追求完美的多端合并。先实现“本地优先+最后写入者胜出”，保证用户离线写数据不丢失，然后通过冲突提示让用户自己选择版本。随着用户量增长，再迭代软删除、历史版本回溯等机制。

2. **将同步队列的幂等性作为核心设计原则**
   每个操作应该携带全局唯一的ID，服务器端实现幂等：收到重复的操作ID时，直接返回成功，避免因网络重试导致重复执行。本地同步模块也应支持“跳过已同步的操作”，防止因快照不一致导致的无限循环。

3. **优先暴露同步状态，而不是隐藏复杂性**
   不要试图让用户完全感知不到离线。相反，在界面角落显示“3条待同步”“同步失败：点击重试”等轻量提示，让用户对数据状态有掌控感。用户能接受的延迟远高于不可预知的数据丢失。
