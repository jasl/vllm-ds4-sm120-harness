# 离线优先的信任契约：构建轻量级任务管理应用的数据一致性架构

对于多数前端工程师而言，“离线优先”往往被简化为“断网时也能看到缓存页面”。但当我们要构建一个真正可用的任务管理应用时，这条定义远远不够。用户会把关键任务托付给应用，他们关心的不是“能不能看到”，而是“我编辑的内容会不会丢失”“多端编辑会不会乱掉”。离线优先的本质，是在网络不稳定的世界里重建用户的信任——这要求我们同时解决数据持久化、同步一致性、冲突处理和可观测性四层问题。本文将从架构角度拆解一个轻量级离线优先任务管理应用的设计思路。

## 一、离线优先的三块基石

离线优先并非单点技术，而是三组基础设施的协同工作。

**IndexedDB 是可信的本地权威数据源。** 浏览器缓存（如 Cache Storage）主要用于静态资源，不适合存放结构化业务数据。IndexedDB 提供了事务支持与索引查询，可以用来存储任务实体、同步队列、本地时间戳和版本号。每次用户操作，应用应该先写入 IndexedDB（乐观更新UI），再考虑同步——这确保即便立刻断网，数据也不会丢失。注意，IndexedDB 的读写本身是异步的，设计时需处理好竞态条件，比如在连续快速操作场景下借助事务批次提交。

**Service Worker 是离线能力的调度中心。** 它的作用不止是拦截网络请求并返回缓存——更关键的是，它可以在后台（即便浏览器标签页关闭）监听连接恢复事件，触发同步流程。Service Worker 生命周期独立于页面，因此它适合管理那些不需要用户主动交互的同步任务。不过要注意，Service Worker 中的 IndexedDB 操作需要显式打开数据库，且其全局作用域容易让人忽略状态泄漏问题，建议每个同步任务都独立获取连接。

**后台同步与重试队列是保证最终一致性的核心。** 仅靠 Service Worker 的 `sync` 事件不够——它只保证“最终会尝试一次”，但无法处理失败重试。我们需要自定义一个重试队列，通常也存放在 IndexedDB 中。每条记录包含操作类型、任务ID、变更内容、本地版本号以及重试次数和退避时间。每次同步失败后，队列根据指数退避策略（例如 1s、2s、4s……上限 1 分钟）安排下一次尝试，直至达到最大次数后转入“失败待人工处理”状态。

## 二、两类典型的冲突场景

有了基础设施，冲突处理才是离线优先应用真正的分水岭。以下两种场景在任务管理应用中几乎必定发生。

**场景一：同一任务在多端被编辑。** 用户 Alice 离线时将任务标题改为“买牛奶”，同一时刻 Bob（或同一用户的另一个设备）在线将同一任务的标题改为“买面包”。当 Alice 设备恢复连接后，同步请求到达服务器，服务器面临着两个版本的冲突。最简单的策略是“最后写入者获胜”（LWW），即比较服务器收到请求的时间戳，后到的覆盖先到的。但这会丢失 Bob 的修改。更稳妥的做法是引入版本向量：客户端本地维护一个“服务器版本号”，每次同步时带上这个版本号；服务器校验如果客户端的版本号小于当前服务器版本号，则判定冲突，返回冲突标志和服务器当前数据，让客户端决定是自动合并（例如任务管理场景可以先记录两版标题，收在备注里）还是弹窗让用户选择。

**场景二：离线删除与在线更新交叉。** 假设 Alice 离线删除了任务 T，同时 Bob 在线将 T 的截止日期改为了明天。当 Alice 设备同步时，发送了删除指令；但 Bob 的更新指令也在同一时间被同步服务器接收。服务器收到顺序不确定。如果先处理删除，再处理更新，更新会往一个已删除的任务上写数据，造成幽灵记录。解决方案是采用“软删除”：删除时不真正移除数据库记录，而是标记一个 `deleted` 字段和删除时间戳。当冲突发生时，服务器可以比较两个操作的时间戳：如果删除时间早于更新时间，则视为删除无效，保留更新；反之，则真正删除并返回 410。客户端收到 410 后，需要从本地同步队列中丢弃该任务的后续变更，并清除本地数据。

## 三、简化数据流：本地写入到最终一致

一个设计良好的同步数据流应像过河一样有清晰的路标。下面是理想流程：

1. **本地写入**：用户在界面上创建/修改/删除任务。应用立即更新 IndexedDB 中的任务实体（乐观渲染），同时将操作记录（包含 `operationType`、`taskId`、`newData`、`localVersion` + `serverVersion` 副本）插入到同步队列的“待同步”表。

2. **排队与调度**：若浏览器在线，借助 `navigator.onLine` + 事件监听，立即触发队列消费；若离线，队列静候。Service Worker 注册 `sync` 事件，当连接恢复后台自动拉起。

3. **同步**：队列消费者每次取出一个变更记录，附带当前本地记录的 `serverVersion`，向 API 发送请求（例如 PUT /tasks/:id 带上版本号）。服务器检查版本号：匹配则更新并返回新的 `serverVersion`；不匹配则返回冲突响应。

4. **确认与回滚**：收到 200，客户端将该变更标记为“已同步”，并更新本地任务的 `serverVersion`。收到冲突，客户端根据预设策略（如自动合并或弹窗）处理，生成新的变更记录入队，并标记旧记录为“冲突已解决”。收到失败（网络错误、500等），记录重试次数，按退避策略延迟重试；若重试耗尽，标记为“同步失败”，通知用户手动处理。

## 四、可观测性：让同步过程透明可追

离线优先应用的用户最怕“悄无声息的数据丢失”。因此，我们必须让同步状态可见。可观测性设计分两层：

**用户层**：状态栏图标显示当前同步进度（比如“3个变更待同步”“同步中…”），点击可展开详情。对于失败条目，提供“重试”或“查看详情”入口。冲突发生时，给出清晰的差异对比（例如原标题 vs 新标题）让用户选择。

**技术层**：在 IndexedDB 中建立一个“同步日志”表，每条日志记录：变更 ID、操作类型、请求时间、响应状态（成功/冲突/失败）、服务器返回的冲突详情、重试次数。定期（或每次同步完成）将这些日志聚合上报到服务端的可观测性平台。关键指标包括：**同步失败率**（失败次数 / 总同步次数）、**冲突率**（冲突次数 / 总同步次数）、**平均恢复时间**（从第一次失败到最终成功的时间间隔的均值）。这些数据可以帮助团队发现合并策略的缺陷（例如 LWW 导致频繁覆盖）、网络稳定性问题，或服务器接口的瓶颈。

## 五、三条项目落地建议

1. **开始用最简单的冲突策略**：对任务管理而言，绝大多数场景可以用“最后写入者获胜 + 版本号校验”兜底。不要在 MVP 阶段引入 CRDT 或自定义合并算法，先让用户能正常编辑、同步，并通过日志观察冲突发生的频率和类型，再决定是否升级策略。软删除是必须的，它成本极低却能解决大量删除-更新交叉问题。

2. **将同步队列状态机显式暴露给 UI**：每个任务的生命周期应包括 `synced`、`pending`、`conflict`、`failed` 四种状态，UI 上对应不同图标（绿色对勾、灰色旋转、黄色问号、红色感叹号）。用户看到“失败”状态时，能点击一键重试或查看冲突详情。不要让队列在后台默默工作却无据可查。

3. **离线测试要写入自动化流程**：使用 Chrome DevTools 模拟离线、弱网、断线恢复，但不能只靠手动测。编写端到端测试（如使用 Puppeteer + 模拟网络条件），覆盖：正常离线编辑 → 恢复后同步成功；多端同时编辑同一任务 → 触发冲突并得到可理解的提示；离线删除 + 在线更新 → 数据不出现幽灵记录。测试通过后再发布。

离线优先不是一个时髦的前缀，而是一套围绕“信任”设计的工程决策。当你把每一步同步都变成可见的、可重试的、可解释的过程时，用户会真正把任务交给你——哪怕他们正在飞机上、隧道里，或者信号极差的会议室。
