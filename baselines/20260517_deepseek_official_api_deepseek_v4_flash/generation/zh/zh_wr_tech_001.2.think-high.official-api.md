# 离线优先的任务管理：从数据孤岛到用户信任的工程实践

当你的用户在地铁隧道里创建了一个任务，两分钟后回到地面，却发现它消失了——这不是技术故障，而是你亲手破坏了他对你的信任。离线优先（Offline-First）从来不只是“断网也能用”的技术噱头，它本质上是一场关于**数据主权与一致性**的契约：用户承诺他们的操作会被尊重，而你承诺在连接断断续续的现实网络中，不会让任何一次点击成为孤儿。

对于有一年以上经验的前端或全栈工程师来说，构建一个轻量级的任务管理Web应用是一个绝佳的演练场：它足够简单，却能折射出离线优先设计的几乎所有核心挑战。下面我们从基础设施、冲突处理、数据流和可观测性四个维度拆解设计思路。

## 一、三大基础设施的职责分工

离线优先的架构依赖三个彼此独立又协同工作的组件：IndexedDB、Service Worker 和后台同步/重试队列。它们的分工可以用一个类比理解：**IndexedDB 是离线仓库，Service Worker 是离线路口的交警，而重试队列则是失物招领处**。

### 1. IndexedDB：本地持久化的“真相源”
IndexedDB 是浏览器端的结构化数据库，它承担了**离线状态下的主存储**角色。当用户创建或修改任务时，数据首先完整写入 IndexedDB，而不是直接发起网络请求。这意味着即使用户立刻断网，页面刷新后数据依然存在。设计时要注意三点：
- **Schema 应该与 API 模型松耦合**：本地需要额外的元数据字段，例如 `syncStatus`（pending / synced / failed）、`localUpdatedAt` 等，这些字段不会发送到服务端，但用于跟踪本地的未同步变更。
- **事务性写入**：任务管理中的排序、标签关联等操作往往涉及多个对象，利用 IndexedDB 的事务机制保证原子性，避免半写入状态。

### 2. Service Worker：网络请求的“代理与缓存”
Service Worker 负责拦截页面发出的所有网络请求（包括 API 调用），并决定如何响应。在离线优先设计中，它的主要职责不是缓存静态资源，而是**为数据同步提供“温和的桥接”**：
- 当网络可用时，它可以将同步请求转发给后台同步 API（Background Sync）；
- 当网络不可用时，它直接返回一个“离线排队成功”的占位响应，让前端无需感知网络状态。
- 更重要的是，Service Worker 的独立生命周期使其能在页面关闭后继续完成同步，这是纯前端代码做不到的。

### 3. 后台同步与重试队列：网络恢复后的“反熵机制”
**后台同步（Background Sync）** 是 Service Worker 中注册的一次性事件——当浏览器检测到网络恢复时，触发该事件，执行之前排队的同步请求。如果同步失败，则进入**重试队列**，通常采用指数退避策略（例如第一次等待5秒，第二次30秒，第三次5分钟）。这个队列需要具备幂等性：同一个任务插入请求被重复执行时，服务端应该能够去重（例如通过客户端生成的 UUID 作为任务ID）。

## 二、两类典型的冲突场景

离线优先应用的最大痛点不是技术实现，而是**用户行为的时空错位**。同一个任务可能在多个设备上被编辑，也可能在离线时被删除，而在线上又被他人更新。下面是两个必须处理的缩影。

### 场景一：多端编辑同一任务
用户 A 在手机（离线）上将任务“买咖啡”的截止时间改为下午3点，同时用户 B 在电脑上将其改为下午5点。当 A 的设备恢复网络，两个版本都试图同步到服务端。**冲突的本质是：谁对最终数据拥有发言权？**

常用策略有：
- **最后写入者胜利（LWW）**：依赖时间戳，但极易丢失用户意图。如果 A 比 B 晚同步，A 的修改会覆盖 B 的，即使 A 的操作时间更早。
- **主动合并**：对于任务标题这类冲突，可以取两个版本并让用户手动选择（类似 Git 合并冲突）。但对于截止时间这样只能取一个值的字段，通常需要结合业务规则（例如取较晚的截止时间，或者取较早的）。

### 场景二：离线删除与在线更新交叉
用户离线时删除了一个任务，但该任务在线状态下被另一个团队成员添加了评论。当删除请求同步到服务端时，应该如何处理？**直接删除会丢失评论，保留则会破坏用户“已删除”的心理模型。**

合理的做法是引入一个**软删除 + 墓碑（Tombstone）** 机制：本地删除操作实际是向服务端发送一个“标记删除”指令，服务端将该任务标记为 deleted，但保留其历史数据（包括评论）。同时，如果服务端检测到在线更新（评论）发生在离线删除之后，可以返回一个冲突警告，让用户选择是“强行删除并丢弃评论”还是“撤销删除并保留评论”。

## 三、简化的数据流：从本地写入到最终一致

我们以一个任务创建操作为例，梳理完整的预期数据流：

1. **本地写入**：用户在断开网络的情况下点击“新建任务”。前端立即在 IndexedDB 中插入一条记录，状态为 `pending`，并分配一个客户端生成的 UUID。UI 渲染出这条任务，用户感知到操作已经成功。
2. **排队**：同时，一个包含任务数据的同步请求被放入 IndexedDB 中的同步队列表（或在 Service Worker 中注册同步事件）。此时任务处于“待发送”状态。
3. **同步触发**：当网络恢复时，Service Worker 的 `sync` 事件被触发，遍历队列中的待同步项。对于每一条，向服务端发送 HTTP PUT 或 POST 请求。
4. **确认**：服务端收到请求后，验证数据格式、检查冲突，并返回带有服务端ID（如果原本没有）、时间戳和确认状态的响应。前端收到确认后，将本地记录的 `syncStatus` 改为 `synced`，并用服务端返回的时间戳更新本地。
5. **失败回滚或重试**：如果同步失败（如服务端返回 500 或超时），该请求不会立即丢弃，而是**回滚**：本地任务状态变更为 `failed`，同时将错误原因写入一个日志对象。重试队列会在退避后再次尝试。超过最大重试次数后，任务标记为 `permanent_failed`，并通知用户手动处理。

## 四、可观测性：看得见的“数据健康度”

离线优先系统像一座冰山：用户只看到 UI 的流畅性，而工程团队需要关注水下的同步质量。以下是三个必须监控的维度：

### 1. 同步失败率（Sync Failure Rate）
定义：单位时间内同步请求失败的次数 / 总同步请求次数。失败原因需要细分：网络错误、服务端冲突、数据格式错误、超时等。建议在 IndexedDB 中为每条同步记录添加 `lastError` 和 `retryCount` 字段，并通过定期上报或埋点日志发送到后端监控系统。

### 2. 冲突率（Conflict Rate）
定义：需要用户介入解决的冲突次数 / 总同步成功次数。如果这个值突然升高，说明离线编辑的时间窗口过长，或者多端协作场景变得复杂。可以通过服务端返回的 `conflict` 状态码来统计，并标记冲突类型（编辑冲突、删除冲突等）。

### 3. 平均恢复时间（MTTR - Mean Time to Recovery）
定义：从同步失败开始到该任务被成功同步（或人工解决）的平均时间。这个指标衡量了重试机制和用户自愈流程的效率。可以通过记录每条同步失败的 `createdAt` 和 `resolvedAt` 来计算。如果 MTTR 超过设计阈值（如5分钟），说明退避策略或通知机制需要优化。

## 五、三条项目落地建议

最后，无论你的任务管理应用规模多大，以下三点能帮助你避免常见的坑：

1. **从第一天起就设计客户端 ID 和数据版本号**：不要依赖服务端自增 ID 来识别离线数据。使用 `uuid` 作为主键，并给每条记录一个单调递增的版本号（或最后修改时间），这是所有冲突解决策略的基础。
2. **对用户透明但不骚扰**：在 UI 上明确显示“已保存到本地”和“等待同步”的状态（例如小圆点或进度条），但不要弹出模态框打断用户。只有当同步永久失败或发生严重冲突时，才用非侵入式通知引导用户手动解决。
3. **定义“同步边界”**：并非所有数据都需要离线可用。对于任务管理应用，任务的元数据（标题、截止时间、负责人）应该离线同步，而附件、评论的长列表可以按需按需下载，避免 IndexedDB 存储爆炸。在需求评审时就和产品确认哪些字段属于“离线核心字段”。

离线优先不是一锤子买卖，它是一次对用户体验的长期投资。当你的用户在地铁里创建的那条任务，在出站后依然安静地躺在列表里时，他收获的不仅仅是一个功能，而是一份信任。而这份信任，正是通过每一行对 IndexedDB 的写入、每一次 Service Worker 的重试、每一个冲突合并算法的精心设计累积起来的。
