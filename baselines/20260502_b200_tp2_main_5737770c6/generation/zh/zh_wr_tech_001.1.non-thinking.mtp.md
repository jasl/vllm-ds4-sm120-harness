# 离线优先，不只是“断网能用”：轻量级任务管理应用的设计哲学

对于前端工程师来说，离线优先（Offline-First）常常被简化为“在没网的时候也能看到页面”。然而，在真正构建一个面向日常使用的任务管理应用时，我们会发现，离线优先的核心价值远不止于此——它关乎**用户对数据的信任**，以及**跨设备、跨网络环境下的数据一致性**。

## 离线优先的真正挑战：信任与一致性

当用户离线创建了一个任务“下午3点开会”，又在另一台设备上同步删除了同一个任务，那么当两台设备都恢复联网时，系统该如何抉择？这类问题才是离线优先设计的真正难点。

用户不会关心底层的同步算法，但他们能直观感受到“我明明删了，为什么又出现了”或“我明明添加了，为什么丢了”。这种体验一旦出现，用户对应用的信任将迅速瓦解。因此，离线优先的设计目标不是“尽可能多地同步”，而是“让用户觉得数据从未混乱过”。

## 三大基础设施：分工明确的离线能力

要构建可靠的离线优先应用，我们需要三种核心能力，它们各有职责，不能混为一谈。

### IndexedDB：离线存储的基石

IndexedDB 负责在浏览器端持久化数据。它的职责不是“缓存”，而是“本地数据副本”。任务列表、任务状态、更新时间戳等关键数据都应该存储在 IndexedDB 中。当我们说“离线也能用”，本质上是让用户在断网时依然能完成 CRUD 操作，而这些操作都作用于本地的 IndexedDB 副本。

### Service Worker：网络代理与拦截

Service Worker 的作用是接管应用发出的网络请求。对于离线优先场景，它需要做到：
- 拦截 API 请求，当在线时转发至服务器，获取最新数据后更新 IndexedDB；
- 当离线时，直接返回 IndexedDB 中的数据，或者请求失败时提供友好的降级响应。

**关键点**：Service Worker 不应承担数据冲突解决的逻辑，它只负责请求的路由与缓存策略。冲突解决应该在更高的应用层处理。

### 后台同步与重试队列：让离线操作最终落地

当用户在离线状态下修改了任务（例如标记为已完成），这个修改需要被记录为一个“待同步操作”。这个过程通常由一个队列来管理，队列中的每一项包含：
- 操作类型（创建、更新、删除）
- 操作数据（修改后的任务内容）
- 时间戳
- 重试次数

当网络恢复时，后台同步机制（例如 Background Sync API）会触发队列的消费。每个操作按顺序执行，一旦失败则标记重试，直至成功或达到上限。

**重试队列的设计核心**：不是“尽力而为”，而是“可追溯的可靠性”——每次失败都要记录原因，为后续分析和用户反馈提供依据。

## 冲突场景：离线设计的真正试金石

### 场景一：同一任务在多端被编辑

用户A在手机离线时修改了任务“买牛奶”的完成时间为“今天下午4点”，用户B在电脑在线时将其修改为“今天下午5点”。当手机恢复联网后，系统面临冲突。

一个简单的策略是**“最后写入者胜出”**，但这会丢失用户B的修改。更好的做法是引入**操作向量时钟**或**基于时间戳的合并**，但更务实的方式是：**记录两个版本，显示冲突标记，让用户手动选择**。对于轻量级任务管理应用，后者更易实现且更符合用户直觉。

### 场景二：离线删除与在线更新交叉

用户离线时删除了一个任务，但在线期间另一个用户修改了该任务。当离线用户联网后，删除操作与更新操作同时到达服务器。这里的问题在于：删除操作通常意味着“该数据不存在”，而更新操作则暗示“该数据存在”。

一种可行的处理方式是：**将删除操作视为一种特殊的“更新”**（将状态标记为“已删除”），而不是真正从数据库移除记录。这样，当冲突发生时，系统可以检测到“一方已删除”和“另一方已更新”，并向用户提示：“该任务已被删除，但你同步了修改，是否恢复？” 这种方式避免了数据凭空消失，也保留了用户操作的信息。

## 简化数据流：本地写入 → 排队 → 同步 → 确认 → 回滚/重试

一个典型的数据流如下：

1. **本地写入**：用户修改任务，数据立即更新到 IndexedDB。用户感受到秒级响应。
2. **排队**：修改被包装为一个操作项，插入重试队列（同样存储在 IndexedDB 中），等待网络消费者处理。
3. **同步**：Service Worker 或后台同步 API 检测到网络可用，开始依次消费队列中的操作。
4. **确认**：服务器成功处理操作后，返回确认响应。本地队列清除该操作项。
5. **失败回滚或重试**：如果服务器返回冲突、超时或错误，操作项保留在队列中，并记录失败原因和重试次数。达到上限后，标记为“永久的同步失败”，并在界面上展示给用户。

这个流程的关键在于：**本地操作先于服务器确认。** 用户不会因为网络问题而等待，而服务器返回的结果会最终决定数据的一致性状态。

## 可观测性：如何知道你的离线设计是否可靠？

离线优先系统的隐患常潜伏在边界条件中。要让团队可观测系统的健康状况，需要关注以下指标：

- **同步失败率**：重试队列中最终标记为失败的操作占总操作的比例。超过 1% 应触发告警。
- **冲突率**：服务器检测到的冲突数量 / 总同步次数。高冲突率可能意味着多端并发修改频繁，或冲突解决策略过于粗暴。
- **平均恢复时间**：从网络恢复连接到队列消费完成的时间差。这个指标能反映后端接口的响应能力和队列处理的效率。

实现方式上，可以在队列消费的每个环节埋点，将失败事件、冲突类型、错误堆栈等信息上报至监控平台（如 Sentry 或内部日志系统）。这些数据不仅能用于排障，还能指导冲突解决策略的优化。

## 项目落地建议

1. **从“乐观 UI”开始，再逐步完善离线逻辑。** 不要试图一开始就构建完美的离线同步引擎。先让用户的操作在本地即时响应，然后逐步加入网络校验和冲突处理，这比一次性实现全套方案更可控。

2. **为每个操作赋予全局唯一 ID 和精确时间戳。** 没有唯一标识，就无法正确追踪操作；没有时间戳，就无法进行顺序化和冲突检测。这是所有离线设计的基础设施，务必在架构设计阶段就确定下来。

3. **在用户界面中提供清晰的状态指示。** 用户有权知道“当前是离线状态”“此任务正在同步”“此处存在冲突”。不要将同步行为隐藏起来。透明度本身就是对用户信任的保护。
