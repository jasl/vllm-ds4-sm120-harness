# 离线优先的轻量级任务管理应用：不止是“断网可用”

## 为什么离线优先不只是“断网也能用”

很多团队对“离线优先”的理解停留在“网络断了还能打开页面”，但这只是最浅层的能力。真正值得关注的是：**用户是否敢把重要数据托付给你的应用**。

试想一个场景：移动端工程师在地铁里修改了一个任务的截止日期，切换4G后另一个浏览器窗口也在编辑同一任务。此时如果没有良好的离线架构，后者提交时会直接覆盖前者，且没有任何提示。用户会立刻失去信任——他们不再认为你管理的数据是“确定的”。

离线优先的本质，是**在网络不稳定甚至断网的情况下，依然能保障数据写入的确定性与最终一致性**。它意味着应用承诺：“你在我这里做的每一次操作，都不会因为网络波动而丢失或错乱。” 这种信任一旦建立，用户对应用的依赖程度会显著提升，尤其在差旅、工地、偏远地区等场景下，离线能力甚至成为选型的决定性因素。

## 三块核心基建：各司其职

要实现离线优先，需要三条技术路线协同，职责清晰，互不重叠。

**IndexedDB 是“本地事实”**。当用户执行创建、修改、删除时，数据首先写入 IndexedDB，页面立即以本地数据呈现。它不关心网络，只保证写入持久化。这意味着即使浏览器关闭再打开，本地修改依然存在。IndexedDB 的另一价值是存储同步状态：每条记录可以附带一个 `syncStatus` 字段（如 `pending`、`synced`、`conflict`），方便后续追踪。

**Service Worker 是“离线管家”**。它拦截网络请求，在离线时从缓存返回静态资源，并在后台管理数据同步。但一个常见误区是让 Service Worker 直接处理写入逻辑——这会让调试变得混乱。正确的做法是：Service Worker 只负责资源缓存和触发后台同步，而具体的读写操作交给主线程或 Web Worker 处理。

**后台同步/重试队列是“纠错引擎”**。当网络恢复时，主线程将 `pending` 状态的本地记录逐条发送到服务器。发送失败（如超时、服务器拒绝）的记录进入重试队列，按指数退避策略重试，同时更新 `syncStatus` 为 `failed` 并记录失败原因。重试队列需要支持持久化，避免浏览器重启后忘记未完成的任务。

## 冲突场景：离线设计的照妖镜

**场景一：同一任务多端编辑**。工程师 A 在离线状态下将任务标题从“修复登录 bug”改为“修复注册 bug”，而 B 在线状态下将同一任务的优先级从高改为中。同步时，服务器需要判断哪个字段是冲突的。一种策略是字段级合并：标题按 A 的，优先级按 B 的，双方都不全量覆盖。这需要服务器返回合并后的版本，并标记用户 A 和 B 各自修改的部分。

**场景二：离线删除与在线更新交叉**。用户在离线时删除了一个任务，而在线时同事更新了该任务的描述。同步时，服务器收到两条记录：一条说“删除”，另一条说“更新”，且两者基于不同的版本号。合理的设计是：以删除优先——因为“我明确要移除它”的意图比“我还要修改它”更强烈，除非更新记录的时间戳显著晚于删除操作，此时需提示用户确认。

冲突的本质是用户意图的交锋。没有银弹，需要根据业务语义选择优先级规则，并确保用户知情。

## 简化数据流：从本地写到最终一致

整个数据流可以概括为五步：本地写入 → 排队 → 同步 → 确认 → 重试或回滚。

1. **本地写入**：用户操作后，立即更新 IndexedDB，UI 表现为“已保存”。
2. **排队**：将变更记录放入本地同步队列，每条记录包含操作类型、字段、时间戳、本地生成的 UUID。
3. **同步**：Service Worker 监测到网络恢复，通知主线程从队列中取数据，发送给服务端 API。
4. **确认**：服务器返回 `200 OK` 和当前版本号，主线程更新 IndexedDB 的 `syncStatus` 为 `synced`。
5. **重试或回滚**：若服务器返回 `409 Conflict`，则主动拉取服务端最新数据，在本地进行字段级合并，并将合并结果展示给用户确认后重新提交。若服务器超时，记录失败原因，30秒后重试，最多重试 5 次。

每一步都要记录日志，尤其要区分“未发送”和“发送但失败”两种状态，这是后续排查的起点。

## 可观测性：让故障不再隐藏

离线优先最大的陷阱是“静默失败”——应用看起来正常，但数据从未同步到服务器。要解决这个问题，需要建立三个关键指标：

- **同步失败率**：同步队列中最终状态为 `failed` 的记录数占总写入记录数的比例。这个指标一旦超过 0.5%，就值得排查服务端接口或网络环境。
- **冲突率**：服务器返回 `409` 的次数占同步请求数的比例。如果冲突率持续高于 5%，说明多端编辑场景需要优化，比如引入更细粒度的锁或更友好的冲突提示。
- **平均恢复时间**：从同步失败到重试成功的时间间隔。如果平均恢复时间超过 10 分钟，说明重试策略可能过于保守，可以调整重试间隔或增加重试次数。

这些指标可以暴露给运维仪表盘，甚至在前端开发模式下显示给用户——至少在“同步状态”区域给出文字提示，让用户感知到数据是否安全。

## 三条落地建议

1. **先做单向同步，再优化冲突**：初期只支持“离线写入→上线同步”，暂不处理多端冲突。先让用户感受到离线写入的确定性，再逐步加入字段级合并和冲突提示，避免过度设计。

2. **优先保障本地写入性能**：IndexedDB 操作不要阻塞 UI 渲染，写入失败时 UI 应显示“本地保存失败”而不是“网络错误”。离线优先的第一优先级是本地可靠，网络同步是第二优先级。

3. **给用户一个“同步状态”面板**：在页面底部或侧边栏常驻一个区域，显示“已同步”、“待同步 X 条”、“同步失败 Y 条”。这不仅是可观测性的体现，更是对用户信任的承诺——让他们知道应用正在诚实工作。
