# 离线优先的轻量级任务管理应用：不只是“断网能用”

## 为什么离线优先值得认真对待

很多团队做离线功能，出发点只是“用户断网时别崩”。但实际上，离线优先设计的价值远不止于此。当你告诉用户“你的数据在本地已保存，网络恢复后会自动同步”，这传递的是一种信任承诺——用户不再担心点击按钮后数据丢失。同时，这意味着你需要维护一个比在线应用更严格的数据一致性模型：本地与服务器之间的状态必须收敛到一致，否则用户会看到“刚才删除的任务又出现了”或“同一任务出现两个版本”。

问题的核心在于：用户离线时的操作是“事实”，服务器在线时的状态也是“事实”，冲突不可避免。离线优先不是回避冲突，而是设计可预测、可恢复的冲突处理机制。

## 三块技术砖石的职责分工

对应到浏览器环境，三个核心组件各司其职：

**IndexedDB** 是本地数据持久层。它存储任务记录、元数据、同步状态。选择 IndexedDB 而非 localStorage，是因为任务数据可能结构化且需要索引（如按创建时间、完成状态查询），而且 IndexedDB 支持事务，能保证批量操作的原子性——这是数据一致性的基础。

**Service Worker** 是网络代理层。它拦截 fetch 请求，缓存 API 响应，并在离线时提供“虚假的在线体验”。但要注意：Service Worker 不应承担业务逻辑，它只负责资源层面的缓存与请求代理。离线时用户创建的任务，Service Worker 会优雅地让请求“静默失败”，交由后台同步机制处理。

**后台同步与重试队列** 是状态协调层。这是真正的“离线优先”引擎：当网络恢复时，按时间顺序或优先级重新提交离线期间堆积的操作。重试队列需要记录操作类型、目标资源、请求体、时间戳，并实现指数退避。如果某个操作反复失败，需标记为“失败”状态，触发人工介入。

## 两类典型冲突场景

**场景一：同一任务在多端被编辑**

用户在手机端离线时将任务标题改为“买牛奶”，在电脑端在线时将同一任务标题改为“买酸奶”。两端同步后，谁赢？通常有三种策略：最后写入胜利（简单但可能不合理）、基于版本号（谁先到达服务器谁生效）、差异合并（适用于文本字段但实现复杂）。对于任务管理这类轻量场景，“最后写入胜利”加“冲突标记”是务实之选——保留胜出版本，将败出版本作为“冲突副本”附在记录中，由用户确认。

**场景二：离线删除与在线更新交叉**

用户离线删除了任务A，同时在线端有人更新了任务A的描述。同步时，删除请求与更新请求可能交错到达。如果删除先到达，更新操作应返回“资源不存在”并标记为失败。如果更新先到达，删除操作应生效。这要求每条操作携带资源版本号，服务器按版本顺序处理，而不是单纯按时间排序。更棘手的情况是：用户离线删除了一个任务，但服务器上该任务已经被其他人引用了——这时需要软删除（标记 deletedAt 字段）而非硬删除，以便冲突时能回滚。

## 简化数据流：从写入到恢复

1. **本地写入**：用户点击“创建任务”，数据先写入 IndexedDB，状态标记为“待同步”，UI 立刻更新（乐观更新）。此时用户感知不到延迟。

2. **排队**：同时，一条操作记录（如 `{type: 'create', resourceId: 'uuid', data: {...}, timestamp: t, retryCount: 0}` ）被推入本地重试队列。

3. **同步**：Service Worker 监测到网络恢复，触发后台同步事件。同步器从队列中取出操作记录，逐一发送 HTTP 请求到 API。

4. **确认**：服务端成功处理后返回确认。同步器将 IndexedDB 中的记录状态改为“已同步”，并从队列中移除该操作。

5. **失败回滚或重试**：如果请求失败（如网络闪断、服务端冲突），根据错误类型处理——网络错误重试（指数退避），数据冲突则标记该记录为“冲突”，保留本地和服务器两版数据供用户选择。若超过最大重试次数（如10次），标记为“同步失败”并触发告警。

## 可观测性：别让“离线优先”成为黑箱

离线功能一旦出现异常，用户不会像在线应用那样刷新即可恢复。因此，可观测性是保障用户信任的底线：

**同步失败率**：在 IndexedDB 中为每条操作记录增加 `syncStatus` 字段（pending / success / failed / conflict）。定期统计 failed 和 conflict 的占比。如果超过阈值（如5%），说明冲突处理或重试逻辑有问题。

**冲突率**：记录每个任务发生冲突的次数、解决耗时、用户手工合并的占比。高冲突率可能意味着“最后写入胜利”策略不适合你的用户群体，需要考虑更精细的冲突解决 UI。

**平均恢复时间**：从用户第一条离线操作到该操作状态变为“已同步”的时间差。MTTR 过高说明重试队列或后台同步触发机制需优化。

这些指标可以通过本地日志上传至监控系统（如 Sentry 或自定义埋点），但不适合实时查询——离线应用的数据上报本身也需遵循离线优先原则，可以设计批量上报队列。

## 三条落地建议

1. **从“支持离线查看”开始，逐步过渡到“支持离线写入”**。先让用户断网时能看到缓存的任务列表，再开放创建和编辑。每一步都要评估数据冲突概率和用户预期管理。

2. **冲突解决默认采用“时间线式”而非“一刀切”**。不要自动丢弃任意一方数据。即使采用“最后写入胜利”，也要保留失败版本的历史记录，允许用户回看和手动恢复。

3. **为同步失败设计可见的 UI 提示**。在任务卡片上显示“同步中”“同步失败”“有冲突”等状态标签，并提供“点击查看冲突详情”的入口。用户需要知道自己的操作是否安全落地，这是离线优先建立信任的关键。
