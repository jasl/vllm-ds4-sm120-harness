# 离线优先的任务管理：当应用学会了“等待”

对于习惯了“联网即正义”的前端工程师来说，“离线优先”往往被简单地理解为缓存静态资源和 Service Worker 的魔法——断网时用户还能看到界面，好像就赢了。但真正交付一个离线优先的任务管理应用，你会发现技术只是冰山一角，水面之下的用户信任、数据一致性、冲突处理才是真正考验架构设计能力的地方。

## 为什么离线优先不只是“断网也能用”

大多数用户对“离线”的认知是“没网了，应用瘫痪”。但离线优先的真正目的是：无论网络如何，应用都像在线时一样可靠。这意味着用户写下一个待办事项时，这个操作必须是实时的、不可逆的——即使手机正处于飞行模式。

这种对用户承诺的破坏，会直接击穿信任。试想：用户离线创建了三个任务、标记了两个已完成，结果联网后应用告诉他“对不起，这些操作被覆盖了”。用户不会认为是网络问题，只会认为“这个软件不靠谱”。因此，离线优先本质上是一个关于承诺的问题：应用承诺用户的每一次操作都是有效输入，而非“等网络好了再说”的草稿。

数据一致性在这里变得微妙。在线应用依赖数据库事务保证 ACID，而离线优先应用必须将“一致性”拆解为“最终一致性”，并且在冲突发生时有明确的裁决规则——不是技术规则，而是用户可理解的业务规则。

## 三个核心角色的分工

要实现上述目标，浏览器端通常需要三个组件各司其职：

**IndexedDB** 是离线存储的根基。它不像 localStorage 那样只存字符串，而是支持结构化数据、索引和事务。对任务管理来说，每个任务是一个对象，可以按时间、状态、标签建立索引。更重要的是，IndexedDB 支持读写事务，这为本地数据完整性提供了基础。

**Service Worker** 是网络请求的仲裁者。它拦截 fetch 事件，决定请求是走网络还是从缓存响应。对离线优先应用来说，Service Worker 的角色不再是“先缓存再网络”的被动加速，而是“优先写入本地、再异步同步”的调度枢纽。它需要知晓哪些请求是“写操作”、哪些是“读操作”，并为写操作生成唯一的本地变更 ID。

**后台同步和重试队列** 负责从离线到在线的桥梁。当网络恢复时，Service Worker 可以触发 SyncEvent，但更可靠的做法是维护一个持久化的重试队列存储在 IndexedDB 中，记录每条待同步的变更。队列中的每条记录包含：变更类型、数据载荷、本地时间戳、重试次数。重试需要指数退避，并且要对重复提交做幂等防护——用户离线点“完成”十次，在线后只能生效一次。

## 冲突场景：不是理论问题，是日常事故

场景一：同一任务在多端被编辑。用户在手机上把“买牛奶”的优先级改为“高”，同时在笔记本上把描述改为“买脱脂牛奶”。当两台设备都联网同步时，冲突发生了。解决这类冲突的常见策略是“最后写入者胜出”，但如果两个操作都是用户在不同设备上同时按下的，简单的时间戳仲裁并不公平。更优的做法是字段级合并：优先级取手机端、描述取笔记本端，或者标记为“冲突”让用户手动解决。

场景二：离线删除与在线更新交叉。用户离线删除了任务 A，而同一时刻，另一位协作成员在线更新了任务 A 的截止日期。当本地同步时，服务端看到一条删除指令和一条更新指令，顺序决定结果。如果删除先到达，更新会被丢弃；如果更新先到达，删除会使后续所有操作失效。这里需要引入“墓碑机制”——删除不真正移除记录，而是标记为已删除，并保留其版本号，以便检测这种交叉冲突。

## 简化数据流：一次离线操作的完整旅程

当一个任务在离线状态下被创建时，数据流是这样的：

1. **本地写入**：任务直接写入 IndexedDB，状态为“本地”，并带上一个 UUID 作为本地 ID。
2. **排队**：一个变更记录被追加到重试队列，包含操作类型（create）、本地 ID、数据体、时间戳。
3. **同步**：当网络可用时，后台同步任务将队列中的记录逐个发送到服务端。每发送一条，记录进入“同步中”状态。
4. **确认**：服务端返回成功响应，并附上服务端生成的唯一 ID。本地将任务 ID 更新为服务端 ID，状态改为“已同步”，并移除队列记录。
5. **失败回滚或重试**：如果服务端返回 409（冲突）或 500（服务端错误），记录保留在队列中并增加重试次数。当重试次数超过阈值时，标记为“同步失败”，并由用户决定是手动解决冲突还是丢弃变更。

## 可观测性：看得见的才是可管理的

离线优先应用最怕的就是“用户坏掉了，但日志是干净的”。你需要在客户端和服务端都埋点：

- **同步失败率**：客户端上报每个变更的同步失败次数和原因（网络超时、服务端冲突、未知错误）。如果某个版本的客户端同步失败率突然升高，说明有兼容性问题。
- **冲突率**：服务端统计每 1000 次写操作中发生冲突的次数。冲突率不应超过 1%，否则说明离线窗口太长或冲突解决策略有问题。
- **平均恢复时间（MTTR）**：从变更进入队列到最终确认的平均时间。如果这个值超过 10 秒，说明后台同步的触发机制不够灵敏，或者重试策略过于保守。

客户端可以提供一个“同步状态”面板，让用户看到当前是否有未同步的变更、最近一次同步时间、失败的变更列表。这不仅增加了透明度，也是用户信任的最后一道防线。

## 落地建议

1. **从“只支持离线创建”开始，不要一步到位。** 离线修改和离线删除会引入大量冲突场景，先让用户离线创建任务、在线同步，等冲突处理逻辑成熟后再开放全功能。
2. **用“乐观更新 + 后台验证”替代“等网络确认再更新 UI”。** 用户按下按钮立即更新本地视图，异步同步失败时再回滚或突出提示。这是用户体验的底线。
3. **不要自己造冲突解决协议，复用已有的 CRDT 或 OT 思想。** 对自己来说，实现一个字段级的冲突合并器就够了；对团队来说，研究一下 CRDT 的“无冲突复制数据类型”可以省去未来三年的修 bug 时间。

离线优先不是锦上添花的 feature，而是让用户在任何环境下都能信任你的应用的基石。它教会我们一件事：优秀的应用不仅知道如何响应，更知道如何等待。
