# 离线优先任务管理：从“断网能用”到“数据信任”的设计之道

作为一名经历过多次网络抖动和数据丢失的前端工程师，我越来越意识到“离线优先”绝不是一个技术噱头，而是一种对用户承诺的实现——让应用在任何网络条件下都能可靠运行。本文将以一个轻量级任务管理 Web 应用为例，梳理离线优先设计的核心逻辑与实战要点。

## 一、离线优先的真正价值：信任比功能更重要

很多团队对离线能力的理解止步于“断网时能看缓存数据”。但用户需要的远不止这些：当他们在高铁隧道里修改了一个任务截止日期，三小时后联网却发现数据被覆盖，这种体验会彻底摧毁用户对应用的信任。

离线优先的本质，是**建立一套用户对数据状态的可预期机制**。用户需要明确知道：我的改动是否被记录？同步后会不会丢失？冲突发生时，系统站在谁那边？这种“数据确定性”才是离线应用的核心竞争力。

## 二、三大技术组件的职责划分

设计离线优先系统，关键在于让 IndexedDB、Service Worker 和后台同步机制各司其职，形成闭环。

### 1. IndexedDB：本地的“真相源”
它不是缓存，而是用户当前可见数据的权威存储。所有用户操作必须先写入 IndexedDB，并标记同步状态（pending/synced/failed）。这意味着即使用户强制刷新页面，改动也不会丢失。

### 2. Service Worker：网络代理与离线网关
Service Worker 拦截所有 fetch 请求，根据网络情况决定返回缓存还是发起网络请求。更重要的是，**它不负责处理业务逻辑**——只做请求应答的调度。当检测到网络恢复时，会触发一个“同步事件”通知主线程。

### 3. 后台同步/重试队列：有序的数据管道
这是整个系统的“定时炸弹”。所有待同步的操作（创建→更新→删除）按时间顺序进入一个队列，由同步管理器逐个处理。每个操作都包含：操作类型、数据版本号、时间戳。失败时根据错误类型决定重试（网络错误）还是打标等待人工介入（服务端拒绝）。

## 三、两类典型的冲突场景

### 场景A：多端编辑同一任务
用户 A 在手机端将任务标题改为“准备周报”，用户 B 在电脑端同时将其改为“编写季度报告”。联网同步时，两条更新记录时间戳接近。

合理策略是 **“最后写入者胜出” + 版本号校验**。但如果两者时间差小于 1 秒，则应触发冲突标记，将两个版本同时展示给用户手动合并。这种“宁可麻烦用户，不可丢失数据”的态度，正是信任的来源。

### 场景B：离线删除与在线更新交叉
用户离线时删除了任务 X，但服务器上总共有三次更新记录。联网后，同步队列里的“删除”操作可能与新收到的“更新”记录形成矛盾。

我的处理方式是：**将删除操作转化为“软删除”标记**（isDeleted: true），而非直接移除数据。同步时如果发现已删除的任务在服务端有更新，则标记冲突并保留删除决定，但为用户展示“有个任务已被删除，但它有最新版本”的提示。

## 四、简化数据流：从本地到云端

```
用户操作 → 1. 写入 IndexedDB（标记pending）
          → 2. 加入同步队列（包含版本号）
网络可用 → 3. 队列逐个发送（含If-Unmodified-Since头）
          → 4. 服务器校验版本号
              ├─ 成功：返回确认ID → 本地标记synced
              └─ 冲突/失败：返回错误码
                  ├─ 网络错误 → 保留队列，指数退避重试
                  └─ 版本冲突 → 标记冲突，通知用户介入
```

关键细节：队列应支持**幂等重放**——同一个操作发送两次，服务器能识别重复并返回相同结果。这依赖于操作 ID 的唯一性。

## 五、可观测性：没有度量，就没有改进

很多团队只关注功能开发，却忽略了离线系统的“黑盒问题”。以下几个指标必须埋点：

- **同步失败率**：按操作类型（create/update/delete）统计，超过阈值自动告警
- **冲突率**：多端冲突在所有同步操作中的占比，用于判断是否需优化冲突策略
- **平均同步恢复时间（MSRT）**：从队列开始同步到全部完成的时间，衡量系统健壮性
- **队列深度**：待同步操作的数量，短时间内激增可能暗含批量写入问题

建议在本地存储一个**同步日志表**，记录每个操作的时间、状态、错误码、恢复方式。用户遇到数据问题时，可以直接导出这个日志作为调试依据。

## 六、落地建议

1. **先做好离线写，再考虑离线读**：用户最不能接受的是“写了却没了”。优先确保离线写入不丢失，读缓存可以后续迭代优化。

2. **为每个数据记录添加“版本向量”**：简单的数字版本号在单端场景下够用，多端协作推荐使用 Lamport 时钟或向量时钟，这样冲突检测更精确。

3. **预留“数据救援”入口**：设计一个“手动同步”按钮，并在设置中提供“导出本地数据”功能。当自动同步出现死循环时，这是最后的安全网。

离线优先不是为工程师的自嗨，而是为用户在不确定性中提供确定性。当你的用户在无信号的地下车库里还能安心修改任务、信任数据不会丢失时，这套系统的价值就真正落地了。
