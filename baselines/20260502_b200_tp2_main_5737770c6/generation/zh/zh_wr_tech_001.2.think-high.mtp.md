# 离线优先任务管理：当“断网”不再是意外，而是常态

## 一、离线优先的真正意义：从功能补丁到用户信任

很多团队在开发离线功能时，习惯性地将其视为“网络不好时的兜底方案”——缓存一些数据，断网时能读，联网后同步。但这样的设计往往忽略了最关键的一点：**离线优先不是为了让应用在断网时“可用”，而是让用户在任意网络状态下都能获得一致的、可预期的体验。**

用户信任的建立，来源于两个基本事实：其一，我离线时创建或修改的任务，不会因为网络恢复而丢失；其二，如果发生冲突，应用能给出清晰的解释，而不是无声地覆盖我的操作。数据一致性在这里不是技术债，而是产品承诺。

## 二、核心组件的职责划分

一个健壮的离线优先架构，通常由三个核心组件协同工作，各自承担不可替代的职责。

**IndexedDB** 是本地数据库，负责持久化存储。它的角色不是“缓存层”，而是**应用的主数据源**。用户的所有操作首先写入 IndexedDB，再考虑同步到远端。设计时需要注意：IndexedDB 的存储结构应与服务端的 schema 尽量对齐，但允许记录本地状态字段（如 `syncStatus: 'pending' | 'synced' | 'conflict'`）。

**Service Worker** 承担网络代理和后台同步调度的职责。它拦截 fetch 请求，决定是返回本地缓存数据，还是转发到网络。更重要的是，Service Worker 可以注册 `sync` 事件——当网络从离线变为在线时，浏览器会触发该事件，允许你在后台执行同步逻辑，而无需用户刷新页面。

**后台同步/重试队列** 是一个轻量级的任务队列，通常基于 IndexedDB 或内存实现。它记录所有待同步的操作（如“创建任务A”“更新任务B的截止日期”），并按照先进先出的顺序执行。每次同步失败时，队列需要决定：是立即重试（指数退避），还是标记为失败并等待用户干预。

## 三、冲突场景：不止是“谁先谁后”

冲突不是边缘情况，而是离线优先架构的核心挑战。以下是两类典型的冲突场景及其处理思路。

**场景一：同一任务在多端被编辑**

用户A在离线状态下修改了任务“采购清单”的描述为“买鸡蛋和牛奶”，用户B同时在另一设备上将同一个任务的状态改为“已完成”。当两者恢复网络并同步时，服务端发现同一个 `taskId` 的 `description` 和 `status` 字段产生了分歧。

解决这类冲突，常见的策略是“基于字段级最后写入获胜（Last Write Wins）”，但这样会丢失用户A的修改。更合理的方式是**“操作转换”或“CRDT（无冲突复制数据类型）”**。对于任务管理这类数据结构简单的场景，一个实用的方案是：将每个字段的更新记录为独立的时间戳，合并时保留最近修改的值；如果两个字段互不冲突，则完全合并。若无法合并（如同时修改了相同字段），则通知用户手动解决。

**场景二：离线删除与在线更新交叉**

用户离线时删除了任务“周末聚会筹备”，而另一个用户在线将这个任务的参与者列表更新了。当删除操作同步到服务端时，服务端需要判断：如果没有其他修改，直接删除该任务；但如果任务在其他端已被更新，就产生冲突。

一种处理方式是“墓碑标记”：删除操作不真正删除记录，而是将任务标记为 `deleted: true`，同时保留其他修改。服务端合并时，检查是否有其他非删除操作发生在删除时间之后——若有，则拒绝删除并回滚，向执行删除的设备返回冲突，提示用户“该任务已被更新，是否仍要删除？”。

## 四、简化的数据流：本地优先，同步为辅

一个典型的离线优先数据流可以概括为五个阶段：

1. **本地写入**：用户执行操作（创建、更新、删除），应用立即将数据写入 IndexedDB，状态标记为 `pending`。此时用户看到的是即时响应，无网络等待。
2. **排队**：操作被序列化为一个同步任务（包含操作类型、数据、时间戳等），推入同步队列。队列本身也存储在 IndexedDB 中，确保应用被关闭后不丢失。
3. **同步**：当 Service Worker 检测到网络恢复（或定时触发 `sync` 事件），从队列中取出任务，按序发送到服务端 API。此处需处理幂等性——同一操作应能被安全重放。
4. **确认**：服务端处理请求后返回确认（如新的 `updatedAt` 时间戳）。本地将对应记录的状态改为 `synced`，并从队列中移除该任务。
5. **失败回滚或重试**：若同步失败（网络中断、服务端校验失败、冲突发生），将任务标记为 `failed`，并记录失败原因。简单失败（如网络波动）使用指数退避自动重试；冲突或校验失败需保留在队列中，等待用户手动处理。

## 五、可观测性：量化离线系统的健康度

离线优先系统的故障往往是静默的——用户可能在一周后才发现某个任务没同步成功。因此，可观测性是保障用户信任的最后一道防线。

需要记录的核心指标包括：
- **同步失败率**：`失败任务数 / 总同步任务数`。若超过某个阈值（如5%），应触发告警。
- **冲突率**：`冲突次数 / 总同步成功次数`。冲突率高通常意味着数据模型的设计有缺陷，或冲突解决策略过于简单。
- **平均恢复时间（MTTR）**：从同步失败到最终成功（或手动解决）的平均时间。这个指标反映了系统处理异常的能力，也暗示了用户是否被长时间“卡”在冲突状态。

日志方面，每个同步任务都应保留完整的生命周期：入队时间、开始同步时间、结束时间、失败原因（如 HTTP 状态码、服务端错误消息）。建议将这些信息输出到应用的日志服务中，便于排查问题。

## 六、三条项目落地建议

1. **先设计数据模型，再写代码。** 离线优先的数据模型需要包含本地状态字段（syncStatus、localTimestamp、conflictData），同时与服务端约定好冲突检测的锚点（如基于 `updatedAt` 或版本号）。数据模型不稳固，后续所有同步逻辑都会变成打补丁。

2. **从“单用户单设备”场景开始，逐步扩展。** 如果团队是第一次实践离线优先，不要一开始就支持多端实时协作。先确保单个用户在离线/在线切换时数据不丢失，再增加冲突检测和合并逻辑。复杂度要逐步引入。

3. **让用户拥有最终决定权。** 自动冲突解决策略再聪明，也无法覆盖所有业务场景。应该提供一个“冲突面板”，清晰展示哪两个操作产生了冲突，并允许用户选择保留哪个版本。用户不是来帮你 debug 的，但你需要给他们控制权。
