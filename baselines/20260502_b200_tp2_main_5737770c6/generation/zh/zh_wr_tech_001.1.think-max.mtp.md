# 离线优先任务管理应用：从“断网可用”到用户信任

## 为什么离线优先不只是“断网也能用”

多数团队对“离线优先”的第一反应是“加个 Service Worker 缓存页面就行了”。这当然能提升加载速度，但真正的离线优先应用要解决的核心问题是：**当网络不可靠时，用户如何信任自己做出的每一个操作都被保留，并且最终与云端一致。**

以一个任务管理应用为例。用户在地铁上标记了一个任务完成，又新建了一个子任务，还调整了优先级。如果应用只是把数据写进本地存储，等网络恢复后回传云端，这看起来很简单。但实际场景中，用户可能同时打开两个标签页，也可能在手机和电脑上交替操作。当两端都在离线状态下修改了同一个任务，云端就会收到两个都有合法时间戳的版本——这时，用户信任就会崩塌。

离线优先的本质，不是“断网能看”，而是“断网能用，且用完不丢、不冲突”。这需要开发者从架构层面处理数据一致性，而不只是加一个离线缓存层。

## 三大支柱的职责分工

设计这样一套系统，三个技术组件缺一不可，各有明确边界。

**IndexedDB** 承担本地数据库的角色。它存储任务实体的完整数据、本地变更操作的日志，以及同步状态的元数据。关键在于，它不仅存最新数据，还要存“变更操作序列”——这是后面回滚和冲突解决的依据。相比 localStorage，IndexedDB 支持事务、索引和大量结构化数据，更适合记录任务的一步步变化。

**Service Worker** 充当网络请求的拦截器和离线时的代理。它的核心职责是两件：一是缓存静态资源让页面能加载，二是拦截发往 API 的请求。当网络不可用时，SW 不直接转发请求，而是把请求信息存入 IndexedDB 的一个“待同步队列”。这样，前端组件不需要自己判断网络状态、手动管理重试——这些交给 SW 统一调度。

**后台同步与重试队列** 负责把本地的待同步操作，按顺序推给云端，并在失败时自动重试。这里的“顺序”很关键：如果用户先新建了分组A，又在里面新建了任务B，那么同步时必须先发A再发B，否则云端会报错。重试队列要实现指数退避，而不是每分钟疯狂重试。同时要区分“网络暂时不通”和“数据校验失败”——前者可以重试，后者需要通知用户手动解决。

## 冲突场景：构造逻辑，而非逃避

冲突是离线优先中最棘手的问题，这里举两个典型案例。

**场景一：同名同父任务，两端同时创建。** 用户在手机端新建“准备周报”任务，归属“工作”分组；同事在电脑端也新建了“准备周报”任务，归属同一个分组。两端都离线。网络恢复后，这两条记录都尝试同步到云端。如果云端简单用标题去重，可能误删一条；如果不处理，就会出现两条同名任务。解决方案是：云端为每一条记录生成全局唯一ID。同步时，本地ID和云端ID映射。遇到标题、分组都相同的情况，通过两端各自的任务创建时间戳和本地设备ID组合，形成“确定性排序”规则——比如按时间戳 + 设备号字典序决定保留哪个。

**场景二：离线删除 vs 在线更新。** 用户在离线状态下删除了任务A，但网络断开前，同步队列中还有一条“更新任务A截止时间”的操作尚未发出。网络恢复后，重试队列先发送了“更新截止时间”的操作（因为它排在前面），云端成功接受了这条更新。接着本地又发送“删除任务A”，云端发现A存在，但它的版本号已经变了——这种情况下，简单的覆盖会导致那笔更新丢失。合理的做法是：云端记录每条数据的历史版本号。删除操作必须携带该数据被删除前的版本号。如果版本号对不上，说明有更新发生在删除之前，云端应该返回冲突，让本地用户选择“保留更新并取消删除”还是“强制删除，丢弃那份更新”。

## 简化数据流：写本地、排队、同步、确认、回滚

用户操作一条任务时，流程如下：

1. **本地写入**：操作直接写入 IndexedDB，用户界面立即反映变更。此时显示一个“同步中”或“等待联网”的标记。

2. **排队**：同时，一个描述该操作的日志（包含操作类型、实体ID、操作前的数据快照、操作后的数据快照、时间戳）被写入“待同步队列”。这个队列是 FIFO 的。

3. **同步**：Service Worker 监测到网络可用，或通过后台同步 API 被唤醒，从队列头部取出日志，逐个向云端 API 发送请求。

4. **确认**：云端成功处理请求后，返回确认消息，包含云端生成的最新版本号。本地 IndexedDB 更新该任务的版本号，并标记“已同步”。队列中对应的日志被移除。

5. **失败回滚或重试**：如果云端返回网络错误，重试队列调度下一次尝试。如果云端返回冲突（版本号不一致），则本地根据冲突策略决定：是回退到云端版本并通知用户，还是强制推送本地版本。如果决定回滚，则从日志中取出操作前的数据快照，恢复该任务在 IndexedDB 中的状态。用户会看到任务回到了操作前的样子，并收到一条“同步冲突，已还原”的提示。

## 可观测性：从黑盒到可控

用户不会在意你的技术细节，但你需要知道系统是否健康。至少需要监控四个指标：

**同步失败率**：按操作类型（新建、更新、删除）统计。如果某类操作失败率异常，说明云端 API 接口逻辑有问题，或前端构造的数据格式有误。

**冲突率**：以“每千次操作”为单位。冲突率高，说明多端并行编辑的场景频繁发生，可能需要引入乐观锁或提前在UI上提示“其他人正在编辑”。

**平均恢复时间**：从用户操作到云端确认的时间。这是最重要的用户体验指标。如果时间过长，说明重试策略过保守，或 API 本身慢。需要区分“网络延迟导致的时间”和“冲突导致的手动解决时间”。

**未同步操作积压数**：队列长度。如果长期积压超过几百条，可能说明同步机制存在死锁，或者用户操作速度远超后台同步能力。

这些指标可以通过 IndexedDB 中“同步日志”表的统计实现：每条日志带时间戳、操作类型、最终状态（成功/失败/冲突）、重试次数。定期汇总后上传到后端用于大屏展示或告警。

## 三条落地建议

1. **从“单设备离线”起步，不要一上来就做多端协同**。先确保单个用户断网后能正常操作，网络恢复后数据不丢。等稳定后，再引入跨设备冲突处理。跨设备同步是另一个复杂度量级。

2. **为每一个操作保留“操作前后快照”，而不是只存最新状态**。这是冲突回滚的底线。如果只存最新值，一旦冲突发生，你无法知道用户之前删了什么、改了哪个字段。

3. **设计清晰的可观测性，尽早暴露问题**。不要把同步流程做成“黑盒，用户反馈才排查”。从第一天起，在开发者面板或运维界面展示同步队列长度、冲突次数、重试间隔。否则等线上用户抱怨“我的任务不见了”，你将无从溯源。
