# 离线优先的任务管理应用：从“断网可用”到“用户信任”

当我们在讨论“离线优先”时，最容易陷入的误区是把它等同于“断网时能打开页面”。真正的离线优先，意味着用户面对的不是一个降级体验的替代品，而是一个在任何网络条件下都能保持完整功能、并在网络恢复后优雅同步的真实系统。尤其是对于任务管理这类高频交互的应用，用户输入的每一个“已完成”或“明天截止”都承载着实际的工作承诺——数据丢失或冲突带来的不仅是技术故障，更是信任崩塌。

## 离线优先背后的信任契约

传统在线应用假定网络永远是好的，数据以服务器为唯一权威来源。离线优先则反转了这个假设：用户的本地设备被视为第一现场，服务器是协调者而非主宰者。这种设计传递了一个信号——“你的操作不会因为网络波动而丢失”。但信任的建立并不容易：当用户离线修改了一个任务，然后在线时看到另一个设备上同样的任务已被同事调整，系统如何裁决？这不仅是算法问题，更是对用户预期的管理。

## 三大技术支柱的分工

实现可靠的离线优先，需要三类技术的协作，它们各自解决不同层面的问题，缺一不可：

**IndexedDB** 承担本地持久化职责。它不同于 localStorage 的键值对限制，能存储复杂的结构化数据，支持索引和事务。在任务管理中，每个任务实体、版本号、本地修改时间戳都需要以可靠的 schema 存入 IndexedDB。它是离线操作的“主场”，所有 CRUD 都在这个本地数据库上执行，保证用户操作零等待。

**Service Worker** 作为网络代理层，拦截应用发出的请求。它最重要的能力是“编程式缓存”：不仅缓存静态资源，还可以策略性地缓存 API 响应。当用户在线时，Service Worker 可以配置为“网络优先，失败后回退到缓存”或“缓存优先，后台刷新”等策略。它的存在让应用在弱网环境下仍然能够加载数据，而不必等待网络超时。

**后台同步** 与 **重试队列** 解决的是写操作的一致性问题。IndexedDB 中的数据在离线时产生“变更记录”，这些记录不会立即发送到服务器，而是入队等待。后台同步（如 Background Sync API）确保即使浏览器关闭，这些待发送的变更也会在网络恢复后尝试执行。重试队列则需要加入指数退避策略——第一次失败后等 10 秒，第二次 30 秒，第三次 2 分钟，这样既能避免反复冲击服务器，也能在临时故障恢复后及时同步。

## 冲突场景：不可能避免，但可以优雅处理

冲突是分布式系统的常态。在任务管理应用中，至少有两类典型冲突需要明确策略：

**场景一：同一任务的多端编辑**。用户 A 在电脑上把任务标题改为“撰写周报”，用户 B 在手机上同时改成了“完成周报草稿”。当两个离线客户端先后恢复网络并尝试同步时，服务器需要判断谁更“权威”。一个可行的方案是“最后写入者获胜”结合时间戳比较，但更稳妥的方式是引入版本向量或操作转换（OT）——让每个客户端保留操作的上下文，服务器基于操作序列应用变更。对于轻量级应用，更现实的策略是标记冲突并让下次登录时由用户手动合并。

**场景二：离线删除与在线更新的交叉**。用户离线删除了一个任务，而同一时刻在线时，另一个协作者更新了该任务的状态为“已完成”。当离线客户端同步时，服务器收到删除指令但目标任务已被更新。直接删除会丢失更新记录。这里需要采用“墓碑”机制：删除操作本身也是一个带有时间戳的变更，服务器比较两个变更的先后顺序，如果是删除在先，则丢弃后来的更新请求；如果是更新在先，则需要向删除方返回冲突，询问用户是否确认删除。

## 简化的数据流：从写入到确认

以一个典型的“新建任务”操作为例，离线优先的数据流可以拆解为五个阶段：

1. **本地写入**：用户点击“保存”后，应用直接写入 IndexedDB，任务被标记为“未同步”状态，ID 使用本地生成的 UUID。界面立即更新，延迟为零。
2. **排队**：后台将本次操作（类型、数据、本地ID、时间戳）推送到同步队列（同样存在 IndexedDB 中，确保浏览器关闭也不丢失）。
3. **同步**：当网络可用时，后台同步事件触发，队列逐个取出变更，通过 Service Worker 发往服务器。服务器校验并应用变更，返回服务器生成的最终 ID 和系统时间戳。
4. **确认**：客户端接收成功后，更新本地记录的状态为“已同步”，并将本地 UUID 与服务器 ID 关联。
5. **失败回滚或重试**：如果同步失败（服务器 500、网络超时），变更保留在队列中，重试机制按时序增加间隔重试。若同一变更失败超过 N 次，则标记为“同步失败”，在界面上提示用户手动处理。极少数情况下，服务器判定冲突不可自动解决，则返回冲突标记，客户端将该任务置入“有冲突”状态，等待用户介入。

## 可观测性：让系统的瓶颈透明化

离线优先系统最难维护的部分在于“看不见的状态”——那些静静躺在本地数据库中的未同步变更，或者同步队列中反复重试的请求。如果不加以观测，用户可能某天发现不同设备上的任务不一致，而开发者毫无头绪。关键指标包括：

- **同步失败率**：分设备、分操作类型的失败比例。如果删除操作的失败率高，可能是冲突处理策略有问题。
- **冲突发生率**：每千次操作中因版本冲突需要用户手动解决的次数。冲突率上升往往意味着团队协作模式在变化。
- **平均恢复时间（MTTR）**：从用户恢复网络到待同步队列完全清空所花的平均时间。如果 MTTR 超过数分钟，需要检查同步接口性能或重试策略。

这些数据可以来自客户端埋点：每次同步结束时上报耗时、结果状态。服务器端日志统计冲突类型。在 Dashboard 上联合分析，能够快速定位是网络问题、代码 bug 还是用户行为变化。

## 落地建议：从环境搭建到运维

1. **先定义冲突策略再写代码**。不要等技术细节完成后才考虑冲突。在原型阶段就应该明确“最后写入者获胜”“用户手动合并”还是“按操作语义合并”三种策略的适用范围，并在 PRD 中注明每种 CRUD 操作的冲突处理方案。

2. **在测试中覆盖“网络劣化”场景**。使用 Chrome DevTools 的 Network Throttling、MacOS 的网络调节工具，模拟从 3G 到完全断网的多种情况。测试不仅要覆盖在线→离线→在线的链路，还要测试多个设备同时离线、不同顺序恢复网络这种复杂场景。

3. **把同步状态暴露给用户**。不要隐瞒“系统正在后台同步”的事实。在 UI 上显示同步进度条或待同步条目数，让用户知道操作是否已经被确认。许多冲突之所以变成用户投诉，只是因为用户没看到“还有 5 条变更正在等待同步”的提示。

离线优先不是一种技术噱头，而是一种对用户时间的尊重。当用户在地铁隧道里修改了明天要交给客户的报告截止日期，他应该得到的是信任，而不是“操作失败，请重试”。实现这一点，需要的是对数据流每一个环节的敬畏，以及将可观测性作为系统基础设施一部分的自觉。
