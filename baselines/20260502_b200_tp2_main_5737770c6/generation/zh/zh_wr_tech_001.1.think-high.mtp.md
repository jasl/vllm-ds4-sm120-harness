# 离线优先的轻量级任务管理：不只是断网，更是信任工程

作为一个已经写过不少 CRUD 页面的前端工程师，你可能早就习惯了“网络通畅”这个隐形假设。但一旦你尝试构建一个真正的离线优先应用，你会发现，“断网也能用”只是冰山一角。离线优先的真正难点，在于如何让用户在无网络、弱网络、多设备之间，依然相信自己的数据是安全的、一致的、不会丢的。这本质上是一个信任工程问题。

## 离线优先的核心挑战：信任与一致性

很多团队一开始把离线优先理解为“本地缓存 + 定时同步”，但这会埋下很多坑。用户最怕的不是断网，而是数据“静默丢失”或“莫名其妙被覆盖”。比如，用户在地铁上标记了一个任务为“已完成”，回到办公室打开电脑，发现那个任务又变回了“待处理”——这不是功能缺陷，这是信任崩溃。

离线优先的设计目标，不是让用户感觉不到断网，而是让用户在任何网络状态下，都能对自己数据的最终一致性有信心。这就要求我们在架构层面显式地处理网络边界，而不是把它当成异常情况。

## 三大基础设施分工

实现离线优先，有三个基础设施缺一不可，但它们的职责边界经常被混淆：

**IndexedDB** 负责本地持久化。它是你的“本地数据库”，而不是缓存。任务数据、操作日志、冲突记录都存这里。它的核心价值是：即使浏览器崩溃、用户关闭标签页，数据也不会丢。比 localStorage 更大的容量、更丰富的数据结构、支持索引查询，让它成为主存储的首选。

**Service Worker** 负责网络代理与离线拦截。它运行在独立线程，可以拦截页面发出的所有 fetch 请求。离线时，它充当“假服务器”，从本地缓存返回资源。但注意：很多人的误区是让 Service Worker 直接操作 IndexedDB——这会绕过后台同步机制，导致数据流混乱。Service Worker 的最佳职责是：管理静态资源和 API 请求的离线缓存策略，但不直接处理业务数据的读写。

**后台同步与重试队列** 负责网桥。后台同步（Background Sync API）允许你在网络恢复时自动触发同步逻辑。重试队列则是一个更通用的实现方案：每次本地写入时，同时将操作记录（操作类型、时间戳、数据快照）写入一个专门的重试表。同步过程就是消费这个队列：按顺序发送请求，成功则删除记录，失败则保留并标记重试次数和最后一次失败原因。

## 冲突场景：无法回避的现实

即使有再好的同步机制，冲突也必然发生。两个典型的场景：

**场景一：同任务多端编辑**。用户在手机端将任务标题改为“买牛奶”，同时在电脑端将这个任务标题改为“买豆奶”。如果两台设备都离线，后续同步时就会产生冲突。简单的“后写覆盖”策略会导致至少一方的意愿被丢弃。更合理的方式是：基于时间戳 + 操作类型的合并策略，比如“编辑标题”与“编辑标题”冲突时，可以标记为冲突并让用户裁决，或者采用 CRDT（无冲突复制数据类型）方案自动合并。

**场景二：离线删除与在线更新交叉**。用户离线时删除了一个任务，但同一任务在线上被另一个用户（或同一用户的其他设备）更新了备注。同步时，本地说“删除”，服务端说“更新”，这就产生了冲突。这里需要引入“墓碑标记”机制：删除操作不能真的删掉本地记录，而是标记为“已删除”并保留一个墓碑记录。同步时，服务端要比较操作的先后顺序，如果是删除时间晚于更新时间，则执行删除并返回确认；否则，服务端可以拒绝删除并告知用户“该任务已被修改”。

## 简化数据流：从本地写入到最终确认

一个典型的离线优先数据流可以抽象为五个阶段：

1. **本地写入**：用户点击“新建任务”，立即写入 IndexedDB，并生成一个本地唯一 ID（UUID），同时将操作写入重试队列（状态为“待同步”）。
2. **排队**：页面或 Service Worker 检测到网络可用，开始消费重试队列，按顺序发送请求。
3. **同步**：请求到服务端后，服务端根据本地 ID 和时间戳判断是否冲突。若无冲突，执行操作并返回同步确认。
4. **确认**：收到服务端确认后，更新本地记录的状态为“已同步”，并从重试队列中移除该操作。
5. **失败回滚或重试**：若请求失败（网络超时、服务端返回 409 冲突），则保留操作记录，标记上次失败原因和重试次数。超过最大重试次数（比如 5 次）后，标记为“失败”，通知用户人工处理。

## 可观测性：同步健康度的度量

离线优先系统最怕的是“不知道哪里出了问题”。如果没有可观测性，你很难判断是同步阻塞、冲突率过高，还是某个用户的数据一直没同步成功。

至少需要关注的三个指标：

- **同步失败率**：单位时间内重试队列中标记为失败的记录占总同步请求的比例。突然升高可能意味着服务端接口变化或网络状况恶化。
- **冲突率**：服务器返回 409 的比例。冲突率过高，说明你的冲突检测策略过于严格，或者用户的离线编辑时间窗口太长。可以考虑缩短离线编辑的过期时间。
- **平均恢复时间**：从用户提交操作到收到同步确认的平均耗时。这个指标反映了同步链路的速度和稳定性，也直接影响了用户的等待体验。

这些数据可以通过埋点上报到监控系统。每次同步操作结束时，上报一条包含操作类型、是否冲突、是否重试、耗时等信息的日志。这样你才能在问题发生前预判，而不是等用户投诉。

## 三条落地建议

如果你打算在下一个项目中尝试离线优先架构，这里有三条务实的建议：

**第一条：先做“本地优先”，再考虑“离线”**。很多人一上来就规划 Service Worker 和后台同步，结果连基本的本地存储设计都没做好。先把 IndexedDB 的数据模型设计清楚，确保本地 CRUD 功能完整且可脱离网络运行，再逐步加入同步逻辑。离线能力是增量，不是替换。

**第二条：为冲突策略预留可配置空间**。不要一开始就锁定一个冲突解决策略。业务不同，策略也不同。比如任务管理可能更适合“用户裁决”，而笔记应用可能更适合“自动合并”。设计时让冲突策略可插拔，方便后续调整。

**第三条：把“同步失败”作为一等公民来对待**。不要只在控制台打印错误日志。在 UI 层给用户明确的反馈：某个操作还在排队、某个操作同步失败需要重试、某个操作因为冲突需要手动处理。用户看到状态，才会相信系统没有静默丢失他的数据。

离线优先不是炫技，而是对用户数据负责的一种设计态度。当你不再假设网络可靠，你才会真正学会如何构建值得信赖的应用。
