# 离线优先的轻量级任务管理：从“能用”到“可信”

## 一、离线优先的真正代价

大多数工程师对离线优先的第一反应是“缓存数据，断网也能操作”。但深入设计过这类系统的开发者都知道，离线优先的核心不是数据本地化，而是**用户对系统的信任**。一个任务管理工具，如果用户辛辛苦苦整理了代办清单，却在网络恢复后出现了重复、丢失或覆盖，用户的信任就会崩塌。

信任的建立来自于两个维度：**数据一致性**（用户在不同设备、不同时间看到的任务状态应当一致）和**操作确定性**（用户的操作要么成功，要么明确告知失败原因）。离线优先的设计，本质上是在网络不确定的环境下，为用户提供一种“确定性”的幻觉。

## 二、三驾马车：IndexedDB、Service Worker、后台同步队列

离线优先体系有三个核心组件，职责必须清晰分离。

**IndexedDB** 承担本地数据仓库的角色。它不是缓存，而是“可信的本地副本”。任务管理应用的所有增删改操作，第一步永远是写入 IndexedDB，然后才是尝试同步到服务端。现实选择 IndexedDB 而非 LocalStorage，是因为需要支持结构化查询、事务和较大数据量。关键的设计细节是：每条任务记录都应携带一个本地递增的版本号或操作时间戳，用于后续冲突检测。

**Service Worker** 是网络请求的中介者。它的核心职责不是缓存静态资源（那是PWA的常规用法），而是拦截应用发往服务端的 API 请求，并在网络不可用时将请求“排队”。更准确地说，Service Worker 应该具备一个**请求持久化队列**——当 `fetch` 失败时，将请求体、URL、方法等信息存入 IndexedDB 的队列表中，并注册后台同步事件。

**后台同步/重试队列** 是离线优先的“心脏”。它运行在主线程之外，即使在用户关闭页面后仍能触发。当浏览器检测到网络恢复，会唤醒 Service Worker，后者从队列中取出请求逐个发送。这里的关键设计是**指数退避重试**和**幂等性保障**——同一个任务创建请求不能因为重试而产生两条记录。

## 三、冲突场景：用户不会按你的预设操作

冲突不是边缘情况，而是离线优先系统的常态。至少需要处理两类典型场景。

**场景一：多端编辑同一任务。** 用户在手机上离线修改了任务标题，同时在电脑上在线修改了同一任务的截止日期。当两端都上线同步时，服务端收到两个针对同一任务的“部分更新”。解决方案不能是“谁最后写入谁覆盖”，因为这会丢失一方的重要修改。实践中常采用**基于字段级别的合并**：如果两个操作修改的是不同字段，则直接合并；如果修改了同一字段，则根据时间戳或用户定义的优先级规则裁决，冲突记录提示用户手动处理。

**场景二：离线删除与在线更新交叉。** 用户A离线删除了一个任务，用户B在线重命名了该任务。当A上线同步删除操作时，服务端发现该任务已被B修改。直接执行删除会丢失B的工作，拒绝删除则导致A看到“明明删了却又出现”。合理的做法是**软删除标记**：任务标记为“已删除但未同步”，同时保留B的修改。在服务端，需要为每次操作生成唯一的事件ID，系统根据事件ID判断冲突，生成一个“待决议冲突列表”供用户审查。

## 四、数据流：从本地到远端的一次完整旅程

一个离线优先的操作生命周期如下：

1. **本地写入**：用户点击“完成任务”，应用首先更新 IndexedDB 中对应的记录状态，页面立即反映变化（乐观更新）。同时生成一条同步日志，包含操作类型、目标ID、操作时间戳、本地版本号。

2. **排队**：同步日志被添加到 IndexedDB 的同步队列中。如果网络在线，立即出队发送；如果离线，留在队列中等待。

3. **同步**：Service Worker 通过后台同步事件或定期 `fetch` 尝试将队列中的操作发往服务端API。每个请求携带同步日志的全部元数据。

4. **确认**：服务端接收操作后，执行冲突检测逻辑。如果无冲突，返回确认响应（包含服务端版本号），应用将本地同步日志标记为“已确认”，更新本地版本号。

5. **失败回滚或重试**：如果服务端返回冲突或错误，应用需要评估。对于可重试的错误（如网络超时），重新入队等待下次同步。对于冲突，应用将冲突记录存入本地“冲突列表”，并在任务界面上给出视觉提示，引导用户手动解决。

## 五、可观测性：看不见的离线世界需要监控

离线系统的可观测性比在线系统更重要，因为故障发生时的用户往往处于无网络状态，无法主动报告问题。以下是必须追踪的指标：

- **同步失败率**：每次同步尝试失败时，记录失败的API端点、失败原因（网络错误/冲突/数据格式错误）。按用户和设备维度聚合，识别出“总是同步失败”的异常设备。
- **冲突率**：统计每次冲突事件的频率、冲突类型（字段冲突/删除更新冲突）、冲突解决方式（自动合并/用户手动解决/以谁为准）。高冲突率通常是服务端冲突检测策略不合理的信号。
- **平均同步恢复时间**：记录每个操作从入队到确认耗时。这个指标反映了用户在多长时间内处于“数据不确定”状态。如果大部分操作的恢复时间超过30秒，说明后台同步触发机制需要优化。
- **队列积压深度**：IndexedDB 中未同步的操作数量。监控这个指标可以快速发现“网络恢复但同步卡死”的异常情况。

这些数据应通过应用内埋点上传，即使本地操作失败，也要确保错误日志能被可靠上报（例如单独的错误上报队列，优先于普通操作同步）。

## 六、落地建议：少踩坑，多思考

1. **先做“同步模型”再做“UI交互”**：很多团队先做离线编辑功能，再考虑数据冲突。正确的顺序是：先在纸上画出数据流图，明确每个操作的幂等性方案、冲突合并策略，再做界面。UI的交互细节（如冲突提示弹窗、同步进度条）都依赖于底层的同步模型。

2. **为每条记录预留版本号字段**：从第一次建表开始，就给每条任务加上 `localVersion` 和 `serverVersion` 两个字段。前者由客户端递增，后者由服务端维护。无法处理冲突的离线系统，本质上是“假离线”。

3. **主动制造“不完美”的测试环境**：不要只在WIFI下测试。搭建一个网络模拟器，随机阻断请求、随机产生延迟、模拟服务端部分冲突。只有在这种环境下跑过一遍完整流程，你才能信任自己的离线系统。
