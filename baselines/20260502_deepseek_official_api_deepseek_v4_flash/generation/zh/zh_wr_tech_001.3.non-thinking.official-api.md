# 离线优先任务管理：从“断网可用”到“数据可信”的设计哲学

## 一、离线优先的真正挑战：用户信任的基石

大多数开发者对离线优先的第一反应是“断网也能用”，这没错，但远远不够。真正的离线优先，核心在于**用户对数据的信任**——用户相信他提交的任务在任何网络条件下都不会丢失，相信多端操作后的结果符合预期，相信即使设备离线，应用也不会产生不可逆的数据错误。

这种信任一旦被破坏（比如离线提交的任务在同步后丢失，或者两个设备编辑同一个任务导致混乱），用户就会回到“在线才敢操作”的保守模式，离线优先的价值就丧失了。因此，离线优先设计的第一原则不是“能用”，而是“可靠”。

## 二、三大基础设施的分工协作

### 1. IndexedDB：本地数据的主战场
IndexedDB 承担的是**离线状态下的数据持久化**。所有用户操作——创建任务、修改截止日期、标记完成——都首先写入 IndexedDB。它必须能够存储完整的数据模型，并支持本地查询、排序、分页。这里的关键设计是：本地数据是“源”，而不是“缓存”。用户看到的 UI，始终直接读取 IndexedDB，而非依赖网络请求的响应。

### 2. Service Worker：网络请求的调度中心
Service Worker 的角色在于**拦截网络请求并为离线状态提供降级策略**。但它不应负责数据同步逻辑——那是后台同步模块的工作。SW 的主要职责是：当应用发起同步请求时，如果网络不可用，将请求缓存到内部队列；当网络恢复时，按顺序重放请求。它更像是“网络层的路由器”，确保请求不会因为断网而丢失。

### 3. 后台同步与重试队列：数据一致性的守门人
这是离线优先最核心的组件。它管理着一次操作的完整生命周期：**本地写入 → 排队等待同步 → 发送同步请求 → 确认服务端结果 → 更新本地状态（成功或失败回滚）**。当同步失败时，重试队列会根据指数退避策略反复尝试，并记录每次失败的上下文。这个队列需要具备持久化能力，即使浏览器关闭，待同步的操作也不会丢失。

## 三、冲突场景：最容易被忽视的设计盲区

### 场景一：同一任务在多端被编辑
用户 A 在手机上将任务“写周报”的截止时间改为“周五”，同时用户 B 在电脑上将同一个任务的优先级改为“高”。当两端同步时，服务端收到两个修改，但修改的是不同的字段。

**设计思路**：采用“基于字段的合并”策略。服务端记录每个字段的修改时间戳，当发现两个修改不冲突时（修改不同字段），自动合并；当修改同一字段时，以最后写入为准，或者将冲突标记为“需要用户裁决”，推送到一个冲突列表。

### 场景二：离线删除与在线更新交叉
用户 A 离线删除了任务“买咖啡”，同时用户 B 在线将这个任务的描述改为“买拿铁”。当用户 A 恢复网络时，同步队列发送删除请求，但此时服务端已有用户 B 的更新。

**设计思路**：引入“软删除”机制。本地删除时，先将任务标记为“待删除”，但保留数据。同步时，服务端收到删除请求，如果发现任务在线上已被更新，则返回冲突信息，客户端将这些冲突任务放入“待处理”列表，由用户决定是“确认删除”还是“保留并接受更新”。如果没有冲突，则真正删除，并清除本地数据。

## 四、简化数据流：从写入到确认的五个阶段

1. **本地写入**：用户点击“添加任务”，数据先写入 IndexedDB，UI 即时更新。此时数据有一个本地 ID 和状态“pending”。
2. **排队**：后台同步模块将此次操作（新增任务 ID xxx）加入持久化队列，标记为待同步。
3. **同步**：当网络可用时，同步模块按顺序取出队列中的操作，发送 HTTP 请求到服务端。发送前会锁住该操作，防止重复处理。
4. **确认**：服务端处理成功，返回响应（可能包含服务端生成的正式 ID）。客户端用服务端 ID 更新本地数据，将状态改为“synced”，并从队列中移除该操作。
5. **失败回滚或重试**：如果服务端返回错误（如数据校验失败），客户端将本地状态改为“failed”，并在 UI 上显示错误标记，用户可手动触发重试。如果是网络错误，则保留状态“pending”，等待下一次同步尝试。

## 五、可观测性：让离线系统“有迹可循”

离线优先系统最大的隐患在于“黑盒”——用户不知道操作是否成功，开发者不知道同步是否卡住。因此，可观测性必须从设计第一天就植入：

- **同步失败日志**：记录每次失败的操作 ID、失败原因（网络超时、服务端拒绝、冲突）、重试次数、时间戳。
- **冲突率监控**：统计一段时间内发生冲突的操作占总同步操作的比例。冲突率突然升高可能意味着多端协作场景出现异常，或者数据模型设计有问题。
- **平均恢复时间**：从操作入队到最终同步成功（或被用户确认失败）的平均时间。这个指标直接反映同步链路的健康度。如果恢复时间持续上升，说明重试策略或服务端处理能力需要优化。

这些数据可以写入本地日志表，并在网络可用时上报到监控系统，不需要额外开发复杂的工具链。同时，在应用的“设置”页面上提供一个“同步状态”面板，让用户也能看到待同步数量和失败记录——透明度本身就是信任的来源。

## 六、项目落地建议

1. **从冲突最少的场景开始迭代**：先实现“单人单设备”的纯离线功能，再逐步加入多设备同步。不要一开始就设计复杂的冲突解决机制，那会成为过早优化。
2. **为每一条数据记录设计明确的“修改时间戳”和“版本号”**：这是解决一切冲突的基础。没有时间戳，任何合并策略都是空中楼阁。
3. **在开发阶段强制模拟网络降级**：使用工具将网络延迟设为 2-3 秒，并随机断开连接，尽早发现同步队列的死锁或丢失操作的问题。这比上线后等待用户投诉要高效得多。
