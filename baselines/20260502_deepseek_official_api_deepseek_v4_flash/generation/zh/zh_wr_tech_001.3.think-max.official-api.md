# 离线优先设计：构建值得信任的任务管理应用

在移动办公、协作频繁的今天，轻量级任务管理应用几乎是每个团队的刚需。开发者往往首先关注功能完整性和界面流畅度，而“离线优先”则容易被简化为“缓存页面、断网也能看”。但当我们真正将一个任务管理应用推向生产环境，离线优先的核心挑战远不止缓存：它关乎用户是否敢于完全依赖你的应用——即信任。用户需要确信，即便在电梯里无网状态下创建的任务，在多个设备间编辑的条目，最终都不会丢失、不会矛盾、不会静默覆盖。这种信任建立在数据一致性的基础上，而不仅仅是可用。

## 技术支柱：各司其职的三个角色

实现离线优先需要三个关键技术组件协同工作，它们各有侧重，缺一不可。

**IndexedDB** 是本地持久化的基础。它提供结构化的、类数据库的存储能力，适合存储任务对象、元数据以及“待同步操作”的序列化记录。与 LocalStorage 相比，它能存储更大体积的数据，且支持索引和事务，方便我们维护本地副本与服务器状态的差异。它的职责是成为客户端数据的一等公民，而不仅仅是缓存。

**Service Worker** 身处浏览器与网络之间，它负责两件事：一是拦截网络请求，从本地缓存中提供静态资源（使应用本身离线可用）；二是通过 `SyncManager` 注册“后台同步”事件。当用户设备从离线状态恢复网络时，Service Worker 被唤醒，触发同步逻辑。它并不直接处理业务逻辑，而是扮演调度者的角色，确保同步任务在合适的时机被执行。

**后台同步/重试队列** 是业务层面的仲裁器。当用户进行增删改操作时，应用首先将操作记录写入 IndexedDB 中的“操作日志”队列，然后尝试立即发给服务器。如果失败，队列保留操作，并注册一个后台同步事件。Service Worker 在连接恢复时读取队列中的操作，逐个向服务器发送，并根据响应决定是删除操作（成功）还是增加重试次数（失败）。常见的策略是“指数退避”和“最大重试次数”。超过限制的操作应转入“失败队列”等待人工介入。

这三个组件构成了离线优先的骨架：IndexedDB 提供“放得下”，Service Worker 提供“等得起”，队列提供“不丢不糊”。

## 冲突场景：一致性必须面对的硬仗

在多端离线/在线混合的场景下，冲突不可避免。轻量级任务管理虽然业务简单，但冲突类型仍需要精心设计。

**场景一：同一任务在多端被编辑。** 用户 A 在手机端离线时修改了“购买食材”任务的截止日期，用户 B 在电脑端同时修改了该任务的描述。当两端先后上线同步时，服务器需要决定以哪一版本为准。如果采用“最后写入胜利”，后同步的数据会覆盖先到的数据，用户 B 可能丢失自己的修改。一种更合理的做法是记录每一次操作的因果依赖（例如使用 Lamport 时间戳或版本向量），服务器检测到冲突时拒绝自动合并，将改动分别存入“冲突分支”，并通知用户手动解决。在轻量级应用中，最简单的策略是保留先到达的版本，并将后到达的版本作为“建议变更”存入注释，由用户确认。无论如何，不能静默覆盖。

**场景二：离线删除与在线更新交叉。** 用户离线时删除了一个任务，但同一任务在服务器端已被另一个用户更新了备注。当用户上线执行同步时，本地的删除操作（通常是 DELETE 请求）与服务器端的更新操作（PUT 请求）形成矛盾。简单地按时间顺序执行可能会导致：删除请求到达后任务被移除，而后续更新请求又重建了该任务（或返回 404 导致错误）。更健壮的方式是使用墓碑标记（tombstone）：本地删除时将该任务标记为“已删除”并保留其 ID；服务器收到删除请求后标记删除，并拒绝针对该 ID 的后续更新。更新请求如果基于旧版本（version tag），服务器可以返回冲突，强制客户端刷新后再操作。

## 简化数据流：从写入到确认的四步征程

可以想象一个任务创建操作的生命周期：

1. **本地写入**：用户点击新建任务，应用将任务对象（含本地唯一 ID、版本戳、时间戳）写入 IndexedDB，状态设为“待同步”。UI 立即显示该任务，无需等待网络。
2. **排队**：与此同时，应用将“创建任务”这个操作（包含完整 payload）放入同步队列（IndexedDB 中的操作日志）。队列的每条记录都有类型、重试次数、创建时间。
3. **同步**：如果设备在线，应用立即从队列中取出操作发起请求到服务器。若失败（网络错误或服务器繁忙），操作保留在队列，重试次数加 1，并注册后台同步。若设备离线，Service Worker 监听到网络恢复，自动唤醒同步流程，逐条处理队列。
4. **确认或回滚**：服务器成功处理后返回确认（含服务端 ID 和最终版本戳）。应用更新本地数据的版本戳，将操作从队列中移除，任务状态变为“已同步”。如果服务器返回拒绝（例如冲突或校验失败），应用根据策略决定回滚：或是将本地数据恢复为操作前快照，或是标记为“冲突”等待用户决策。若经过多次重试仍然失败，则转入失败列表，并通知用户手动处理。

整个过程理想状态是“最终一致”，但用户感知始终是即时响应、数据无损。

## 可观测性：知道你的系统有多可靠

离线优先系统最大的风险在于“看不见的问题”。用户可能在离线时做了很多操作，后台悄悄失败了多次但没有人知道。因此，必须将可观测性纳入设计。

**同步失败率**：每次队列操作失败（尤其是超过重试阈值）都应记录失败原因（网络超时、服务端错误、冲突等）。聚合这些数据，可以评估网络状况和服务器稳定性。

**冲突率**：记录服务器返回冲突的次数及冲突类型。高冲突率可能暗示应用的设计鼓励了并发编辑，或时间戳/版本机制不够合理。

**平均恢复时间（MTTR）**：从操作写入队列到被标记为已同步的平均耗时。这个指标反映了同步延迟，过高的 MTTR 可能意味着后端处理慢或重试间隔不合理。

这些指标可以通过在本地日志中埋点，借助 Web Vitals 或自定义事件上报到监控平台（如 Sentry、开源的 StatsD 等）。有了数据，团队才能有针对性地优化冲突解决策略或重试算法。

## 三条落地建议

1. **为每条记录赋予版本戳，拒绝无冲突检测的静默合并。** 这是避免数据覆盖的最后防线。轻量实现可以用简单整数版本号，每次同步时携带，服务端验证是否与当前版本一致。不一致直接返回冲突。

2. **将操作日志视为核心实体，给予和业务数据同等的持久化保护。** 很多应用只把日志当作临时队列，意外丢失导致不同步。将日志设计为 IndexedDB 中的一个独立对象仓库，支持按时间排序、分页、重新入队，确保即使页面崩溃也不会丢失待同步的操作。

3. **设立“冲突面板”和“同步健康看板”，让异常可见。** 用户端提供一个入口，展示等待手动解决的冲突条目。运维端通过看板观察失败率、队列深度和平均恢复时间。定期 Review 这些指标，根据实际数据调整重试策略、超时时间和冲突解决算法，而不是靠猜。

离线优先不是一次性功能，而是一个持续演化的信任构建过程。从第一天就考虑数据一致性，你的任务管理应用才能让用户放心地在任何地方写下每一件小事。
