# 离线优先任务管理的正确打开方式：信任、冲突与可观测性

当用户打开一个 Web 应用，发现页面一片空白，提示“网络连接不可用”——这是几乎所有在线优先应用的原罪。离线优先（Offline-First）试图解决这个场景，但其真正价值远不只是“断网也能用”。对于一个轻量级任务管理应用，离线设计关乎用户对你的信任：用户是否相信自己的操作在断网时不会丢失？当多端同时编辑时，数据能否自动收敛到合理状态？这些问题一旦处理不好，离线能力反而会成为 bug 的温床。

## 离线优先的真正含义：从可用到可信

如果只是把数据缓存到本地，网络恢复后再全量覆盖服务端，那等于用更复杂的方式做了一款“假离线”应用。真正的离线优先意味着：
- **用户在任何状态下都能操作**，包括创建、编辑、删除任务，应用不因网络抖动而卡顿。
- **用户的操作是确定性的**，即使网络中断，本地所见即所得，且后续同步不会丢失或篡改已确认的结果。
- **数据一致性是可预期的**，当冲突发生时，系统有明确的策略（比如最后写入获胜、手动合并或版本向量）让用户理解发生了什么。

这三层要求直接关联用户信任：如果离线任务被静默覆盖，用户就会担心“我做了那么多，怎么不见了？” 信任一旦破裂，离线功能便失去意义。

## 三大引擎：IndexedDB、Service Worker 与后台同步队列

支撑离线优先的底层基础设施有三个，各自承担不同职责，不能混为一谈。

### IndexedDB：本地的事实源
IndexedDB 是浏览器提供的 NoSQL 数据库，用来存储完整的数据副本。在任务管理应用中，所有任务记录、元数据、最后同步时间戳都应持久化于此。它的角色是“本地事实源”——离线状态下，UI 完全基于 IndexedDB 渲染，用户的所有 CRUD 操作直接写入该数据库，而不是先尝试请求网络。写入成功后立即更新界面，给用户“操作成功”的即时反馈。

### Service Worker：网络流量的守门员
Service Worker 拦截网络请求，主要负责缓存策略和离线页面渲染。对于任务管理，通常采用“网络优先，回退到缓存”的策略——先尝试从服务端拉取最新数据，如果失败则返回 IndexedDB 中的缓存。但 Service Worker 本身不负责数据同步，它的核心使命是让应用即使在离线也能加载，并能让部分 API 请求“暂时挂起”。注意，Service Worker 不能长期持有 IndexedDB 写入后的状态，它更适合做静态资源的缓存和简单 API 的离线响应。

### 后台同步与重试队列：网络恢复后的桥梁
这部分是离线优先的“灵魂”。浏览器提供 Background Sync  API，允许注册一个同步事件，当网络恢复时自动触发。但更可靠的方案是自行实现一个**重试队列**，存储在 IndexedDB 中。每次用户操作（创建/更新/删除）后，除了写入本地数据库，还要向队列插入一条待同步记录，包含操作类型、任务 ID、变更数据和一个递增的序列号。Service Worker 或主线程在监听到网络恢复时，按顺序消费队列，逐条发给服务端。遇到失败（如网络超时、服务端冲突）则保留在队列中，标记重试次数和下一个重试时间，采用指数退避策略。这个队列就是离线与在线之间的“事务日志”。

## 两类典型的冲突场景

离线优先应用最难的不是让数据存到本地，而是当网络恢复后，多端或多次操作发生的冲突如何化解。

### 场景一：同一任务被多端同时编辑
用户 A 在手机端将任务“写报告”的截止日期改为明天，用户 B 同时在电脑端将该任务的优先级改为“高”。这两个修改没有先后顺序，都发生在离线状态下。当两端网络恢复，同步队列中的记录先后到达服务端。此时服务端可能收到两条针对同一任务的修改，分别修改了不同字段。如果简单采用“最后写入覆盖”，则先到的修改会被丢弃，导致用户 A 的日期变更丢失。更好的策略是**基于字段级别的合并**：服务端维护每个字段的版本号，只有当本地的版本号大于等于当前服务端版本时才写入，否则触发冲突记录并让用户下次打开时手动解决。

### 场景二：离线删除与在线更新交叉
用户 A 离线删除了任务“采购清单”，同时用户在 B 在线或离线状态下更新了该任务的内容。当 A 的删除操作同步到服务端时，服务端发现该任务已被 B 更新过。如果直接删除，则 B 的修改永久丢失；如果拒绝删除，则 A 的操作被忽略。常见的处理方式是**软删除 + 墓碑记录**：先给任务打上“delete-pending”标记，保留最新版本，然后由服务端根据时间戳和操作类型做仲裁：如果删除发生在更新之前（按本地时间），则最终删除；如果更新发生在删除之后，则拒绝删除并提示用户。这看起来复杂，但可以通过一个简单的冲突解决窗口让用户判断——比如在同步页面里展示“XX任务已被其他设备更新，是否仍要删除？”

## 简化数据流：本地写入→排队→同步→确认→回滚

整个过程可以概括为五个步骤：

1. **本地写入**：用户点击“创建任务”后，应用立即将新任务插入 IndexedDB，分配一个临时 ID，并更新 UI。用户感觉“已经创建成功”。
2. **排队**：同时将一条同步记录（操作类型、临时 ID、完整数据）推入 IndexedDB 中的队列表，状态为“pending”。
3. **同步**：当网络可用（通过 navigator.onLine 或 Service Worker 检测），后台脚本按时间顺序取出 pending 状态的记录，将操作发送至服务端 API。发送时携带临时 ID，服务端会返回真实 ID 和确认时间戳。
4. **确认**：收到服务端 200 响应后，将本地 IndexedDB 中的临时 ID 替换为真实 ID，并将队列记录标记为“synced”或直接删除。
5. **失败回滚或重试**：如果请求失败（如 409 冲突、502 服务器错误），根据失败类型决定：对于冲突类错误，将队列记录标记为“conflict”，并在 UI 中提示用户手动解决；对于网络错误，保留队列并设置重试时间戳（例如 5 秒后重试，最多 10 次）。

这个流程确保任何操作都有迹可循，即使中途应用关闭，下次启动时也能从队列中恢复未完成的同步。

## 可观测性：量化离线系统的健康度

没有可观测性的离线系统就像在夜间开车不开灯。作为工程师，需要持续了解以下指标：

- **同步失败次数与失败率**：记录每条同步记录最终状态（success / conflict / timeout / 其他错误）。可以在 IndexedDB 中维护一张统计表，每次同步完成时写入。结合用户 ID 和事件时间，能及时发现某个用户频繁冲突或网络异常。
- **冲突率**：统计冲突记录占总同步记录的比例。如果冲突率高于阈值（如 5%），说明用户多端使用频繁，可能需要优化合并算法或提示用户上线前先手动同步。
- **平均同步恢复时间（Mean Time to Sync）**：从用户最后一次离线操作到该操作在服务端确认的时间差。如果恢复时间过长，可能暴露后台同步逻辑问题（如指数退避过慢、网络恢复检测不灵敏）。
- **队列积压深度**：待同步队列中 pending 和 conflict 记录的数量。高积压表示网络长期不可用或循环重试失败。

这些数据可以通过本地日志上报至服务端监控系统（如 Sentry、Grafana），或在前端调试面板中展示给开发者。用户视角下，可以提供一个“同步状态”页面，显示最近一次同步时间、待处理任务数量，让用户对数据流动有感知。

## 项目落地建议

1. **从“最小可离线”开始，逐步增加冲突策略**：第一版只保证最核心的创建和编辑操作离线可用，冲突策略用最简单的“最后写入获胜”。不要一开始就做字段级合并或复杂冲突 UI，先跑通流程，第二步再根据实际冲突率引入手动解决机制。

2. **善用时间戳与版本号，但不要依赖客户端时间**：永远以服务端时间戳为准，客户端只负责记录本地操作顺序。使用单调递增的版本号或 LWW（Last Writer Wins）时，要保证服务端是唯一的时间源，避免用户调手机时间导致数据错乱。

3. **为每个操作提供幂等标识**：在每一条同步记录中生成唯一 ID（例如 UUID），服务端根据该 ID 去重。这能防止因网络重试导致同一操作被执行多次（如创建两条重复任务）。同时，服务端应支持批量确认，减少单个请求开销。

离线优先不是银弹，但它能让 Web 应用在弱网环境下依然保持可用性和用户信任。处理好本地存储、同步队列和冲突解决，再辅以可观测性度量，你的任务管理应用就能从“能用”进化到“敢用”。
